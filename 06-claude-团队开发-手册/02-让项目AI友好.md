# 第 2 章 · 让项目"AI 友好"

一个项目对 AI 友好不友好,直接决定 AI(以及新人)在里面干得好不好。这一章讲怎么组织项目结构和文档,让 AI 一上手就能高效工作——而且这些做法对**任何** AI 工具都有效,不只是 Claude Code。

---

## 2.1 "AI 友好"几乎等于"好工程"

好消息先说:让项目对 AI 友好,基本上就是把好工程实践做到位。AI(和新人)在一个项目里高效工作,需要的东西高度一致:

- 清晰、一致的目录结构 → 知道东西放哪、该往哪加
- 成文的约定和命令 → 知道怎么构建、测试、部署
- 好的文档 → 知道为什么这么设计、有哪些坑
- 可靠的测试 → 知道改对没改对(地面真相,第五册第 3 章)
- 清晰的类型/接口 → 减少歧义和猜测

**这些对人和 AI 都好。** 所以"让项目 AI 友好"不是额外负担,是借 AI 落地的契机,把本该做好的工程基础做扎实。而且——它们全在**工具中立层**(第 0 章),对用任何 AI 工具的人都有效。

## 2.2 项目结构约定

AI 在结构清晰、约定一致的项目里表现最好。几条实用约定:

### 一致的目录布局

```
project/
├── src/              # 源码,按职责分层
│   ├── api/          #   路由层
│   ├── services/     #   业务逻辑
│   └── repos/        #   数据访问
├── tests/            # 测试,镜像 src 结构
├── scripts/          # 构建/部署/工具脚本
├── docs/             # 架构和约定文档
├── CLAUDE.md         # Claude Code 项目记忆(配置层)
├── CONTRIBUTING.md   # 团队约定(工具中立层)
└── README.md         # 项目入口
```

关键不是用哪种布局,而是**一致**——同类东西放同样的地方,AI 据此能推断"新的 X 该放哪"。

### 清晰的分层和边界

如果你的项目有架构规则(如"API 层不写业务逻辑""数据访问只走 repos 层"),**把它写下来**(2.4 节)。AI 会遵守成文的规则,但猜不出没说的规则。

### 标准化的命令

把"怎么构建/测试/部署"标准化成命令,别让人(和 AI)去猜:

```makefile
# Makefile —— 工具中立,所有人和所有 AI 工具都能用
dev:      ## 启动开发环境
	docker compose up -d && uvicorn app:main --reload
test:     ## 跑全量测试
	pytest
lint:     ## 代码检查
	ruff check . && ruff format --check .
```

有了标准命令,AI 的验证指令就简单可靠:"改完跑 `make test`"。而且这对用任何工具的人都一样。

## 2.3 文档作为契约

文档是项目和"任何上手者(人或 AI)"之间的契约。三份核心文档,各有分工:

| 文档 | 写什么 | 给谁 | 层 |
|---|---|---|---|
| **README.md** | 项目是什么、怎么跑起来 | 所有人、所有 AI | 工具中立 |
| **CONTRIBUTING.md** | 团队约定:分支、提交、PR、代码风格、怎么加新功能 | 所有贡献者、所有 AI | 工具中立 |
| **架构文档**(docs/) | 为什么这么设计、分层规则、关键决策 | 深入开发者、AI | 工具中立 |

**写"无法从代码推断的事"**(第一册第 3 章的原则):命令、约定、坑、边界、设计理由。代码结构本身 AI 自己能读,你要补的是代码里看不出的"为什么"和"不成文规定"。

## 2.4 关键模式:工具中立文档 + 各工具配置引用它

**这是混合工具团队的核心实操,直接回应"队友用别的工具怎么办"。**

问题:Claude Code 用 `CLAUDE.md` 作项目记忆,但 Cursor 用 `.cursorrules`、别的工具用 `AGENTS.md`、还有人裸写不用任何 AI 工具。如果你把项目约定只写进 `CLAUDE.md`,那只有 Claude Code 用户能受益。

解法:**把实质性约定写进一份工具中立的文档,各工具的配置文件去引用/导入它。**

```
              ┌──────────────────────────────────┐
              │  CONTRIBUTING.md(或 docs/约定.md)  │  ← 实质内容的唯一来源
              │  · 分层规则  · 命令  · 风格  · 坑    │     (工具中立,人也读)
              └──────────────────────────────────┘
                  ▲              ▲              ▲
       @import 引用 │   引用       │   引用        │  人直接读
              ┌─────┴────┐  ┌─────┴─────┐  ┌─────┴────┐
              │ CLAUDE.md │  │.cursorrules│  │ 裸写的人  │
              │(Claude)   │  │ (Cursor)   │  │          │
              └──────────┘  └───────────┘  └──────────┘
```

Claude Code 的 `CLAUDE.md` 支持 `@import` 引用其他文件,所以:

```markdown
<!-- CLAUDE.md -->
# 项目记忆

完整的团队约定见 @CONTRIBUTING.md。
以下是 Claude Code 特有的补充:
- 跑测试用 `make test`,不要直接 pytest（环境变量在 Makefile 里）
- 提交前必过 `make lint`
```

这样:

- **实质约定只有一份**(CONTRIBUTING.md),改一处所有工具同步,不会各自维护一份导致漂移。
- **Claude Code 用户**:CLAUDE.md 自动加载 CONTRIBUTING 的内容 + Claude 特有补充。
- **Cursor 用户**:`.cursorrules` 里同样引用或粘贴 CONTRIBUTING 的要点。
- **裸写的人**:直接读 CONTRIBUTING.md。

> **AGENTS.md 这个新趋势**:越来越多 AI 编码工具在向一个工具中立的约定文件 `AGENTS.md` 收敛。如果你的团队工具混杂,可以把约定主文件命名为 `AGENTS.md`(或 CONTRIBUTING.md),让 `CLAUDE.md` 等各工具配置去 `@import` 它。核心思想不变:**实质内容放工具中立的一份,各工具配置只做引用和工具特有补充。**

## 2.5 CLAUDE.md 在团队里怎么写

CLAUDE.md 是 Claude Code 配置层的项目记忆(第一册第 3 章)。团队场景的要点:

- **进 git,团队共享**(项目级 `CLAUDE.md` 或 `.claude/CLAUDE.md`)。团队成员 clone 即继承。
- **保持精简**:它每次会话占上下文。实质约定 `@import` 自 CONTRIBUTING,CLAUDE.md 只放 Claude Code 特有的补充和指引。
- **个人偏好不要进项目 CLAUDE.md**:那是团队共享的。个人偏好放 `~/.claude/CLAUDE.md`(用户级)或 `CLAUDE.local.md`(gitignore)。

一份团队项目 CLAUDE.md 的好样子:

```markdown
# 项目:订单服务

团队完整约定见 @CONTRIBUTING.md（分层规则、命令、代码风格都在那）。

## Claude Code 特有补充
- 跑测试:`make test`（不要直接 pytest,环境变量在 Makefile）
- 数据库迁移:`make migrate`,不要手改 migrations/ 下已有文件
- 改完 API 层,同步更新 docs/api.md

## 已知的坑
- tests/integration/ 依赖本地 Redis,没起会大面积超时而非报连接错
```

## 2.6 用 `/init` 起步

给一个还没有 CLAUDE.md 的项目起步:

```
/init        # Claude 扫描项目,自动生成 CLAUDE.md 初稿
```

然后按 2.5 的原则改造:把实质约定挪进 CONTRIBUTING.md,CLAUDE.md 改成 `@import` + 特有补充。

## 2.7 让项目 AI 友好的检查清单

- [ ] 目录结构一致,同类东西放同样的地方
- [ ] 构建/测试/部署标准化成命令(Makefile / scripts/)
- [ ] README 让人能跑起来项目
- [ ] CONTRIBUTING.md(或 AGENTS.md)成文记录团队约定——**工具中立**
- [ ] 架构规则、分层边界、坑都写下来了
- [ ] CLAUDE.md `@import` CONTRIBUTING + Claude 特有补充,而非重复一份
- [ ] 个人偏好不在项目 CLAUDE.md 里
- [ ] 有可靠的测试,AI 能据此自我验证

> 注意:这一章的产物大多在**工具中立层**——这是有意的。一个 AI 友好的项目,首先是个好项目,对所有人和所有工具都友好。Claude Code 的 CLAUDE.md 只是在这个好基础上,给 Claude Code 用户的一层薄薄的便利。
