# 第 7 章 · 评估、可观测性与可靠性

agent 是非确定性的——同样的输入,两次运行可能走完全不同的路。这给"怎么知道它好不好""出问题怎么查""怎么让它可靠"带来独特挑战。这一章讲 agent 的评估、可观测性和可靠性工程。

---

## 7.1 为什么 agent 难评估

普通函数:输入 X → 永远输出 Y,好测。agent:输入 X → 这次走 A 路径输出 Y1,下次走 B 路径输出 Y2。**非确定性**是 agent 评估的根本难题:

> agent 做动态决策,运行之间非确定——**即使 prompt 完全相同,两次也可能不同。**

所以你不能用"断言输出等于期望值"来测 agent。需要不同的方法。

## 7.2 从小处开始评估

**别一上来就建大评估集。** Anthropic 的经验:

> 从约 **20 个代表真实使用模式的查询**开始。

为什么 20 个就够起步?因为 agent 的改动往往产生**戏剧性效果**——"一个 prompt 调整可能把成功率从 30% 提到 80%"。这种量级的变化,20 个 case 就能清晰检测到。等系统成熟了再扩大评估集。

这也呼应第四册第 0 章的评估驱动方法论——先有可度量的标准和测试,再优化。

## 7.3 用 LLM 当裁判(规模化评估)

agent 输出往往是开放式的(一份报告、一段代码),没有唯一正确答案,没法用正则或精确匹配评。解法:**用一个 LLM 当裁判**,对照评分标准(rubric)打分。

典型的评分维度:

```
- 事实准确性(factual accuracy)
- 引用准确性(citation accuracy)
- 完整性(completeness)
- 来源质量(source quality)
- 工具效率(tool efficiency)
```

一个 LLM 裁判对照这些标准评分,能规模化到"数百个输出",同时"保持一致、与人类判断对齐"。

```python
def llm_judge(client, query, agent_output, rubric):
    prompt = f"""你是一个严格的评估者。对照标准给 agent 的输出打分。

查询:{query}
agent 输出:{agent_output}
评分标准:{rubric}

对每个维度打 1-5 分,给出理由。最后给总分和是否通过。"""
    return call(client, prompt)
```

注意:呼应第 2.2 节的路由——用大小合适的模型当裁判,简单评估用 Haiku 省钱。

## 7.4 人工测试抓边界情况

**LLM 裁判规模化,人工测试抓边界。两者都要。**

> "人测 agent 能发现评估漏掉的边界情况"——包括幻觉的答案、系统故障、来源选择偏差。

自动评估覆盖广度,人工测试覆盖你想不到的角落。Anthropic 明确说:人工测试在自动化之外仍然"必不可少"。别迷信全自动评估。

## 7.5 可观测性:看见非确定的系统

agent 非确定,所以**可观测性比传统软件更重要**——你没法靠"重现 bug"来调试一个每次走不同路径的系统。

### 全链路追踪(Tracing)

> 实现"完整的生产追踪",**监控 agent 的决策模式和交互结构**。

关键:追踪的是 agent 的**决策模式和结构**(它在每步选了什么工具、走了什么路径),而不一定要看对话内容。Anthropic 的做法是监控结构而不访问对话内容——既能诊断根因,又保护隐私。

你要能回答:这次运行,agent 在第几步偏离了?它为什么选了那个工具?哪一步开始跑错了轨迹?——看得见这些,才能调试。

### 透明性(回顾第 0 章)

好 harness 让 agent 的规划步骤可见(第 0.3 节透明性原则)。Claude Code 显示 TODO、工具调用,就是可观测性的体现。**看不见的 agent 是没法信任、没法改进的 agent。**

## 7.6 可靠性:对抗错误复合

第 6.6 节提过错误复合,这里讲怎么对抗。**agentic 系统里错误会复合**:

> agent 跨很多工具调用维护状态。小故障会变成灾难——一步失败,能让 agent 探索完全不同的轨迹。传统软件里的小问题,在 agent 里可能让它整个跑偏。

三个可靠性工程手段:

### ① 检查点(Checkpoints)

定期保存 agent 的状态。出错时能从断点恢复,而非从头再来。第 5 章的结构化记笔记、git 状态跟踪都是检查点的形式。

### ② 重试逻辑 + 优雅降级

工具失败了,别让整个 agent 崩溃——把错误喂回模型(第 3.3 节、第二册第 4.4 节的 `is_error`),让它换方法。结合模型智能处理优雅降级。

### ③ 能从出错点恢复的系统

设计 harness 时,要能"从 agent 出错时所在的位置恢复",而不是丢失所有进度。这对长程 agent 尤其关键——跑了两小时在第 99 步崩了,不能从第 1 步重来。

## 7.7 部署:彩虹部署(Rainbow Deployment)

更新一个有状态的 agent 系统有个特殊难题:**正在跑的 agent 怎么办?** 你不能像无状态服务那样直接替换——那会打断进行中的 agent。

Anthropic 的解法是**彩虹部署**:

> "逐步把流量从旧版本转移到新版本,同时让两者都运行",防止代码改动打断进行中的 agent。

旧版本继续服务它已经在跑的 agent,新流量走新版本,等旧 agent 都完成了再下线旧版本。这是有状态 agent 系统的部署模式。

## 7.8 可靠性与护栏的关系

回顾第 3.5 节的护栏。护栏(权限、沙箱、停止条件)和本章的可靠性工程(检查点、重试、追踪)是一体两面:

- **护栏**防止 agent 做坏事(安全)。
- **可靠性工程**防止 agent 做错事后崩溃、或让你查不出原因(稳健)。

两者都是 harness 的职责(第 0 章),共同把一个"能自主"的 agent 变成一个"能放心自主"的 agent。

## 7.9 端到端实战:一套可复用的评估脚手架

前面讲了评估的原则(20 条查询起步、LLM 裁判、rubric)。这一节给你**一套完整、可运行、改改就能用**的评估脚手架。你的 agent/prompt 改动后,跑一遍就知道是变好还是变坏——这是把第四册第 0 章"评估驱动"落到地的工具。

### 三个文件

> 👉 **可直接运行的完整三文件**：[`示例代码/评估脚手架/`](示例代码/评估脚手架/)（`eval_dataset.py` / `rubric.py` / `run_eval.py`，配好 key 即可 `python run_eval.py`）。下面逐个读。

一套最小评估系统就三部分:**数据集**(代表真实使用的查询)、**rubric**(评分标准)、**runner**(对每版跑分、汇总)。

`eval_dataset.py` —— 你的 20 条查询(从真实使用里挑,这里用客服 agent 举例):

```python
# 每条:查询 + 这条期望命中的要点（供裁判参考，不必是标准答案）
DATASET = [
    {"id": 1, "query": "退款要满足什么条件？",
     "expect": "提到七天内、保留原包装"},
    {"id": 2, "query": "你们发什么快递？偏远地区多久到？",
     "expect": "提到顺丰、偏远 3-5 天"},
    {"id": 3, "query": "我想退一个用了一个月的东西",
     "expect": "应说明超出七天、不符合退款条件，而非编造可以退"},
    # … 凑到约 20 条，覆盖正常 / 边界 / 容易让它幻觉的情况
]
```

`rubric.py` —— 评分标准(清晰、可独立判断,见 7.3):

```python
RUBRIC = """对照以下维度给 agent 的回答打分（每项 1-5 分）：
1. 事实准确性：是否只依据知识库、没有编造？
2. 完整性：是否覆盖了问题的关键点？
3. 恰当拒答：当知识库没有答案或不符合政策时，是否如实说明而非硬编一个？
4. 简洁性：是否直接、无废话？
任一维度 ≤2 分即判定该条不通过（pass=false）。"""
```

`run_eval.py` —— 跑分器(完整可运行):

```python
import json
import anthropic
from eval_dataset import DATASET
from rubric import RUBRIC

client = anthropic.Anthropic()

def run_agent(query: str) -> str:
    """被测系统：换成你真实的 agent/prompt。这里用占位实现。"""
    resp = client.messages.create(
        model="claude-opus-4-8", max_tokens=1024,
        system="你是客服助手，只依据知识库回答，没有就如实说。",
        messages=[{"role": "user", "content": query}],
    )
    return next(b.text for b in resp.content if b.type == "text")

def judge(query: str, answer: str, expect: str) -> dict:
    """LLM 裁判：对照 rubric 打分，返回结构化结果（7.3 节）。"""
    resp = client.messages.create(
        model="claude-opus-4-8", max_tokens=1024,
        messages=[{"role": "user", "content": f"""{RUBRIC}

查询：{query}
该回答期望命中：{expect}
被评估的回答：{answer}

以 JSON 输出：{{"scores": {{"准确性": n, "完整性": n, "拒答": n, "简洁": n}}, "pass": true/false, "reason": "..."}}"""}],
        output_config={"format": {"type": "json_schema", "schema": {
            "type": "object",
            "properties": {
                "scores": {"type": "object", "additionalProperties": {"type": "integer"}},
                "pass": {"type": "boolean"}, "reason": {"type": "string"},
            },
            "required": ["scores", "pass", "reason"], "additionalProperties": False,
        }}},
    )
    return json.loads(next(b.text for b in resp.content if b.type == "text"))

def main():
    passed = 0
    fails = []
    for case in DATASET:
        answer = run_agent(case["query"])
        verdict = judge(case["query"], answer, case["expect"])
        mark = "✅" if verdict["pass"] else "❌"
        print(f"{mark} [{case['id']}] {case['query'][:20]}…")
        if verdict["pass"]:
            passed += 1
        else:
            fails.append((case["id"], verdict["reason"]))

    print(f"\n通过率：{passed}/{len(DATASET)} = {passed/len(DATASET)*100:.0f}%")
    if fails:
        print("\n失败的 case（拿去针对性改进）：")
        for cid, reason in fails:
            print(f"  [{cid}] {reason}")

if __name__ == "__main__":
    main()
```

### 跑起来

```bash
python run_eval.py
```

输出长这样:

```
✅ [1] 退款要满足什么条件？…
✅ [2] 你们发什么快递？偏远地区…
❌ [3] 我想退一个用了一个月的…

通过率：18/20 = 90%

失败的 case（拿去针对性改进）：
  [3] agent 说可以退款，但用了一个月已超出七天，属于编造
```

### 怎么用它驱动改进

这套脚手架的价值,在于把"改 agent"变成可度量的科学(第四册第 0 章):

```
改 prompt/agent → 跑 run_eval.py → 看通过率变化 + 失败 case → 针对失败改一处 → 重跑
```

- **基线**:先用现在的版本跑一遍,记下通过率(比如 18/20)。
- **改一处**:比如针对 case 3 的幻觉,在 system 加一句"不确定或超出政策时如实说明,绝不编造"。
- **重跑**:看通过率有没有升、且**没弄坏别的 case**(这就是保留数据集的意义——防回归)。
- **一次只改一处**(第四册第 0.4 节纪律),否则不知道是哪处起的作用。

### 三个升级方向

这是最小版,真实用时按需加:

1. **加人工抽查**(7.4):自动评估覆盖广度,但定期人工看几条裁判结果,确认裁判本身没跑偏——裁判也是模型,也会错。
2. **裁判用更便宜的模型**:评估要跑很多次,裁判可以用 Sonnet/Haiku 省钱(被测系统该用什么用什么)。
3. **存历史**:把每次运行的通过率和失败 case 存下来,看趋势——哪次改动让它退步了,一目了然。

> **关键心法**:别等系统成熟了才建评估。**20 条就能起步**,而且 agent 改动的效果往往很戏剧(一个 prompt 调整可能把通过率从 30% 拉到 80%,7.2 节)——这种量级,20 条就测得出来。今天就建,比"以后再说"强得多。

## 7.10 评估与可靠性清单

把 agent 推向生产前:

- [ ] 有约 20 个代表真实使用的评估查询
- [ ] 用 LLM 裁判 + 清晰 rubric 做规模化评估
- [ ] 有人工测试抓边界情况(别全自动)
- [ ] 有全链路追踪,能看 agent 的决策路径
- [ ] 关键节点有检查点,能从出错处恢复
- [ ] 工具失败喂回模型而非崩溃(优雅降级)
- [ ] 长程任务能从中断处恢复,不丢进度
- [ ] 有状态系统用彩虹部署,不打断进行中的 agent
- [ ] 监控决策模式(可在不看内容的前提下做,保护隐私)
