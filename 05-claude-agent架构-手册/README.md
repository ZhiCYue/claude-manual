# Agent 架构、动态工作流与多智能体协作（第五册）

> 编写时间：2026-06，基于 Anthropic 官方工程文章《Building Effective Agents》《Effective Context Engineering for AI Agents》《Multi-Agent Research System》及前四册积累。
> 前置阅读：第二册（API / Agent SDK）打底，第四册（Prompt Engineering）的 agentic 章节是本册的 prompt 基础。这一册讲**怎么架构**一个 agent 系统。

## 这册讲什么

前四册你学会了：用 Claude Code（①）、调 API 手写 agent 循环（②）、造 MCP 工具（③）、把话说对（④）。这一册回答一个更高层的问题：

> **当任务复杂到需要 LLM 自主、多步、甚至多个 agent 协作完成时,系统该怎么架构?**

这是把零件组装成机器的一册。你会学到：harness（运行 agent 的编排框架）怎么设计、工作流的五种经典模式、自主 agent 的解剖、工具与环境接口（ACI）、上下文工程、以及**多智能体协作自动完成整个任务**的架构与成本权衡。

## 全书主线：从简单到复杂，且只在值得时才复杂

Anthropic 反复强调、也是本册的最高原则：

> **从最简单的方案开始,只有当复杂度能切实改善结果时才增加它。** agentic 系统用延迟和成本换性能——这个权衡要谨慎对待。LLM 领域的成功不是构建最复杂的系统,而是构建**适合你需求的正确系统**。

这条主线贯穿全册。每讲一个更复杂的模式,都会同时讲"它什么时候**不**值得"。多智能体能把研究质量提升 90%,但烧 15 倍的 token——什么时候这笔账划算,是真正的工程判断。

## 学习路径建议（3 周）

```
第 1 周  00–02 章：harness、workflow vs agent、五模式。
         练习：把你做过的一个多步任务,识别它该用哪种工作流模式
第 2 周  03–05 章：自主 agent、ACI、上下文工程。
         练习：给一个 agent 设计工具集,应用"防错"和"token 效率"原则
第 3 周  06–08 章：多智能体、评估可靠性、架构蓝图(本册核心)。
         练习：判断你的某个任务该不该上多 agent,算一笔 token 成本账
```

## 三条贯穿全册的硬规则

1. **从简单开始,复杂度要挣来**：单次 LLM 调用 + 检索 + 示例能搞定的,别上 agent；workflow 能搞定的,别上 multi-agent。复杂度必须用可度量的改善来证明自己（第 1 章）。
2. **上下文是有限资源**：token 越多,注意力越稀释（"context rot"）。把上下文当人的工作记忆来管理——每个 token 都消耗注意力预算（第 5 章）。
3. **多 agent 烧钱,要算账**：多智能体约用 15 倍于聊天的 token。只在价值能证明这个倍数时才用——宽度优先的研究值,共享上下文的编码通常不值（第 6 章）。
