## A. 双层设计速查(全册核心)

```
① 工具中立层 —— 对所有人生效,是团队标准的执行者
   项目结构 · 文档(CONTRIBUTING/AGENTS.md) · CI · git pre-commit ·
   linter · PR/commit 规范 · 分支保护 · AI 代码审查纪律

② Claude Code 配置层 —— 只对 Claude Code 用户,是便利层
   .claude/settings.json · CLAUDE.md · .mcp.json · commands · agents

铁律:要对所有人生效的规则,放①强制;②是锦上添花,不是标准的执行者。
```

## B. "放哪一层"决策速查

| 我想要… | 放哪 |
|---|---|
| 提交前必过 lint(所有人) | ① git pre-commit + CI |
| PR 必有测试和审查(所有人) | ① CI required check + 分支保护 |
| 密钥绝不进仓库(所有人) | ① gitignore + git hook 扫描 + CI 扫描 |
| 团队约定的唯一来源 | ① CONTRIBUTING.md / AGENTS.md |
| 不直推 main(所有人) | ① 分支保护 |
| Claude Code 改完即时 format | ② .claude/settings.json Hook |
| Claude Code 常用命令免询问 | ② .claude/settings.json permissions |
| Claude Code 团队命令/MCP | ② .claude/commands / .mcp.json |
| 强制登录组织、最低版本(Claude Code 用户) | ② 企业 managed policy |

## G. 常见问题(FAQ)

**Q1:团队里有人用 Cursor/Copilot,这套规范还有用吗?**
有用,而且本来就该这么设计。把所有"必须做到"的标准放工具中立层(CI/git hook/分支保护),它对所有人生效,无论用什么工具。Claude Code 配置只是给 Claude Code 用户的便利层。见第 4 章。

**Q2:怎么让团队标准对所有人强制?**
用工具中立的执行者:CI required checks、分支保护、git pre-commit hooks、linter 配置。这些不依赖任何 AI 工具。别用 Claude Code 的 hook 当团队标准的执行者——它只约束用 Claude Code 的人。

**Q3:团队约定该写哪、怎么让各工具都能用?**
写进一份工具中立的文档(CONTRIBUTING.md 或 AGENTS.md),作为唯一来源。CLAUDE.md `@import` 它,Cursor 的 .cursorrules 引用它,裸写的人直接读。改一处全同步。见第 2.4 节。

**Q4:AI 生成的代码怎么审、谁负责?**
提交者负全责,和手写代码一样。AI 代码走相同(甚至更严)的审查,不豁免。重点看:形似神不似、过度工程、幻觉 API、安全漏洞。与是哪个 AI 工具无关。见第 5 章。

**Q5:哪些配置该进 git?**
能进的都进(配置即代码)。工具中立层(linter/CI/pre-commit/CONTRIBUTING)+ Claude Code 团队配置(settings.json/CLAUDE.md/.mcp.json/commands/agents)。只有个人本地的 settings.local.json 不进。见第 3 章。

**Q6:怎么控制团队成本和模型?**
按任务选模型(日常 sonnet,难任务 opus/fable),effort 按需,监控 /usage。组织级用 availableModels(managed policy)限制可选模型。见第 7.2、7.3 节。

**Q7:怎么防 AI 把密钥读进上下文或提交进仓库?**
多重防护:gitignore + git hook 密钥扫描 + CI 扫描 + 不硬编码(全工具中立)+ Claude Code deny .env*(额外第一道防线)。任何单一防线都不够,尤其不能只靠单一工具。见第 6.4 节。

**Q8:从哪开始落地?**
分四阶段:个人采纳(几个人验证价值)→ 项目落地(一个项目改 AI 友好 + 建工具中立护栏)→ 团队推广(处理混合工具、定规范)→ 组织治理(managed policy、成本)。每阶段验证再扩大。见第 8 章。

## I. 全册一页纸

```
核心问题:怎么把 Claude Code 落地进真实(混合工具)团队的日常开发?

核心原则:双层设计
  ① 工具中立层 = 团队标准的执行者(对所有人:CI/git hook/分支保护/文档/审查)
  ② Claude Code 配置层 = 便利层(对 Claude Code 用户:settings/CLAUDE.md/命令)

混合工具的答案:要对所有人生效的规则放①强制,②不依赖、只增强

三条硬规则:
  1. 要对所有人生效 → 放工具中立层强制
  2. AI 写的,人负责(与工具/模型无关)
  3. 配置即代码,能进 git 都进 git

落地路线:个人采纳 → 项目落地 → 团队推广 → 组织治理(每阶段验证再扩大)
```
