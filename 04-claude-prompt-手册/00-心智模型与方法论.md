# 第 0 章 · 心智模型与方法论

这一章不教任何具体技法，只建立两个东西：一个正确的心智模型（prompt 到底是什么），和一套方法论（怎么系统地改进 prompt，而不是凭感觉瞎调）。

---

## 0.1 Prompt 是什么，不是什么

**Prompt 是你给模型的全部输入文本**——系统提示、对话历史、当前问题、附带的文档，合在一起。模型读这段文本，输出下一段文本。就这么简单，也就这么深。

它**不是**：

- 不是咒语或魔法词。"请""务必""作为专家"这些词本身没有魔力，有用是因为它们传递了真实信息（礼貌不改变行为，但"作为放射科医生"提供了角色上下文）。
- 不是一次性的。好 prompt 是迭代出来的，不是一次写对的。
- 不是万能的。有些问题（延迟、成本、模型能力上限）prompt 解决不了。

它**是**：

- 是你和模型之间唯一的接口。模型对你的意图的全部理解，都来自这段文本。
- 是可工程化的。有原理、有技法、可度量、可复现。
- 是杠杆。同样的模型，好 prompt 和差 prompt 的产出可能天差地别。

## 0.2 核心隐喻：聪明但全新的同事

整本书的主线，再说一遍：

> **把 Claude 当成一位绝顶聪明、但对你的规范和工作流毫无背景的新员工。**

这个隐喻能帮你做几乎所有 prompt 决策：

- 新同事不知道你的"显然"——所以要**给上下文**（第 1 章）。
- 新同事看范例学得快——所以**给示例**（第 2 章）。
- 新同事面对一堆混杂的材料会懵——所以**用结构**（第 3 章）。
- 新同事需要知道自己的角色——所以**设定角色**（第 4 章）。
- 复杂任务，新同事需要时间想——所以**让他思考**（第 5 章）。

**黄金检验法则**：把你的 prompt 拿给一个对任务背景了解很少的同事，让他照着做一遍。如果他会困惑、会问"这是什么意思""我该输出什么格式"，那 Claude 也会困惑。困惑点就是你该补充的地方。

## 0.3 什么时候该用 prompt engineering，什么时候不该

**这是"进阶"的第一个分水岭。** 不是每个失败的 case 都该靠改 prompt 解决。

| 问题 | 更好的解法 |
|---|---|
| 延迟太高 | 换更快的模型（Sonnet/Haiku）、降 effort、用流式 |
| 成本太高 | 换便宜模型、用 prompt caching（第二册第 6 章）、batches 半价 |
| 撞上模型能力上限 | 换更强模型（Opus/Fable）、把任务拆小 |
| 需要海量专属知识 | 检索增强（RAG）、把知识放进上下文 |
| **可通过措辞引导的行为** | **← prompt engineering 解决这个** |

prompt engineering 擅长解决的是：输出格式不对、语气不合适、该用工具时没用、过度啰嗦或过度简略、推理不够深、不遵循约束……这些"通过把话说对就能改善"的问题。

**先问自己：这个问题，是措辞能解决的吗？** 如果不是，别在 prompt 上死磕。

## 0.4 评估驱动：别凭感觉调 prompt

**这是"进阶"的第二个分水岭，也是业余和专业的真正区别。** 业余选手改 prompt 靠"感觉好像好点了"；专业选手靠**可度量的评估**。

优化 prompt 之前，你需要三样东西：

1. **明确的成功标准**：什么样的输出算"对"？是格式合规？是包含某些要点？是某个指标达标？说不清成功标准，就没法判断改动是变好还是变坏。
2. **可测试的方法**：一组测试用例（输入 + 期望特征），能对每版 prompt 跑一遍看效果。可以是人工评分，也可以是自动检查（正则、JSON schema 校验、用另一个模型当裁判）。
3. **一版初稿 prompt**：拿来改的起点。

有了这三样，迭代就变成科学：

```
初稿 prompt → 跑测试集 → 看哪些 case 失败 → 针对失败改一处 →
重跑测试集 → 确认这处变好且没弄坏别的 → 重复
```

**关键纪律**：

- **一次只改一个变量。** 同时改五处，你不知道是哪处起了作用（或起了反作用）。
- **保留测试集。** 改 prompt 最大的风险是"修好了 A 却弄坏了 B"。每次改完跑全量测试集才能发现回归。
- **量化，别凭感觉。** "感觉好点了"不可靠——人的记忆和注意力会骗你。

## 0.5 技法的优先级顺序

后面的章节是按一个大致的"投入产出"顺序排的。遇到问题时，按这个顺序往下试，通常前面的技法就解决了：

```
1. 清晰、直接、具体（第 1 章）    ← 80% 的问题在这里解决，最高优先级
2. 给上下文和动机（第 1 章）
3. 用示例（第 2 章）             ← 格式/风格问题的利器
4. 用 XML 结构化（第 3 章）       ← prompt 变复杂时
5. 设定角色（第 4 章）
6. 让它思考（第 5 章）           ← 推理/数学/复杂任务
7. 控制输出格式（第 6 章）
8. 链式分解（第 7 章）           ← 单个 prompt 搞不定的复杂任务
9. agentic / 模型专属调优（第 9、10 章）  ← 进阶场景
```

**先用简单的。** 很多人一上来就堆复杂技巧，其实"把话说清楚"（第 1 章）就能解决大部分问题。技巧是用来解决简单方法解决不了的问题的，不是越多越好。

## 0.6 一个贯穿全册的对比例子

为了让你直观感受 prompt engineering 的威力，记住这个官方例子，后面会反复呼应：

**差的 prompt：**
```
创建一个分析仪表盘
```

**好的 prompt：**
```
创建一个分析仪表盘。包含尽可能多的相关功能和交互。
不要止步于基础——做一个功能完整的实现。
```

差别在哪？第二个**明确要求了"超出基础"的行为**。最新模型不会自动从模糊的 prompt 里推断出你想要"上不封顶"——你想要"超额完成"，就得明说。这是第 1 章的核心，也是整个 prompt engineering 的缩影：**你想要什么，就具体地说出来。**
