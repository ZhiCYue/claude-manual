# 第 1 章 · 工作流 vs Agent

这是设计 agent 系统最根本的一个分叉,也是 Anthropic agent 框架的核心区分。搞清楚它,你才知道一个任务该用确定性的工作流,还是让模型自主的 agent。

---

## 1.1 核心分界

Anthropic 把 agentic 系统分成两类:

| | 工作流(Workflow) | Agent(代理) |
|---|---|---|
| 定义 | LLM 和工具通过**预定义的代码路径**编排 | LLM **动态地自主决定**流程和工具使用 |
| 谁控制流程 | 你的代码(固定路径) | 模型(自主决策) |
| 路径 | 提前确定、可预测 | 运行时决定、不可预测 |
| 类比 | 流水线 | 自主工人 |

**一句话区分**:工作流的步骤是你**提前写死**的;agent 的步骤是模型**当场决定**的。

```
工作流:  输入 → [步骤A] → [步骤B] → [步骤C] → 输出
                （顺序由你的代码固定）

Agent:   输入 → 模型决定下一步 → 执行 → 看结果 → 模型再决定 → …
                （路径由模型动态决定,每次可能不同）
```

## 1.2 共同的基石:增强型 LLM

无论工作流还是 agent,基础构件都是**增强型 LLM**(augmented LLM)——一个被三种能力增强的模型:

```
        ┌─────────────┐
        │     LLM      │
        └──┬───┬───┬──┘
           │   │   │
      检索 ─┘   │   └─ 记忆
              工具
```

- **检索(Retrieval)**:搜索和访问外部信息。
- **工具(Tools)**:与外部服务和 API 交互(第 4 章)。
- **记忆(Memory)**:保留和回忆相关信息(第 5 章)。

现代 Claude 模型能主动生成搜索查询、选择合适的工具、决定保留什么信息。Anthropic 推荐用 **MCP**(第三册)来实现这些增强——一次实现,接入整个生态。

这个增强型 LLM 是你搭建一切的乐高底板。工作流是把它按固定路径串起来;agent 是给它工具和环境,让它自己跑。

## 1.3 何时用工作流,何时用 agent

**决策标准**:

| 用工作流 | 用 agent |
|---|---|
| 任务可预测、定义良好 | 开放式问题 |
| 步骤能提前确定 | 步骤无法硬编码 |
| 要求一致性 | 需要灵活性 |
| 例:把工单分类后走不同模板 | 例:根据任务描述改多个文件 |

**工作流**适合可预测、需要一致性的任务。**agent**适合开放式、步骤数不定、必须模型自主跑很多轮的问题——但它要求你对模型的决策有足够信任,并在沙箱里测好护栏。

agent 的典型场景:

- SWE-bench 编码任务——根据任务描述改多个文件,改几个、怎么改取决于输入。
- 计算机使用(computer use)——Claude 自主操作系统。

## 1.4 最高原则:从简单开始

**这是整本书最重要的一句话,值得反复读:**

> **从最简单的方案开始,只有当复杂度能切实改善结果时才增加它。**

agentic 系统用**延迟和成本**换性能。这个权衡要谨慎对待:

- 很多应用,**单次优化过的 LLM 调用 + 检索 + 示例**就够了。别一上来就上 agent。
- 工作流能搞定的,别上 agent(agent 更难预测、更难调试)。
- 单 agent 能搞定的,别上多 agent(多 agent 烧 15 倍 token,第 6 章)。

> "LLM 领域的成功,不是构建最复杂的系统,而是构建**适合你需求的正确系统**。"

复杂度阶梯,从下往上,只在下一层确实解决不了时才往上爬:

```
1. 单次 LLM 调用(+ 检索 + 示例)   ← 大多数需求停在这
2. 工作流(固定路径编排)
3. 单 agent(模型自主)
4. 多 agent(编排者-工作者)        ← 最贵,最后才考虑
```

## 1.5 该不该建 agent?四问

升到 agent 这一层之前,四个问题全答"是"才值得(呼应第二册第 0 章):

1. **复杂度**:任务是否多步且难以提前完全规定?
2. **价值**:结果是否值得更高的成本和延迟?
3. **可行性**:Claude 在这类任务上确实能干好吗?
4. **错误成本**:错了能否被发现和恢复(测试、审查、回滚)?

任何一个"否",退回工作流或单次调用。

## 1.6 一个判断练习

几个任务,判断该用哪层:

| 任务 | 该用 | 为什么 |
|---|---|---|
| 把客服邮件分类成 3 类 | 单次调用 | 一个 prompt 搞定 |
| 把文档先翻译再校对再润色 | 工作流(链式) | 步骤固定、顺序明确 |
| 根据工单类型走不同处理流程 | 工作流(路由) | 类别已知、分支固定 |
| "把这个设计文档变成一个能跑的 PR" | agent | 步骤不定、要自主探索代码库 |
| "调研这个行业,产出一份深度报告" | 多 agent(可能) | 宽度优先、可并行、价值高 |

注意最后一个标了"可能"——多 agent 是否值得,要算 token 账(第 6 章)。能用单 agent 达到够用的质量,就别上多 agent。

## 1.7 工作流和 agent 不是非此即彼

实战中,一个系统常常**混用**。比如:

- 整体是个工作流(路由 → 分支),但某个分支里是个 agent。
- 整体是个 agent,但它调用的某个工具内部是个固定工作流。
- 编排者是 agent(动态决定派多少 worker),worker 各自是简单的工作流。

所以别把这章的区分当成"二选一的站队",而是当成**一套可组合的构件**。第 2 章讲工作流的五种模式,第 3 章讲自主 agent,第 8 章讲怎么把它们组合成完整架构。
