# 实战示例一 · Vite + React 任务看板（操作指南）

一个用**当前 SPA 主流栈**(Vite + React 19 + TypeScript + TanStack Query)做的任务看板。本指南从零搭到跑通,每步标注用到本册哪一章——让你看到"主流框架 + AI 辅助 + 规格驱动"怎么落地。

> 成品功能:列出任务、添加任务、切换完成状态。用 TanStack Query 管"服务端状态"(这里用一个模拟的异步 API,让你无需后端就能跑)。

---

## 第 0 步:你将得到什么

```
task-board/
├── package.json
├── vite.config.ts
├── AGENTS.md                      # 项目约定(工具中立唯一来源,第4章)
├── e2e/tasks.spec.ts              # Playwright e2e(UI 地面真相,第4章)
└── src/
    ├── main.tsx                   # 入口 + QueryClientProvider
    ├── App.tsx
    └── features/
        └── tasks/                 # 按功能组织(第4章)
            ├── types.ts
            ├── api.ts             # 模拟异步 API
            ├── useTasks.ts        # TanStack Query hooks(第3章状态三分法)
            ├── TaskBoard.tsx      # 组件
            └── TaskBoard.test.tsx # Vitest 组件测试(逻辑地面真相,第4章)
```

本文件夹已提供这些源码文件,你可以直接复制。下面是从零搭的完整流程。

## 第 1 步:脚手架(主流命令)

```bash
# Vite 官方脚手架,react-ts 模板 —— 这是 2026 新建 SPA 的主流起手(第1章)
npm create vite@latest task-board -- --template react-ts
cd task-board
npm install

# 装本示例用到的库
npm install @tanstack/react-query
npm install -D vitest @testing-library/react @testing-library/user-event jsdom @playwright/test
```

> 对照第 1 章:用 Vite 不用 CRA(已停维护)、不用 webpack(SPA 场景 Vite 更优)。`npm run dev` 秒起,反馈循环短——AI 迭代也快。

## 第 2 步:建立 AI 友好的环境(第 4 章)

这一步是"支柱一(好环境)"。先把约定和结构立好,AI(和你)后面才好干活。

1. **写 `AGENTS.md`**(本文件夹已提供)——技术栈、按功能组织、状态三分法、不手动 memo 等约定。这是**工具中立的唯一来源**:

   ```markdown
   # 见本文件夹的 AGENTS.md
   ```

2. **让 CLAUDE.md 引用它**(用 Claude Code 的话):

   ```markdown
   <!-- CLAUDE.md -->
   团队约定见 @AGENTS.md。
   ## Claude Code 补充
   - 开发:npm run dev(端口 5173)
   - 改 UI 后用 Playwright 实际打开页面验证
   ```

3. **按功能组织目录**:`src/features/tasks/`——这个功能的 types/api/hook/组件/测试都在一起(第 4.2 节)。

## 第 3 步:用规格驱动建"任务"功能(第 6-7 章)

"任务看板"是个**较大的功能**,适合走规格驱动(支柱二)。两种方式:

### 方式 A:用 OpenSpec(推荐,第 7 章)

```bash
npm install -g @fission-ai/openspec && openspec init
```

在 Claude Code 里:

```
/opsx:propose "任务看板"
```

AI 生成 spec(proposal/specs/design/tasks)。**你审 spec**——确认它把这些定对了:

- 任务有 id / 标题 / 完成状态
- 服务端状态用 TanStack Query(不塞全局 store)
- 验收:能列出、能添加、能切换完成;切换后 UI 立即反映
- 技术方案:features/tasks/ 下的 types/api/useTasks/TaskBoard

审满意后:

```
/opsx:apply        # AI 照 spec 实现
/opsx:archive      # 完成后归档
```

### 方式 B:不用 OpenSpec,用 Plan Mode(第一册)

较小的功能可以简化:Claude Code 里 `Shift+Tab` 进 Plan Mode,说"实现任务看板,用 TanStack Query 管状态,按 AGENTS.md 的约定",审计划后执行。

> **关键**(第 6 章):无论哪种方式,核心是**先对齐"要建什么"再让 AI 写**。本示例的源码就是这一步该产出的东西——你可以直接用它对照 AI 的产出。

## 第 4 步:核心代码讲解

(完整文件见本文件夹,这里点出关键设计)

**状态三分法(第 3.6 节)** —— `useTasks.ts` 用 TanStack Query 管服务端状态,**不塞全局 store**:

```typescript
// 服务端数据用 useQuery,自动管缓存/加载/错误
export function useTasks() {
  return useQuery({ queryKey: ["tasks"], queryFn: fetchTasks });
}
// 改动用 useMutation,成功后让缓存失效自动重取
export function useAddTask() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: addTask,
    onSuccess: () => qc.invalidateQueries({ queryKey: ["tasks"] }),
  });
}
```

**现代 React(第 3 章)** —— 函数组件 + hooks,**不手动 memo**(React Compiler 处理),逻辑在 `useTasks` hook 里,组件 `TaskBoard.tsx` 保持简洁。

## 第 5 步:测试作为地面真相(第 4.4 节)

这是"支柱一"的关键——让 AI 能验证改对没改对。

**组件测试(Vitest,逻辑地面真相)**:

```bash
npm test           # 跑 TaskBoard.test.tsx
```

**e2e(Playwright,UI 地面真相)**:

```bash
npx playwright install     # 首次装浏览器
npm run e2e                # 跑 e2e/tasks.spec.ts
```

> 让 Claude Code 改完功能"跑 `npm test` 和 `npm run e2e` 确认"——它就有了客观的"对没对"依据,而非靠你肉眼。改 UI 时再让它用 Playwright MCP 实际打开页面截图对比(第 4.4 节)。

## 第 6 步:工具中立质量门(第 5 章)

把检查放 CI + pre-commit,对所有人强制(不只 Claude Code 用户)。`package.json` 的 scripts:

```jsonc
"scripts": {
  "dev": "vite",
  "build": "tsc -b && vite build",
  "lint": "eslint .",
  "typecheck": "tsc --noEmit",
  "test": "vitest run",
  "e2e": "playwright test"
}
```

CI 配置直接用第七册 `模板/ci.yml`(前端版),把 lint/typecheck/test/build/e2e 设为 required check。

## 第 7 步:跑起来

```bash
npm run dev        # 打开 http://localhost:5173
```

你会看到任务看板:能添加任务、勾选完成。改动会通过 TanStack Query 自动反映。

## 这个示例教会你什么

把本册的方法走了一遍完整闭环:

```
选 Vite(第1章) → AGENTS.md 立约定(第4章) → 规格驱动建功能(第6-7章)
→ TanStack Query 管状态(第3章) → Vitest/Playwright 验证(第4章)
→ 质量门进 CI(第5章)
```

**练习**:用规格驱动给它加一个"按完成状态筛选"功能——先 `/opsx:propose "任务筛选"`,审 spec,再 apply。体会"先定 spec"带来的确定性。
