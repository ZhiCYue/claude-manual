# 第 7 章 · 规模化与治理

从一个小组扩到整个组织,需要更强的治理手段:企业级强约束、团队模型分配策略、成本管理、新人 onboarding。这一章讲组织层面的治理——基于 Claude Code 的企业能力,以及通用的团队治理。

---

## 7.1 企业 Managed Policy:不可覆盖的强约束

第 3 章的团队配置可以被项目/个人覆盖。组织层面有些规则不能让人覆盖(安全合规、强制登录指定组织、最低版本)。Claude Code 提供**企业 managed 设置**——优先级最高、用户不可覆盖。

优先级(从高到低):

```
企业 managed policy(IT 下发,不可覆盖) ← 最高
命令行参数
项目本地 settings.local.json
项目 settings.json
用户 settings.json                      ← 最低
```

Managed settings 的三种下发方式:

- **服务端管理**(Anthropic Console):管理员在控制台集中配置,无需碰每台机器。
- **OS 级策略**(MDM / 组策略):macOS 用 `com.anthropic.claudecode` 托管偏好域;Windows 用 `HKLM\SOFTWARE\Policies\ClaudeCode`。
- **文件式**:系统目录下的 `managed-settings.json`(macOS `/Library/Application Support/ClaudeCode/`、Linux `/etc/claude-code/`、Windows `C:\Program Files\ClaudeCode\`)。

企业级常用的强约束:

```json
{
  "requiredMinimumVersion": "2.1.150",
  "forceLoginOrgUUID": "org-uuid-xxxx",
  "allowManagedPermissionRulesOnly": true,
  "permissions": {
    "deny": ["Bash(rm -rf /)", "Read(./secrets/**)", "Read(.env*)"]
  },
  "allowedMcpServers": [{ "serverName": "github" }, { "serverName": "memory" }],
  "deniedMcpServers": [{ "serverName": "filesystem" }],
  "availableModels": ["opus", "sonnet", "haiku"]
}
```

| 字段 | 作用 |
|---|---|
| `requiredMinimumVersion` | 低于此版本不准启动 |
| `forceLoginOrgUUID` | 强制登录指定组织(防用个人号) |
| `allowManagedPermissionRulesOnly` | 只有管理员定义的权限生效,用户/项目规则被屏蔽 |
| `allowedMcpServers` / `deniedMcpServers` | MCP 服务器白/黑名单 |
| `availableModels` | 限制用户能选哪些模型(下一节) |
| `claudeMd` | 注入组织级记忆,用户不可编辑 |

> **管理员可用 `claude doctor`** 查看所有生效的 managed 策略及来源,排查配置问题。

> **双层原则在这里依然成立**:managed policy 是 Claude Code 层最强的约束,但它只约束用 Claude Code 的人。组织级的"必须做到"(如代码安全门)仍要靠工具中立层(企业 CI、组织级分支保护策略)对所有人强制。两者配合,不互相替代。

## 7.2 团队模型分配策略

组织层面要决定:谁能用哪些模型、默认用什么、怎么平衡能力与成本。

### 用 availableModels 限制可选模型

managed/policy 设置里的 `availableModels` 限制用户能切换到哪些模型(对 `/model`、`--model`、`ANTHROPIC_MODEL` 全部生效):

```json
{ "availableModels": ["sonnet", "haiku"] }   // 这个团队只能用 sonnet/haiku
```

切到被禁模型会被拒绝。注意:`availableModels` 在多层设置时会合并去重,要严格控制就放 managed/policy 层(最高优先级)。

### 模型别名与分配建议

Claude Code 的模型别名(官方):

| 别名 | 解析到 | 适合 |
|---|---|---|
| `opus` | 最新 Opus(API 上 4.8) | 复杂推理、架构 |
| `sonnet` | 最新 Sonnet(4.6) | 日常编码 |
| `haiku` | Haiku | 简单/批量任务 |
| `fable` | Fable 5 | 最难、最长的任务 |
| `opusplan` | 计划用 opus、执行用 sonnet | 两全的默认 |
| `best` | 有 Fable 5 就用,否则最新 Opus | 要最强 |

团队分配的实用策略:

- **默认给 `sonnet` 或 `opusplan`**:日常编码 sonnet 够用且省;opusplan 在复杂任务上自动用 opus 规划。
- **opus/fable 按需**:最难的架构、疑难 bug 才上;成本敏感的团队可用 `availableModels` 限制谁能用。
- **第三方后端要 pin 版本**:用 Bedrock/Vertex/Foundry 的团队,部署前用 `ANTHROPIC_DEFAULT_OPUS_MODEL` 等环境变量 pin 具体版本 ID,控制何时升级,避免别名解析到未启用的新模型。

> 注意:`model` 设置只是初始选择,不是强制——用户仍能在 `/model` 里选 Default。要完全控制,组合 `availableModels`(限制可选)+ `model`(初始)+ `ANTHROPIC_DEFAULT_*_MODEL`(控制 Default 解析到哪)。

## 7.3 成本管理

团队规模化,成本要管:

- **按任务选模型**(第 1.5、7.2 节):别一律用最贵的。日常 sonnet,难任务才 opus/fable。
- **effort 按需**:简单任务 `low`/`medium`,别一律 `max`。
- **prompt caching**:Claude Code 自动用(第二册第 6 章),团队共享的大 CLAUDE.md/上下文会被缓存,省钱。注意别在共享配置里放每次都变的东西(破坏缓存)。
- **监控用量**:`/cost`(会话级)、`/usage`(额度);企业可在 Console 看组织级用量。
- **批处理走 Batches**:非实时的批量任务用 Batches API 半价(第二册第 7 章)。

> 成本和能力是权衡,不是越省越好也不是越强越好。团队该建立"什么任务值得什么模型"的共识——这本身可以写进 CONTRIBUTING.md。

## 7.4 新人 Onboarding

AI 友好的项目(第 2 章)+ 配置即代码(第 3 章)让新人 onboarding 大幅加速:

```
新人 git clone
   ↓
README 的 setup 步骤(装依赖、装 pre-commit、配环境)
   ↓
两层配置自动到位:
   · 工具中立层:CI/lint/约定/命令 —— 无论他用什么 AI 工具
   · Claude Code 层:权限基线、团队命令、MCP —— 如果他用 Claude Code
   ↓
读 CONTRIBUTING.md(团队约定的唯一来源)
   ↓
用 Claude Code 调研代码库:"带我过一遍这个项目的架构和数据流"
   （AI 友好的项目让 AI 能快速帮新人建立认知,第一册第 10.7 节)
```

把"新人第一天该做什么"写进 CONTRIBUTING.md 或 `docs/onboarding.md`。一个 AI 友好的项目,新人(借助任何 AI 工具)半天就能达到过去一两周的熟悉度。

## 7.5 度量与持续改进

治理要基于数据,不是凭感觉:

- **代码质量趋势**:CI 通过率、测试覆盖率、PR 返工率。
- **AI 采纳情况**:多少 PR 用了 AI 辅助、效果如何(可在 PR 模板的"AI 辅助说明"里收集)。
- **成本趋势**:组织级 token 用量、按团队/项目分摊。
- **评估驱动改进规范**:呼应第四册第 0 章——团队规范、CLAUDE.md、命令也该迭代。发现某条约定 AI 总不遵守?可能是约定写得不清,改它。

## 7.6 规模化与治理检查清单

- [ ] 组织级强约束用 managed policy(强制登录组织、最低版本、MCP 白名单)
- [ ] 用 availableModels 在团队层面控制可选模型
- [ ] 团队有明确的模型分配策略(默认 sonnet/opusplan,opus/fable 按需)
- [ ] 第三方后端 pin 模型版本
- [ ] 成本可监控(/usage、Console),有"什么任务用什么模型"的共识
- [ ] 新人 onboarding 文档化,clone 即继承两层配置
- [ ] 度量代码质量趋势、AI 采纳、成本,数据驱动改进
- [ ] 记住:managed policy 是 Claude Code 层强约束,组织级底线仍需工具中立层(企业 CI)对所有人强制
