# 第 4 章 · 混合工具团队

这是本册最该认真读的一章,直接回答那个最现实的问题:**团队里不是每个人都用 Claude Code——有人用 Cursor、有人用 Copilot、有人用 GPT 系工具、有人裸写。这套规范还成立吗?**

答案是:**只要你按双层设计组织,完全成立——甚至本来就该这么设计。**

---

## 4.1 现实:团队几乎都是混合工具的

先承认现实:**指望全队统一用一个 AI 工具,是不现实也不必要的。**

- 个人有偏好和习惯,强行统一阻力大、收益小。
- 不同工具各有所长,有人用 Claude Code 做 agent 式开发,有人用 Cursor 做行内补全。
- 工具在快速演进,今天统一了明天又变。
- 还有人(尤其某些场景)就是不用 AI 工具,裸写。

所以正确的目标不是"让全队用 Claude Code",而是:**让团队的工程规范,无论成员用什么工具,都能落地和被遵守。**

## 4.2 核心洞察:标准的执行者必须工具中立

回顾第 0 章双层设计的关键推论,这一章把它讲透:

> **一条团队规则,如果它的执行依赖某个特定 AI 工具,那它就只对用那个工具的人生效。**

举例,团队规则"提交前必须通过 lint":

```
❌ 把它实现成 Claude Code 的 PostToolUse Hook
   → 只有用 Claude Code 的人触发
   → 用 Cursor 的同事:Hook 不存在,照样提交不合规代码
   → 规则失效

✅ 把它实现成 git pre-commit hook + CI required check
   → 任何人 git commit 都触发 pre-commit
   → 任何人开 PR 都过 CI
   → 用 Claude Code / Cursor / 裸写,谁都逃不掉
   → 规则对所有人生效
```

**结论:团队标准的执行者,必须放在工具中立层。** AI 工具的配置(Claude Code 的 hooks/权限、Cursor 的 rules)只能约束用那个工具的人,不能作为团队标准的执行手段。

## 4.3 工具中立层的"执行者"清单

哪些机制是工具中立的、能对所有人强制?这是你的团队标准工具箱:

| 执行者 | 强制什么 | 怎么对所有人生效 |
|---|---|---|
| **CI(GitHub Actions 等)** | 测试、lint、构建、安全扫描 | 所有 PR 都跑,不过不让合 |
| **分支保护(branch protection)** | 必过 CI、必有审查、不准直推 main | 仓库级规则,对所有人 |
| **git pre-commit hooks**(pre-commit / husky) | 提交前 lint、format、密钥扫描 | 任何人 commit 都触发 |
| **linter / formatter 配置** | 代码风格 | 所有编辑器/工具读同一份配置 |
| **PR 模板 + required reviewers** | PR 质量、审查流程 | 仓库级,对所有 PR |
| **CODEOWNERS** | 关键路径必须特定人审 | 仓库级 |

这些全在第 3 章的工具中立层,全部进 git,全部对所有人生效——无论他用什么 AI 工具,甚至不用 AI。

## 4.4 关键实操:一个标准,两层落地

正确的做法不是"二选一",而是**同一个标准在两层都落地**:工具中立层负责**强制**(对所有人),Claude Code 层负责**体验**(让 Claude Code 用户更顺,且尽早发现问题)。

以"代码必须通过 ESLint"为例,三层防护:

```
① 写代码时(Claude Code 层,仅 Claude Code 用户)
   .claude/settings.json 配 PostToolUse Hook:改完文件即时 lint
   → Claude Code 用户当场看到问题,体验最顺
   （用 Cursor 的人有他自己工具的等价机制,或没有——没关系,下面兜底）

② 提交时(工具中立层,所有人)
   .pre-commit-config.yaml 配 ESLint
   → 任何人 git commit 都跑,不过不让提交

③ 开 PR 时(工具中立层,所有人)
   CI 把 ESLint 设为 required check + 分支保护
   → 任何人的 PR 不过 lint 都合不进去
```

注意这个分层的精妙:

- **①** 是给 Claude Code 用户的"快速反馈"——写的时候就修,体验最好。但它**只是便利,不是保障**。
- **②③** 是**保障**——无论用什么工具、有没有 ①,提交和 PR 这两道工具中立的关卡都拦得住。

> **一句话记住**:**便利可以工具特定,保障必须工具中立。** 想给 Claude Code 用户额外的顺手,尽管用 Claude Code 的 hooks/权限;但任何"必须做到"的标准,一定要在 ②③ 这种工具中立的关卡上落地。

## 4.5 共享上下文文档:让所有工具读同一份约定

第 2.4 节的模式在混合工具团队里是刚需。再强调一遍:

```
实质约定 → 一份工具中立文档(CONTRIBUTING.md / AGENTS.md)
            ├ Claude Code:CLAUDE.md @import 它
            ├ Cursor:.cursorrules 引用/粘贴要点
            ├ 其他 AI 工具:各自的配置引用它
            └ 裸写的人:直接读
```

这样所有人(和所有 AI 工具)看的是**同一份约定**,改一处全同步。别让每个工具维护自己的一份——那必然漂移成几个不一致的版本。

**实操建议**:

- 主约定文件用工具中立的名字(`CONTRIBUTING.md` 或新兴的 `AGENTS.md`)。
- 各 AI 工具的配置文件做成"薄壳":引用主文件 + 该工具特有的少量补充。
- 在 CONTRIBUTING.md 里明确写一句:"本文件是团队约定的唯一来源,各 AI 工具的配置应引用此文件而非另立一份。"

## 4.6 AI 生成代码的审查:天然工具无关

好消息:**审查 AI 生成的代码,本来就和用什么工具/模型无关。**

无论一段代码来自 Claude、GPT、Gemini 还是人手写,审查的标准是一样的:对不对、安不安全、可不可维护、有没有测试。所以第 5 章的"AI 代码审查纪律"对整个混合工具团队统一适用——你不需要为"Claude 写的"和"Cursor 写的"定不同的审查规则。

这也呼应第二条硬规则:**AI 写的,提交的人负责。** 谁提交,谁对那段代码负全责,谁负责让它通过审查——和代码出自哪个工具无关。

## 4.7 不同账号/订阅的现实

混合工具还带来账号层面的现实,团队要面对:

- 成员可能用不同的 Claude 订阅(Pro / Max / Team / 企业),默认模型不同(第 7 章)。
- 用第三方后端(Bedrock / Vertex)的成员,模型 ID 和默认模型又不同。
- 有人用个人订阅、有人用公司统一采购。

**应对原则**:团队标准建立在**产出**上(代码质量、测试、规范),而不是**用什么工具/账号产出**上。CI 不关心你的代码是 Opus 写的还是 GPT 写的还是你手写的——它只关心代码过不过测试、过不过 lint、有没有人审。把关卡设在产出上,工具和账号的差异就被吸收掉了。

## 4.8 混合工具团队检查清单

判断你的团队规范在混合工具下成不成立,逐项问:

- [ ] 每条"必须做到"的规则,是否在工具中立层强制(CI / git hook / 分支保护)?
- [ ] 有没有哪条团队标准只靠 Claude Code 的 hook/权限执行?(有 → 它对用别的工具的人失效,补上工具中立的强制)
- [ ] 团队约定是否有一份工具中立的唯一来源(CONTRIBUTING / AGENTS.md)?
- [ ] 各 AI 工具的配置是否引用那份来源,而非各维护一份?
- [ ] AI 代码审查标准是否统一(不分工具/模型)?
- [ ] 团队关卡是否设在"产出"上而非"用什么工具"上?

> **本章一句话总结**:不要试图统一团队的 AI 工具——那既不现实也没必要。要做的是:**把所有"必须做到"的标准,放在工具中立层强制;把 AI 工具的配置当成各自的便利层。** 这样无论队友用什么,团队规范都成立。
