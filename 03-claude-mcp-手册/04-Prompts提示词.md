# 第 4 章 · Prompts 提示词

提示词是 MCP 的第三种原语——**用户控制的、可复用的指令模板**。这一章讲它和工具/资源的区别、怎么定义带参数的模板、以及它在宿主里如何呈现为斜杠命令。

---

## 4.1 提示词是用户控制的

第三次回顾控制模型，这次落到提示词：

- **工具**：模型控制（模型自主调用）。
- **资源**：应用控制（应用决定喂什么上下文）。
- **提示词**：**用户**控制（用户显式触发）。

提示词是预制的指令模板，帮用户完成特定任务。它**需要用户显式调用**，不会自动触发。它能引用可用的资源和工具，编排成完整的工作流。

> **一句话区分三者**：工具是"模型能调的函数"，资源是"应用能读的数据"，提示词是"用户能选的模板"。提示词常呈现为**斜杠命令**——这正是第一册里你用的 `/xxx` 命令的来源之一。

## 4.2 定义提示词

用 `@mcp.prompt()`：

```python
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("review")

@mcp.prompt()
def code_review(language: str, focus: str = "all") -> str:
    """生成一段代码审查的指令模板。

    Args:
        language: 编程语言。
        focus: 审查重点，如 security / performance / all。
    """
    return f"""请审查接下来我提供的 {language} 代码。
重点关注：{focus}。
按严重程度分级输出问题（严重 / 建议 / 可选），
每个问题给出具体的行号和修改建议。"""
```

函数返回的字符串就是要注入对话的提示词文本。参数让模板可定制。

用户在宿主里触发它（如 Claude Code 里 `/review:code_review`），填入参数 `language="Python"`、`focus="security"`，这段文本就被注入对话，引导模型按你设计的方式工作。

## 4.3 带结构的提示词

提示词可以返回多条消息（构造 few-shot 例子或多轮上下文）。返回消息列表：

```python
from mcp.server.fastmcp.prompts import base

@mcp.prompt()
def debug_session(error: str) -> list[base.Message]:
    """启动一次结构化的调试会话。"""
    return [
        base.UserMessage("我遇到了这个错误："),
        base.UserMessage(error),
        base.AssistantMessage("我来帮你排查。先确认几点：这个错误是稳定复现还是偶发？"),
    ]
```

这样你能预置一个对话的开头，把模型引导进特定的工作模式。

## 4.4 提示词如何呈现

提示词是用户控制的，宿主通常通过这些 UI 暴露它：

- **斜杠命令**：输入 `/` 看到可用提示词（如 `/plan-vacation`）。
- **命令面板**：可搜索的提示词入口。
- **专用按钮**：高频提示词做成按钮。
- **右键菜单**：根据上下文推荐相关提示词。

通过 API 直接连 MCP 服务器时，提示词会暴露为 `/mcp__<服务器名>__<提示词名>` 形式的命令。

## 4.5 提示词 vs 系统提示 vs CLAUDE.md

容易混淆，对比一下（呼应第一册）：

| | MCP Prompt | 系统提示（API） | CLAUDE.md |
|---|---|---|---|
| 谁触发 | 用户显式选 | 每次请求都在 | 每次会话自动加载 |
| 来自哪 | MCP 服务器 | 你的代码 | 项目文件 |
| 适合 | 可复用的任务模板，跨宿主共享 | 全局角色和行为 | 项目背景知识 |
| 类比 | 斜杠命令 | 角色设定 | 入职文档 |

MCP 提示词的独特价值：**它跟着服务器走，可以跨宿主复用**。你为"工单系统"服务器写的 `/triage-ticket` 提示词，在任何连了这个服务器的宿主里都能用。

## 4.6 提示词如何编排工具和资源

提示词最强大的用法是**编排**——一个提示词模板引导模型组合使用服务器的工具和资源，完成一个完整工作流。

以第 3 章的工单服务器为例，加一个"分诊"提示词：

```python
@mcp.prompt()
def triage_ticket(ticket_id: str) -> str:
    """对一个工单进行分诊的完整流程。

    Args:
        ticket_id: 要分诊的工单 ID。
    """
    return f"""请对工单 #{ticket_id} 进行分诊：

1. 先读取 tickets://{ticket_id} 资源，了解工单完整内容
2. 用 search_tickets 工具搜索是否有相似的历史工单
3. 判断优先级（参考 tickets://schema 里的优先级定义）
4. 给出分诊结论：优先级、建议负责人、是否重复、处理建议

请按这个顺序执行。"""
```

用户触发 `/triage-ticket 1234`，模型就会按你编排的流程：读资源 → 调工具 → 综合判断。这把"一套标准操作流程"固化成了可复用、可分享的模板——正是第一册第 5 章"把流程沉淀为 Skill"思想的 MCP 版本。

## 4.7 三原语协同：完整图景

到这里三种原语都讲完了。它们如何协同（官方的旅行规划例子，类比到你的场景）：

```
用户触发提示词        →  /plan-trip Barcelona 7天
  （Prompt，用户控制）

提示词引导模型读资源  →  读 calendar://events、travel://preferences
  （Resource，应用控制）          （获取上下文：日程、偏好）

模型据上下文调工具    →  searchFlights()、bookHotel()、createCalendarEvent()
  （Tool，模型控制）             （执行动作：搜航班、订酒店、加日历）
```

一个任务，三种原语各尽其责：**提示词**给出流程框架（用户选），**资源**提供决策所需的上下文（应用给），**工具**执行实际动作（模型调）。这就是 MCP 服务器设计的完整图景——你写服务器时，就是在决定每个能力归到哪一类。
