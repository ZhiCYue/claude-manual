# 第 5 章 · 协作规范的制定

AI 进入团队后,协作规范要更新——尤其是 AI 生成代码的审查纪律和责任归属。这一章讲怎么定 PR、commit、分支规范,以及 AI 时代的代码审查。这些规范**对所有人、所有工具统一适用**。

---

## 5.1 AI 时代,协作规范要回答的新问题

传统协作规范(分支、PR、审查)依然有效,但 AI 让几个问题变得尖锐:

- AI 能快速生成大量代码——**审查跟得上吗?** 审查会不会变成走过场?
- AI 写的代码,**谁负责?** 出了 bug 是"AI 的锅"吗?
- AI 可能产出看起来对、实则有微妙问题的代码——**怎么防"形似而神不似"?**
- 大量 AI 生成的 PR,**怎么保持代码库的一致性和可维护性?**

核心答案就一条,也是第二条硬规则:**AI 写的,人负责。** 围绕这条,展开下面的规范。

## 5.2 第一原则:AI 生成 ≠ 免审

**最重要的规范:AI 生成的代码,和人手写的代码,走完全一样的审查流程——甚至更严。**

- 提交者对 AI 生成的代码**负全责**,就像自己写的一样。"这是 AI 写的"不是任何问题的借口。
- AI 代码**必须经人审查**才能合并。不能因为"AI 写的应该没问题"就跳过。
- 反而要警惕 AI 代码特有的问题(5.4 节),审得更仔细。

把这条写进 CONTRIBUTING.md,作为团队的明确共识:

```markdown
## AI 生成代码的责任
- 你提交的任何代码,无论是否由 AI 生成,你都对它负全责。
- AI 生成的代码与手写代码走相同的审查和测试流程,不享受任何豁免。
- 提交前你必须完整理解你提交的每一行——无法解释的代码不要提交。
```

## 5.3 PR、commit、分支规范

这些是工具中立的团队基础规范,AI 不改变它们,只是让"自动化生成"更容易(第一册/第 3 章的自定义命令可以辅助 Claude Code 用户遵守)。

### 分支策略

```
main           受保护,只能通过 PR 合入,必过 CI + 审查
feature/xxx    功能分支
fix/xxx        修复分支
```

用分支保护(branch protection)硬性强制:不准直推 main、PR 必过 CI、必有至少 1 个审查。**这是工具中立层的强制**(第 4 章),对所有人生效。

### Commit 规范

推荐 Conventional Commits(`feat:` / `fix:` / `docs:` / `refactor:` …),让历史可读、可自动生成 changelog:

```
feat(auth): 支持手机号登录
fix(payment): 修复高并发下的重复扣款
```

用 commitlint(git hook,工具中立)强制格式——对所有人生效。

### PR 规范

PR 描述模板(放 `.github/pull_request_template.md`,对所有 PR 生效):

```markdown
## 改了什么
（简述）

## 为什么
（背景/动机）

## 怎么测的
（测试方法,跑了哪些测试）

## AI 辅助说明
- [ ] 本 PR 含 AI 生成的代码
- [ ] 我已完整理解并能解释所有提交的代码
```

最后那个"AI 辅助说明"是 AI 时代的新增项——不是为了歧视 AI 代码,是为了让审查者知道哪些部分可能需要多留意,以及确认提交者真的理解了代码。

## 5.4 AI 代码审查:多留意什么

AI 生成的代码有一些特有的风险点,审查时重点看(这些与具体是哪个 AI 工具/模型无关):

| 风险 | 表现 | 怎么审 |
|---|---|---|
| **形似神不似** | 代码看起来对、跑得通,但逻辑有微妙错误 | 别只看"能不能跑",看逻辑和边界是否真的对 |
| **过度工程** | 加了没要求的抽象、配置、防御性代码 | 问"这个抽象/检查是必要的吗",砍掉多余的(第四册 9.6) |
| **应付测试** | 硬编码让测试通过,而非真正解决问题 | 看实现是不是通用解,还是只对测试用例有效(第四册 9.7) |
| **幻觉的 API** | 调用了不存在的函数/参数,或用法过时 | 验证用到的 API 真实存在、用法正确 |
| **遗漏的上下文** | 没考虑到代码库里别处的约定或依赖 | 看是否符合本项目的分层规则、命名约定 |
| **复制粘贴的不一致** | 多处相似代码,改了一处漏了别处 | 全局看一致性 |

> **审查心法**:AI 让"生成代码"变快,但"理解和验证代码"的责任没变。审查 AI 代码的本质,是确认**提交者真的理解了它、它真的对、它真的符合团队约定**。审查者不该因为"是 AI 写的"而放松,也不该因此而苛刻——标准和审人写的代码一样。

## 5.5 让审查跟得上:控制 PR 大小

AI 能快速生成大量代码,如果不加约束,会出现"500 行的 PR,审查者扫一眼就批了"——审查名存实亡。应对:

- **小 PR**:一个 PR 做一件事,控制在可认真审查的大小。这条对 AI 代码尤其重要——大的 AI 生成 PR 极易被走过场。
- **要求提交者自审**:开 PR 前自己完整过一遍 diff(第 1.7 节)。第一道审查是提交者自己。
- **让 AI 辅助审查,但人做决定**:可以用 `/code-review`(第一册)或开新会话让 AI 审一遍,但**最终判断和责任在人**。AI 审 AI 是辅助,不是终审。

## 5.6 用 AI 辅助遵守规范(Claude Code 层)

规范本身是工具中立的,但 Claude Code 用户可以用配置层辅助自己遵守(第 3.3 节):

- **`/commit` 命令**:让 Claude 按 Conventional Commits 生成规范的 commit message。
- **`/pr` 命令**:按团队 PR 模板生成 PR,自动跑测试和 lint。
- **`code-reviewer` 子代理**(第一册第 8 章):提交前自审一遍。

这些是给 Claude Code 用户的便利,帮他们更容易遵守工具中立的团队规范。用别的工具的人有他们自己的辅助方式,或手动遵守——规范对所有人一样,辅助手段可以各异(第 4 章)。

## 5.7 协作规范检查清单

- [ ] CONTRIBUTING.md 明确"AI 生成代码,提交者负全责,不豁免审查"
- [ ] 分支保护:不直推 main、PR 必过 CI、必有审查(工具中立强制)
- [ ] Commit 规范 + commitlint 强制(工具中立)
- [ ] PR 模板含"怎么测的"和"AI 辅助说明"
- [ ] 审查者知道 AI 代码的特有风险点(形似神不似、过度工程、幻觉 API……)
- [ ] PR 控制在可认真审查的大小
- [ ] 提交者开 PR 前自审完整 diff
- [ ] (Claude Code 用户)可用 /commit、/pr、code-reviewer 辅助遵守
