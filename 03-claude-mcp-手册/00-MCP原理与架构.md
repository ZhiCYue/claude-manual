# 第 0 章 · MCP 原理与架构

这一章不写一行服务器代码，只建立准确的协议心智模型。理解了它，后面每一章都是顺理成章的推论。

---

## 0.1 MCP 是什么、解决什么问题

MCP（Model Context Protocol，模型上下文协议）是 Anthropic 发起、现已成为行业标准的**开放协议**。它解决一个具体的工程问题：

> **如何让任意 AI 应用，以统一的方式，接入任意外部系统？**

没有 MCP 之前，每个 AI 应用要接入每个外部系统，都得写一套专属的集成代码——N 个应用 × M 个系统 = N×M 套胶水代码。MCP 把它变成 N + M：每个应用实现一次 MCP 客户端，每个系统实现一次 MCP 服务器，然后任意组合即插即用。

你写一个"查询公司工单系统"的 MCP 服务器，它立刻能被 Claude Code、Claude 桌面端、Cursor、VS Code、你自己的程序……所有 MCP 宿主使用。这就是开放协议的杠杆。

> MCP **只**管"交换上下文的协议"，不管 AI 应用怎么用 LLM、怎么管理上下文。它是一根标准的管道，不规定管道两头各自做什么。

## 0.2 三个角色：宿主、客户端、服务器

MCP 是 **客户端-服务器** 架构。三个角色务必分清：

```
┌─────────────────────────────────────────────┐
│  MCP 宿主（Host）—— AI 应用                    │
│  如 Claude Code / Claude 桌面端 / VS Code      │
│                                               │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│   │ 客户端 1  │  │ 客户端 2  │  │ 客户端 3  │   │
│   └────┬─────┘  └────┬─────┘  └────┬─────┘   │
└────────┼─────────────┼─────────────┼─────────┘
         │ 专属连接      │             │
         ▼             ▼             ▼
   ┌──────────┐  ┌──────────┐  ┌──────────┐
   │ 服务器 A  │  │ 服务器 B  │  │ 服务器 C  │
   │（文件系统）│  │（数据库） │  │（Sentry） │
   │   本地    │  │   本地   │  │   远程   │
   └──────────┘  └──────────┘  └──────────┘
```

| 角色 | 是什么 | 例子 |
|---|---|---|
| **宿主（Host）** | 协调和管理一个或多个客户端的 AI 应用 | Claude Code、Claude 桌面端、VS Code |
| **客户端（Client）** | 宿主内部、与某个服务器维持连接的组件 | 宿主为每个服务器创建一个客户端 |
| **服务器（Server）** | 向客户端提供上下文的程序 | 你要写的就是它 |

关键认识：

1. **宿主为每个服务器创建一个专属客户端。** 连 3 个服务器就有 3 个客户端，各维护一条独立连接。
2. **"服务器"指提供上下文的程序，不论它跑在哪。** 本地（用 stdio 传输，和宿主同机）或远程（用 HTTP 传输，跑在云上）都叫服务器。
3. **你写服务器，几乎不碰客户端和宿主。** 宿主和客户端由 AI 应用厂商实现；你只管把你的系统封成一个符合协议的服务器。

## 0.3 两层架构：数据层与传输层

MCP 由两层构成。理解这个分层，你才知道"协议内容"和"怎么传"是两件可以分开的事。

```
┌──────────────────────────────────────┐
│  数据层（内层）                          │
│  JSON-RPC 2.0 协议                      │
│  · 生命周期管理（初始化、能力协商）        │
│  · 三大原语（Tools/Resources/Prompts）   │
│  · 通知（实时更新）                       │
├──────────────────────────────────────┤
│  传输层（外层）                          │
│  · stdio（本地进程，标准输入输出）         │
│  · Streamable HTTP（远程，HTTP+SSE）     │
│  · 鉴权（OAuth、Bearer token 等）         │
└──────────────────────────────────────┘
```

- **数据层**：定义"说什么"。基于 [JSON-RPC 2.0](https://www.jsonrpc.org/)——客户端和服务器互发请求、互相响应；不需要响应时发通知。这是协议的核心，三大原语都在这一层。
- **传输层**：定义"怎么传"。同样的 JSON-RPC 消息，可以走 stdio（本地）也可以走 HTTP（远程）。传输层把通信细节从协议里抽离，所以**换传输方式不改业务逻辑**（第 5 章详解）。

好消息：**SDK 把这两层都封装好了。** 你用 FastMCP 写服务器时，几乎感觉不到 JSON-RPC 和传输的存在——你只写业务函数，SDK 负责协议。但理解它们存在，能帮你看懂调试日志、选对传输方式、排查连接问题。

## 0.4 生命周期：能力协商

MCP 是**有状态**协议，连接建立时有一次握手，叫**能力协商**（capability negotiation）。简化的初始化序列：

```
1. 客户端 → 服务器：initialize 请求
   "我支持协议版本 2025-06-18，我有 elicitation 能力"

2. 服务器 → 客户端：initialize 响应
   "我也支持这个版本，我提供 tools（且工具列表会变化时会通知你）和 resources"

3. 客户端 → 服务器：notifications/initialized 通知
   "握手完成，我准备好了"
```

握手做三件事：

1. **协议版本协商**——确保双方版本兼容，不兼容就终止连接。
2. **能力发现**——双方声明各自支持什么（哪些原语、是否支持通知）。这样就不会去调用对方不支持的操作。
3. **身份交换**——交换名字和版本，用于调试。

你写服务器时，**声明能力这一步由 SDK 自动完成**——你定义了工具，SDK 就在握手时声明"我有 tools 能力"。

## 0.5 三大原语：MCP 的核心

原语（primitives）定义"客户端和服务器能给对方提供什么"。**服务器**能暴露三种原语——这是整本书的主线：

| 原语 | 是什么 | 谁控制 | 协议方法 |
|---|---|---|---|
| **Tools（工具）** | AI 能主动调用的可执行函数（查数据库、调 API、改文件） | **模型** | `tools/list`、`tools/call` |
| **Resources（资源）** | 提供上下文的只读数据源（文件内容、数据库记录、API 响应） | **应用** | `resources/list`、`resources/read` |
| **Prompts（提示词）** | 帮助构建交互的可复用模板（系统提示、few-shot 例子） | **用户** | `prompts/list`、`prompts/get` |

**"谁控制"是区分三者的钥匙**——这个差别决定了一个能力该做成哪种原语：

- **工具是模型控制的**：模型根据对话上下文，自己决定何时调用哪个工具。比如用户问"巴黎天气"，模型自动调 `get_weather`。
- **资源是应用控制的**：由宿主应用决定何时把哪些资源喂给模型。比如应用把当前打开的文件作为资源加入上下文。
- **提示词是用户控制的**：需要用户显式触发，通常呈现为斜杠命令。比如用户输入 `/plan-vacation` 选用"规划假期"模板。

> **第 2、3、4 章分别深入这三者。** 现在只要记住：纠结一个能力该怎么做时，问"谁该控制它"——模型、应用、还是用户。

每种原语都有发现（`*/list`）和使用（`*/call`、`*/read`、`*/get`）的方法。客户端先用 `list` 发现有什么，再用对应方法使用。这个设计让能力列表可以**动态变化**。

### 客户端原语（了解即可）

反过来，**客户端**也能向服务器暴露能力，让服务器作者写出更丰富的交互：

- **Sampling（采样）**：服务器请求客户端的 LLM 做一次补全。这样服务器作者能用上语言模型，又不必在服务器里塞一个 LLM SDK、保持模型无关。
- **Elicitation（征询）**：服务器向用户请求更多信息或确认某个操作。
- **Logging（日志）**：服务器向客户端发日志用于调试。

这些是进阶能力，前几章不涉及。

## 0.6 通知：实时更新

JSON-RPC 的"通知"是不需要响应的消息。MCP 用它做实时同步——典型场景：服务器的工具列表变了（新增功能、工具下线），服务器主动发 `notifications/tools/list_changed`，客户端收到后重新拉取工具列表。

这让 MCP 连接是**动态**的：工具可以随服务器状态、外部依赖、用户权限的变化而增减，客户端不必轮询。前提是服务器在握手时声明了 `"listChanged": true`。

## 0.7 一次完整交互长什么样（JSON-RPC 视角）

把上面串起来，一次"模型调用工具"的底层消息流（你平时看不到，SDK 帮你处理，但理解它有助于调试）：

```
① 初始化握手（0.4 节）

② 客户端发现工具
   → {"method": "tools/list"}
   ← {"result": {"tools": [{"name": "weather_current",
                            "description": "...",
                            "inputSchema": {...}}]}}

③ 模型决定调用，客户端执行
   → {"method": "tools/call",
      "params": {"name": "weather_current",
                 "arguments": {"location": "San Francisco"}}}
   ← {"result": {"content": [{"type": "text",
                              "text": "旧金山当前 68°F，多云"}]}}

④ （可选）服务器工具变化时发通知
   → {"method": "notifications/tools/list_changed"}
```

注意工具返回的是一个 **content 数组**——支持文本、图片、资源等多种内容类型。你写工具时返回的就是这个结构（第 2 章）。
