# 附录 · 速查、模板库与反模式

## A. 技法速查（按优先级）

遇到问题，按这个顺序往下试：

| # | 技法 | 何时用 | 章节 |
|---|---|---|---|
| 1 | 清晰、直接、具体 | 永远先做这个，解决 80% 问题 | 1 |
| 2 | 给上下文和动机 | 指令看似武断时，加"因为……" | 1 |
| 3 | 用示例 | 格式/风格/语气不稳定 | 2 |
| 4 | XML 结构化 | prompt 混杂多类内容、有变量输入 | 3 |
| 5 | 设定角色 | 需要特定语气/专业视角 | 4 |
| 6 | 让它思考 | 推理/数学/复杂任务 | 5 |
| 7 | 控制输出 | 格式/详略不对 | 6 |
| 8 | 链式分解 | 单 prompt 搞不定，或要看中间结果 | 7 |
| 9 | 长上下文技巧 | 大文档/多文档 | 8 |
| 10 | agentic / 模型专属调优 | 构建 agent、用最新模型 | 9, 10 |

## B. 核心原则一句话清单

- **把 Claude 当聪明但全新的同事**——解释得越精确，结果越好。
- **新同事检验法**——prompt 给不懂背景的人看，他困惑则 Claude 也困惑。
- **想要什么就明说**——想要"超额表现"要明确要求，别指望推断。
- **给理由能泛化**——解释为什么，模型会举一反三。
- **说"做什么"而非"别做什么"**——正面描述目标更可靠。
- **示例 3–5 个，相关+多样+结构化**。
- **数据在前、问题在后**——长上下文质量 +30%。
- **不是所有问题靠 prompt**——延迟/成本常换模型更省事。
- **评估先行，一次改一处**——别凭感觉调。
- **新模型更字面更自主**——老 prompt 的过激补丁该拆了。

## C. 可复制的 prompt 片段库

> 这些是起点，按你的场景用评估驱动地调（第 0 章）。

**控制详略——少 markdown 多散文**（第 6.4）：
```
写长文时用清晰流畅的散文，完整段落和句子。markdown 主要保留给行内代码、
代码块和简单标题。不要用项目符号罗列，把要点融进句子。绝不输出一连串过短的项目符号。
```

**让模型动手**（第 9.1）：
```
<默认行动>
默认实施修改而非只建议。意图不明确时，推断最有用的行动并执行，
用工具发现缺失细节而非猜测。
</默认行动>
```

**让模型谨慎**（第 9.1）：
```
<不要先于指令行动>
除非明确要求，不要贸然实现或改文件。意图模糊时默认提供信息、调研、给建议。
</不要先于指令行动>
```

**最大化并行工具**（第 9.2）：
```
如果打算调用多个工具且无依赖，就并行发起所有独立调用。读多个文件时并行读。
但有依赖（用前一步输出作参数）时顺序调用。绝不用占位符或猜测缺失参数。
```

**危险操作先确认**（第 9.3）：
```
考虑行动的可逆性。本地可逆的行动（编辑文件、跑测试）放手做；
难撤销、影响共享系统、或破坏性的行动（删除、git push --force、对外发布），先问用户。
遇到障碍不要用破坏性操作当捷径。
```

**控制子代理**（第 9.4）：
```
任务能并行、需隔离上下文、或是独立工作流时用子代理。
简单任务、顺序操作、单文件编辑、需跨步保持上下文时，直接做。
```

**长任务不提前收尾**（第 9.5）：
```
上下文接近上限时会自动压缩，让你能从中断处继续。不要因 token 预算提前停止任务。
接近上限时把进度存到记忆再刷新。完整完成任务，绝不人为提前停止。
```

**防过度工程**（第 9.6）：
```
避免过度工程。只做明确要求或确实必要的修改。不加超出要求的功能/重构/抽象，
不给没改的代码加文档，不为不可能的场景加校验，不为一次性操作建抽象。
```

**通用解而非应付测试**（第 9.7）：
```
用标准工具写通用解，对所有合法输入正确，不只是测试用例。不硬编码值。
测试是验证正确性，不是定义解。任务不合理或测试有错就告诉我，不要绕过。
```

**防幻觉**（第 9.8）：
```
<调查后再回答>
绝不臆测没打开过的代码。用户引用具体文件时，回答前必须先读。
调查前不对代码下断言，除非确定——给有据、无幻觉的回答。
</调查后再回答>
```

**让模型简洁地叙述**（第 10.4）：
```
工具调用间默认保持安静。只在有发现、改方向、遇阻时写文字，每种一句话。
不叙述常规动作。完成时一两句说明结果。
```

**纯文本数学**（第 6.5）：
```
回复用纯文本，不要 LaTeX/MathJax 或任何标记（\( \)、$、\frac{}{}）。
除法用 "/"、乘法用 "*"、指数用 "^"。
```

## D. 反模式清单（别这么做）

| ❌ 反模式 | 为什么糟 | ✅ 改法 |
|---|---|---|
| "处理一下""优化优化""好看点" | 模糊，模型不知道目标 | 具体描述要什么（第 1 章） |
| 说"别用 X" | 让模型去想 X、反而强化 | 说"用 Y"（第 6.1 节） |
| 只给 1 个示例 | 模式不清，易过拟合 | 3–5 个，多样（第 2 章） |
| 问题放在长文档前面 | 质量低 30% | 数据在前、问题在后（第 8.1 节） |
| 凭感觉改 prompt | 不知改动是好是坏 | 评估驱动、一次一处（第 0 章） |
| 一上来堆复杂技巧 | 浪费精力 | 先把话说清楚（第 1 章） |
| 用 prefill 强制格式 | 4.6+ 会 400 | 结构化输出（第 6.8 节） |
| 用 budget_tokens | Fable5/4.7/4.8 会 400 | adaptive thinking + effort |
| 在新模型上用"CRITICAL/必须" | 过度触发 | 正常陈述（第 9.1、10.2 节） |
| 夸张空洞的角色（"宇宙第一……"） | 不传递可用信息 | 真实具体的角色（第 4.7 节） |
| 所有任务都拆链 | 多此一举 | 需要看中间结果才拆（第 7.2 节） |
| 把所有问题都当 prompt 问题 | 有些该换模型 | 先判断（第 0.3 节） |

## E. 迁移到新模型检查清单

从老模型（4.5 之前）迁移时，逐项检查 prompt：

- [ ] 松绑过激的工具/技能语言（"CRITICAL/必须/如有疑问就用" → 正常陈述）
- [ ] 拆掉"反偷懒"脚手架（"务必详尽"等，A/B 测去掉后的效果）
- [ ] 拆掉强制进度汇报脚手架（新模型自己会做）
- [ ] prefill → 结构化输出
- [ ] `budget_tokens` → adaptive thinking + effort
- [ ] 重新扫 effort（起点 high，按评估选，别反射性拉满）
- [ ] 太简洁就加"完成后总结"
- [ ] 太啰嗦就加"默认安静"
- [ ] 过度工程就加约束
- [ ] 危险操作没确认就加"自主与安全"引导
- [ ] 用评估集验证每处改动
