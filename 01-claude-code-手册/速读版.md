## 第 0 章 · 总览：Claude Code 是什么、它如何工作

### 0.1 定位

Claude Code 是 Anthropic 官方的**终端 AI 编程代理（Agent）**。它和网页版聊天最大的区别是：它不只「回答问题」，它能**直接动手**——读写你的文件、执行 shell 命令、搜索代码库、跑测试、提交 git、调用浏览器。

你用自然语言描述目标，它自己拆解步骤、调用工具、检查结果、修正错误，直到完成。

### 0.2 核心原理：Agent Loop（代理循环）

理解了这个循环，后面所有高级功能都好理解了：

```
你输入指令
   ↓
模型思考：要完成这个目标，下一步该做什么？
   ↓
模型发出「工具调用」（如：读取 main.py / 执行 npm test）
   ↓
Claude Code 程序真正执行该操作，把结果（文件内容/命令输出）喂回给模型
   ↓
模型根据结果继续思考 → 再调用工具 → ……（循环）
   ↓
目标完成，输出总结，把控制权交还给你
```

几个关键认识：

1. **模型本身只会「说话」**。真正读文件、跑命令的是 Claude Code 这个程序（称为 harness，「执行框架」）。模型输出一段结构化的「我想调用 Bash 工具，参数是 `npm test`」，harness 解析并执行，再把输出文本塞回对话。
2. **一切都发生在「上下文窗口」里**。模型每次思考时看到的是：系统提示词 + 你的项目记忆（CLAUDE.md）+ 整段对话历史 + 所有工具结果。上下文是有限资源（约 20 万 token），所以「管理上下文」是高阶用法的核心主题（第 2、3、8 章）。
3. **权限是 harness 控制的，不是模型自觉**。模型每次想执行危险操作，harness 会先问你。这就是为什么权限配置（第 4 章）和 Hooks（第 6 章）是「确定性」的——它们是程序逻辑，不依赖模型听不听话。

### 0.3 内置工具一览

模型可调用的主要工具（理解它们有助于看懂 Claude Code 的工作过程）：

| 工具 | 作用 |
|---|---|
| `Read` / `Write` / `Edit` | 读文件、新建文件、精确字符串替换式修改 |
| `Bash` | 执行 shell 命令（git、npm、python……） |
| `Glob` / `Grep` | 按文件名模式找文件、按正则搜内容 |
| `WebSearch` / `WebFetch` | 搜索网页、抓取网页内容 |
| `Agent` | 派生子代理去独立完成任务（第 8 章） |
| `TodoWrite` | 维护任务清单，你能看到它的计划进度 |

## 第 1 章 · 入门：安装、启动与基本交互

### 1.1 安装与登录

```bash
# 安装（macOS/Linux 原生安装脚本）
curl -fsSL https://claude.ai/install.sh | bash

# 或通过 npm
npm install -g @anthropic-ai/claude-code

# 验证
claude --version    # 例：2.1.174

# 启动后用 /login 登录（Claude.ai 订阅账号或 API Key）
```

### 1.2 启动方式

```bash
cd 你的项目目录
claude                  # 进入交互式会话（最常用）
claude "解释这个项目的结构"   # 带初始指令启动
claude -c               # 继续本目录最近一次会话（continue）
claude --resume         # 弹出列表，挑选历史会话恢复
```

> **原理**：每次会话的完整记录存在 `~/.claude/projects/<项目路径>/` 下。`-c`/`--resume` 就是把历史对话重新装回上下文窗口，所以恢复后它「记得」之前聊过的一切。

### 1.3 基本交互：五个最重要的输入技巧

| 技巧 | 写法 | 说明 |
|---|---|---|
| 引用文件 | `@src/main.py` | 输入 `@` 后有自动补全，文件内容会被精准地放入上下文 |
| 直接跑命令 | `! git status` | `!` 前缀让命令在本会话执行，**输出进入对话**，Claude 也能看到 |
| 快速记忆 | `# 本项目用 pnpm 不要用 npm` | `#` 前缀把这句话存进 CLAUDE.md（第 3 章） |
| 中断 | `Esc` | 随时打断 Claude 正在做的事，补充或纠正指令 |
| 回溯 | `Esc` 连按两下 | 跳回之前的某条消息重新编辑——相当于对话的「时光回退」 |

### 1.4 第一次实战：体验完整的 Agent Loop

在任意项目里试试这条指令：

```
> 找出这个项目里所有的 TODO 注释，按文件整理成一个列表给我
```

观察它的行为：先用 `Grep` 搜索 `TODO`，再 `Read` 几个文件确认上下文，最后输出整理好的列表。这就是 0.2 节讲的循环在真实运转。

再试一个会「动手」的：

```
> 给 utils.py 里的 parse_date 函数写单元测试，写完跑一遍确认通过
```

它会：读函数 → 写测试文件 → 执行 `pytest` → 如果失败就修 → 直到通过。**写完自动验证**是 Claude Code 和「聊天式 AI」的本质区别。

### 1.5 给指令的技巧（很重要）

- **给目标，不给步骤**：「把登录接口的报错信息改成中文」优于「打开 auth.py 第 45 行把字符串改成……」。
- **给验收标准**：「……改完确保 `npm test` 全绿」——它就会自己跑测试自我验证。
- **范围越明确越好**：「只改 `src/api/` 目录下的文件」可以避免它误伤其他代码。
- **复杂任务先让它出计划**：见第 4 章 Plan Mode。

## 第 2 章 · 内置斜杠命令

在会话中输入 `/` 即可看到全部命令。下面按使用频率分组。

### 2.1 日常必用

| 命令 | 作用 |
|---|---|
| `/help` | 帮助 |
| `/clear` | **清空对话上下文**，开始全新任务（高手用得最多的命令之一） |
| `/compact` | 把对话历史压缩成摘要，腾出上下文空间 |
| `/model` | 切换模型（Fable / Opus / Sonnet / Haiku） |
| `/resume` | 恢复历史会话 |
| `/rewind` | 回退到之前的检查点（含代码改动的回退） |

### 2.2 项目与配置

| 命令 | 作用 |
|---|---|
| `/init` | 让 Claude 分析项目并生成 CLAUDE.md（第 3 章） |
| `/memory` | 编辑记忆文件 |
| `/permissions` | 查看/修改权限规则（第 4 章） |
| `/config` | 打开设置（主题、模型等） |
| `/agents` | 管理子代理（第 8 章） |
| `/mcp` | 管理 MCP 服务器（第 7 章） |
| `/hooks` | 配置 Hooks（第 6 章） |
| `/statusline` | 自定义底部状态栏 |

### 2.3 诊断与统计

| 命令 | 作用 |
|---|---|
| `/context` | **可视化展示当前上下文窗口的占用情况**（学习上下文管理的利器） |
| `/cost` | 查看本次会话的 token 消耗与费用 |
| `/usage` | 查看订阅额度使用情况 |
| `/doctor` | 自检安装与配置是否健康 |
| `/status` | 当前会话状态（模型、目录、账号） |
| `/export` | 导出对话记录 |

### 2.4 原理：为什么要 `/clear` 和 `/compact`？

上下文窗口是模型的「工作记忆」，约 20 万 token。随着对话变长：

1. **变慢变贵**：每轮都要重新处理整段历史。
2. **质量下降**：无关的旧信息会干扰当前任务的注意力。

所以高手的习惯是：**一个任务一个会话**。任务完成 → `/clear` → 开始下一个任务。如果任务很长必须延续，用 `/compact` 把历史压缩成摘要（你还可以指定保留重点：`/compact 保留所有关于数据库迁移的讨论`）。

用 `/context` 随时观察占用——你会直观看到系统提示、CLAUDE.md、对话历史、工具输出各占多少。

> **进阶技巧**：上下文快满时 Claude Code 会自动压缩（auto-compact），但主动管理永远比被动压缩效果好，因为你比它更清楚什么信息值得保留。

## 第 3 章 · 上下文与记忆：CLAUDE.md

### 3.1 是什么

CLAUDE.md 是放在项目里的「项目说明书」，**每次会话启动时自动读入上下文**。它解决一个根本问题：模型每次会话都是「失忆」重启的，CLAUDE.md 就是它的长期记忆。

### 3.2 三个层级

| 位置 | 作用范围 | 典型内容 |
|---|---|---|
| `~/.claude/CLAUDE.md` | 你机器上的**所有**项目 | 个人偏好：「回答用中文」「commit message 用英文」 |
| `项目根目录/CLAUDE.md` | 本项目（可提交进 git，团队共享） | 构建命令、架构约定、代码风格 |
| `子目录/CLAUDE.md` | 进入该子目录工作时按需加载 | 该模块的特殊约定 |

### 3.3 快速上手

```bash
# 在项目根目录的会话里执行：
/init        # Claude 会扫描项目，自动生成一份 CLAUDE.md 初稿
```

平时随手积累：

```
# 这个项目跑测试要用 make test，不要直接用 pytest
```

输入以 `#` 开头的内容，Claude Code 会问你存到哪一级 CLAUDE.md。

### 3.4 一份好的 CLAUDE.md 长什么样

```markdown
# 项目：订单服务

## 常用命令
- 启动开发环境：`docker compose up -d && make dev`
- 跑测试：`make test`（不要直接调用 pytest，环境变量在 Makefile 里）
- 数据库迁移：`make migrate`

## 架构要点
- API 层在 `src/api/`，业务逻辑在 `src/services/`，不要在 API 层写业务逻辑
- 所有时间统一用 UTC，存储为 ISO 8601 字符串

## 约定
- 新代码必须有类型注解
- 提交前必须通过 `make lint`
```

**编写原则**：

- **写「无法从代码推断的事」**：命令、约定、坑。代码结构本身 Claude 自己能看。
- **保持精简**：CLAUDE.md 每次会话都占上下文，写 500 行等于每次都浪费几千 token 且稀释重点。
- **像写给新同事的入职文档**，而不是写给机器的配置文件。

> **原理**：CLAUDE.md 的内容被拼接到系统提示附近，处于模型注意力的「高权重位置」，所以这里的指令比对话中途说的话更容易被持续遵守。这也是为什么「反复需要提醒的事」都应该沉淀进 CLAUDE.md。

## 第 4 章 · 权限系统与 Plan Mode

### 4.1 四种权限模式（按 `Shift+Tab` 循环切换）

| 模式 | 行为 | 适用场景 |
|---|---|---|
| **default** | 写文件、跑命令前逐一询问 | 默认；不熟悉的项目 |
| **acceptEdits** | 自动接受文件编辑，命令仍询问 | 信任度较高的常规开发 |
| **plan**（Plan Mode） | **只读**：只能看代码、做调研，不能改任何东西 | 复杂任务先出方案 |
| **bypassPermissions** | 全部放行不询问 | 仅限隔离环境（容器/沙箱） |

### 4.2 Plan Mode：高手的起手式

复杂任务（重构、新功能、不熟悉的代码库）的推荐流程：

```
1. Shift+Tab 切到 plan 模式
2. > 我想把用户认证从 session 改成 JWT，请调研现状并给出实施方案
3. Claude 只读地探索代码，输出一份分步计划
4. 你审阅计划，不满意就继续讨论修改
5. 批准计划 → 自动退出 plan 模式 → 开始动手执行
```

> **原理**：「先计划后执行」之所以效果好，是因为执行中途的模型容易「一条道走到黑」。在只读模式里它被迫先建立全局理解，而你在动手前审阅计划，纠错成本是最低的（改一句话 vs 回滚一堆代码）。

### 4.3 精细化权限：settings.json

权限规则写在三个层级的文件里（后者覆盖前者）：

```
~/.claude/settings.json              # 用户全局
项目/.claude/settings.json           # 项目共享（提交进 git）
项目/.claude/settings.local.json     # 项目个人（git 忽略）
```

示例——允许常用安全命令免询问，永久禁止危险操作：

```json
{
  "permissions": {
    "allow": [
      "Bash(git status)",
      "Bash(git diff *)",
      "Bash(npm test *)",
      "Bash(npm run lint)",
      "Read(**)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Read(.env)",
      "Read(secrets/**)"
    ],
    "ask": [
      "Bash(git push *)"
    ]
  }
}
```

规则语法：`工具名(参数模式)`，`*` 是通配符。`deny` 优先级最高——**即使模型被诱导，harness 也会在程序层面拦截**，这是确定性的安全保障。

> **省心技巧**：会话里直接用 `/permissions` 命令交互式管理；也可以在被询问时选「Always allow」自动写入 settings.local.json。

## 第 5 章 · 自定义斜杠命令与 Skills

### 5.1 最简单的自定义命令

在项目里建一个 markdown 文件，就得到一个斜杠命令：

```bash
mkdir -p .claude/commands
```

`.claude/commands/fix-issue.md`：

```markdown
---
description: 根据 GitHub issue 编号修复问题
---

请处理 GitHub issue #$ARGUMENTS：
1. 用 `gh issue view $ARGUMENTS` 查看 issue 内容
2. 在代码库中定位相关代码
3. 实现修复，并补充对应的测试
4. 跑全量测试确认通过
5. 创建分支并提交，commit message 引用该 issue 编号
```

使用：

```
/fix-issue 123
```

`$ARGUMENTS` 会被替换成你输入的参数（也可用 `$1`、`$2` 取分项参数）。

放在 `~/.claude/commands/` 下则是全局命令，所有项目可用。

### 5.2 Skills：带资源的「重型命令」

Skill 是命令的升级形态——一个目录，除了主指令文件 `SKILL.md`，还可以附带脚本、模板、参考文档：

```
.claude/skills/release/
├── SKILL.md          # 主指令（frontmatter 里写 name 和 description）
├── checklist.md      # 发布检查清单（指令中引用）
└── bump_version.py   # 配套脚本（指令中让 Claude 执行它）
```

`SKILL.md` 示例：

```markdown
---
name: release
description: 执行一次完整的版本发布流程
---

执行发布流程：
1. 阅读 checklist.md 并逐项确认
2. 运行 `python bump_version.py` 更新版本号
3. 更新 CHANGELOG.md
4. 打 tag 并推送
```

> **原理**：Skill 采用「渐进式加载」——平时只有 name/description 占用极少量上下文；当任务匹配时才加载完整正文和附属文件。这让你可以积累几十个 Skill 而不撑爆上下文。**「把流程沉淀为 Skill」是团队使用 Claude Code 的核心进阶模式**：好的提示词不再依赖个人记忆，而是版本化、可共享的资产。

### 5.3 命令 vs CLAUDE.md vs Skill，怎么选？

- **每次会话都该知道的背景** → CLAUDE.md
- **手动触发的固定流程，一段话说得清** → 自定义命令
- **复杂流程，需要附带脚本/模板/文档** → Skill

## 第 6 章 · Hooks：给 AI 加上确定性的规则

### 6.1 解决什么问题

在 CLAUDE.md 里写「每次改完文件要跑 formatter」，模型**大概率**会照做，但不保证。Hooks 把这类要求变成**程序级保证**：在指定事件点自动执行你的 shell 命令，100% 触发。

### 6.2 事件点

| 事件 | 时机 | 典型用途 |
|---|---|---|
| `PreToolUse` | 工具执行**前** | 拦截危险命令、校验参数 |
| `PostToolUse` | 工具执行**后** | 自动格式化刚改过的文件 |
| `UserPromptSubmit` | 你提交输入时 | 注入额外上下文 |
| `Stop` | Claude 完成回合时 | 桌面通知「干完了」 |
| `SessionStart` | 会话启动时 | 加载环境信息 |

### 6.3 实例一：编辑 Python 文件后自动 format

`.claude/settings.json`：

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | { read f; [[ \"$f\" == *.py ]] && ruff format \"$f\"; } || true"
          }
        ]
      }
    ]
  }
}
```

> **原理**：hook 命令通过 **stdin 收到一份 JSON**（包含工具名、参数、结果等），所以上面用 `jq` 取出被编辑的文件路径。hook 的退出码也有含义：`PreToolUse` hook 退出码为 2 时会**阻止**该工具调用，并把 stderr 反馈给模型——这就是「程序否决 AI」的机制。

### 6.4 实例二：完成任务时发 macOS 通知

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"Claude 已完成任务\" with title \"Claude Code\"'"
          }
        ]
      }
    ]
  }
}
```

配好后你可以放心切去做别的事，干完了系统会弹通知。

配置入口：会话里输入 `/hooks` 有交互式界面，或直接编辑 settings.json。

### 6.5 判断标准：什么时候用 Hook？

问自己一句话：**「这件事是『最好做到』还是『必须做到』？」**
最好做到 → 写进 CLAUDE.md（提示性）；必须做到 → 写成 Hook（确定性）。

## 第 7 章 · MCP：连接外部世界

### 7.1 是什么

MCP（Model Context Protocol，模型上下文协议）是一个开放协议，让 Claude Code 接入外部系统——数据库、浏览器、Slack、Jira、内部 API……每接入一个 MCP 服务器，Claude 就多一批可调用的工具。

> **原理**：MCP 服务器是一个独立小程序，向 Claude Code 声明「我提供这些工具，参数格式如此」。harness 把这些工具描述加入模型可见的工具列表；模型调用时，harness 把请求转发给 MCP 服务器执行。本质上是**给 Agent Loop 插上可扩展的工具插槽**。传输方式有 stdio（本地子进程）和 HTTP/SSE（远程服务）两种。

### 7.2 常用操作

```bash
# 添加一个本地 stdio 服务器（以 Playwright 浏览器自动化为例）
claude mcp add playwright -- npx @playwright/mcp@latest

# 添加远程 HTTP 服务器
claude mcp add --transport http sentry https://mcp.sentry.dev/mcp

# 查看 / 删除
claude mcp list
claude mcp remove playwright
```

会话内用 `/mcp` 查看连接状态、进行 OAuth 授权。

作用域：`claude mcp add` 默认存为本人本项目可用；加 `--scope project` 会生成 `.mcp.json` 提交进 git 供团队共享；`--scope user` 全局可用。

### 7.3 实战示例：浏览器自动化

接入 Playwright MCP 后：

```
> 打开 localhost:3000，注册一个新账号，截图给我看注册成功页面
```

Claude 会驱动真实浏览器完成点击、填表、截图。配合开发流程的杀手级用法：

```
> 实现这个登录页的「记住我」功能，写完后用浏览器实际操作验证一遍
```

### 7.4 注意事项

- 每个 MCP 服务器的工具描述都占上下文，**只装真正用的**（用 `/context` 可以看到 MCP tools 的占用）。
- 第三方 MCP 服务器等于给 AI 开了新权限，装之前想清楚来源是否可信。

## 第 8 章 · 子代理（Subagents）

### 8.1 是什么

子代理是 Claude 派生出来的「分身」：拥有**独立的上下文窗口**、可单独限定工具和系统提示，去完成一个子任务，最后只把**结论**汇报回主对话。

### 8.2 为什么需要它（原理）

回顾第 2 章：上下文是稀缺资源。假设你要在大型代码库里搜索某个逻辑，直接搜会把几十个文件的内容全部塞进主对话，任务完成后这些「废料」还一直占着窗口。

子代理的解法：**脏活在分身的上下文里干，主对话只收一段干净的总结**。这是上下文管理的最高形态。

内置的两个常用子代理：

- **Explore**：只读搜索代理，适合「全库找 X」类任务
- **Plan**：架构规划代理，产出实施方案

使用方式很自然，直接说：

```
> 用 Explore 子代理找出整个仓库里所有处理支付回调的代码路径
```

### 8.3 自定义子代理

`/agents` 命令交互式创建，或手写文件 `.claude/agents/code-reviewer.md`：

```markdown
---
name: code-reviewer
description: 资深代码审查员。在代码写完需要审查时主动使用。
tools: Read, Grep, Glob, Bash
---

你是一位严格的代码审查员。审查最近的代码改动：
- 先运行 git diff 查看变更
- 重点检查：错误处理、边界条件、安全隐患、命名清晰度
- 按严重程度分级输出问题，每个问题给出具体修改建议
```

之后可以显式调用：

```
> 让 code-reviewer 审查一下我刚写的代码
```

frontmatter 中 `description` 写得越明确（尤其加上「主动使用」字样），Claude 越会在合适时机自动委派给它。`tools` 字段限制了它的能力——比如审查员只给只读工具，**它就在物理上不可能改你的代码**。

### 8.4 使用建议

- 子代理每次都是「冷启动」，不知道主对话聊了什么——**任务描述要自包含**。
- 适合：搜索调研、独立的审查/测试任务、可并行的批量工作。
- 不适合：需要与你反复交互的任务、依赖大量主对话上下文的任务。

## 第 9 章 · 无头模式与自动化（CI / 脚本 / 定时任务）

### 9.1 `claude -p`：把 Claude 变成一个 Unix 命令

`-p`（`--print`）让 Claude 非交互地执行一次任务并输出结果，于是它可以出现在任何脚本、管道、CI 里：

```bash
# 最简单：问完即走
claude -p "用一句话总结这个项目是做什么的"

# 管道：处理 stdin
cat error.log | claude -p "分析这份日志，找出根本原因"

# 结构化输出，配合 jq 做后续处理
claude -p "统计 src 下各语言的文件数" --output-format json | jq '.result'

# 限定可用工具（无人值守时的安全边界）
claude -p "修复所有 lint 错误" \
  --allowedTools "Read" "Edit" "Bash(npm run lint*)" \
  --permission-mode acceptEdits
```

> **原理**：无头模式跑的是同一个 Agent Loop，只是没有人在终端确认权限——所以必须用 `--allowedTools` / `--permission-mode` 预先声明边界，否则它遇到未授权操作就会停下。`--output-format stream-json` 可以流式拿到每一步工具调用事件，适合在程序里监控它的执行过程。

### 9.2 实战：GitHub Actions 里自动审 PR

```yaml
# .github/workflows/claude-review.yml（节选）
- name: Claude review
  run: |
    claude -p "审查本 PR 的代码变更（git diff origin/main...HEAD），
    重点看安全问题和逻辑错误，输出 markdown 格式的审查报告" \
      --allowedTools "Read" "Grep" "Glob" "Bash(git diff*)" \
      --output-format text > review.md
  env:
    ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
```

（官方也提供现成的 GitHub App：会话里运行 `/install-github-app`，之后在 PR 里 @claude 即可。）

### 9.3 批量处理脚本

```bash
# 给所有缺少 docstring 的 Python 文件补文档
for f in $(grep -rL '"""' src --include="*.py"); do
  claude -p "给 @$f 中所有公开函数补充中文 docstring，不改动任何逻辑" \
    --allowedTools "Read" "Edit" --permission-mode acceptEdits
done
```

### 9.4 会话内的自动化

- `/loop 10m npm test 挂了就修` —— 每 10 分钟循环执行一个任务
- `/schedule` —— 创建定时云端任务（如每天早上跑一次依赖安全检查）
- 长命令让它后台跑：直接说「在后台跑 npm run build，期间我们继续聊」

## 第 10 章 · 高阶工作流实战

### 10.1 「探索 → 计划 → 执行 → 验证」四步法

适用于一切复杂任务的黄金流程：

```
1.（plan 模式）> 阅读支付模块相关代码，先不要写任何东西，
   理解现有的退款流程是怎么实现的
2. > 我要支持部分退款。给出实施计划，考虑对账单模块的影响
3.（审阅计划，批准后退出 plan 模式）> 按计划实施，每完成一步跑一次相关测试
4. > 全部完成后：跑全量测试、自查一遍 diff、写出变更总结
```

为什么有效：步骤 1-2 强迫模型先「想」再「做」；你在步骤 3 之前介入审阅，是纠错成本最低的时点；步骤 4 利用了「模型自我验证比自我感觉可靠」的特性。

### 10.2 测试驱动（TDD）：让验收标准变成铁律

```
> 我们用 TDD：先根据这个需求写测试（此时测试应该失败），
  给我确认测试用例合理后，再写实现让测试通过。不要修改测试来迁就实现。
```

这是公认效果最好的 AI 编程模式之一——测试是**机器可验证的需求**，模型有了明确的「赢的条件」，就不容易跑偏。

### 10.3 多任务并行：git worktree

```bash
# 互不干扰的两个工作副本
git worktree add ../myapp-feature-a feature-a
git worktree add ../myapp-bugfix-b bugfix-b

# 两个终端各开一个 Claude
cd ../myapp-feature-a && claude   # 终端 1：开发新功能
cd ../myapp-bugfix-b && claude    # 终端 2：修 bug
```

每个 Claude 实例有独立的目录和上下文，互不污染。这是把自己升级为「AI 团队 leader」的工作方式：你负责分派任务和验收，多个 Claude 并行干活。

### 10.4 让 Claude 审查 Claude

写完代码后：

```
/code-review        # 内置代码审查，检查刚写的 diff
```

或者更彻底——开一个**全新会话**（或用第 8 章的 code-reviewer 子代理）来审查，避免「写代码的上下文」影响审查的客观性。原理同人类：作者自己校对自己的文章，总是看不出错。

### 10.5 处理超大任务：分而治之 + 文档接力

超出单次会话能力的任务（比如大型迁移），用文件做「外部记忆」：

```
> 把整个迁移任务拆解成可独立完成的阶段，写入 MIGRATION_PLAN.md，
  每个阶段标注完成标准。我们每个会话完成一个阶段，
  完成后更新该文件的进度标记。
```

之后每个新会话只需：`> 读 MIGRATION_PLAN.md，继续下一个未完成阶段`。
这绕开了上下文窗口限制——**持久状态放文件里，不放对话里**。

### 10.6 性能与成本调优

- `/model` 按任务选模型：重构、架构设计用最强模型；批量机械修改用 Sonnet/Haiku 更快更省。
- `--effort low|medium|high|max`（或会话内设置）控制思考深度。
- 善用 `/cost` 和 `/context` 建立直觉：什么操作贵、什么操作占上下文。
