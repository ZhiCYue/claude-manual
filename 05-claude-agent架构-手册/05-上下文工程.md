# 第 5 章 · 上下文工程

Prompt engineering(第四册)是"把话说好"。Context engineering(上下文工程)是更大的命题:**管理 agent 在其整个生命周期里看到的全部信息**。对长程 agent,这是决定成败的核心工程。

---

## 5.1 从提示工程到上下文工程

| | 提示工程(第四册) | 上下文工程(本章) |
|---|---|---|
| 关注 | 写好初始指令 | agent 整个生命周期里的信息策展 |
| 范围 | 一次请求的措辞 | 任意时刻 LLM 可见的全部状态 |
| 问题 | 怎么说得清楚 | 怎么对抗 LLM 的固有约束,最大化 token 的效用 |

Anthropic 的定义:

> 上下文工程是**针对 LLM 的固有约束,优化 token 效用**。工程师必须考虑"任意时刻 LLM 可用的整体状态,以及那个状态可能产生什么行为"。

简单说:prompt engineering 问"这一句怎么说",context engineering 问"agent 跑到第 50 轮时,它的上下文窗口里该有什么、不该有什么"。

## 5.2 上下文是有限资源:Context Rot

**这是本章的根本前提。** LLM 的架构决定了上下文不是越多越好:

- **注意力衰减(context rot)**:研究显示,token 数增加时,模型的检索准确性下降。上下文越满,模型越"找不到"里面的信息。
- **Transformer 架构限制**:token 两两关系是 n² 的,上下文增长收益递减。
- **训练数据偏向**:模型对长序列的经验较少,长程推理较弱。

> **核心隐喻**:把上下文当**人的工作记忆**。每个 token 都消耗模型的"注意力预算"。塞太多,它就像一个被淹没在信息里的人——什么都看了,什么都没记住。

这个前提推出本章所有技术:**既然上下文是有限且会衰减的,就要主动策展——保留高信号的,清掉低信号的。**

## 5.3 系统提示的"恰当高度"

系统提示要在两个极端间取平衡:

```
太具体 ←——————————— 恰当高度 ——————————→ 太模糊
硬编码脆弱逻辑                          不提供具体行为信号
(脆弱、难维护)                          (模型不知道该怎么做)
```

- **太具体**:硬编码一堆 if-else 式的脆弱逻辑,产生脆弱性和维护负担。
- **太模糊**:含糊的指引提供不了具体的行为信号。
- **恰当高度**:"足够具体以有效引导行为,又足够灵活以给模型强有力的启发式。"

实操建议:

- 把系统提示组织成清晰的区块:**背景信息 / 指令 / 工具指引 / 输出描述**。
- 从**最小的 prompt** 在强模型上起步,然后根据失败模式迭代地加清晰度。

呼应第四册第 0 章的方法论和第 10 章("更强的模型需要更少的规定性工程")——别一开始就写满,从最小起步,按失败补。

## 5.4 长程 agent 的三大上下文技术

当 agent 跑很多轮、上下文逼近上限,三种主要技术:

### 技术一:压缩(Compaction / 摘要)

在上下文窗口接近上限时,把对话历史提炼——总结关键细节,丢弃冗余输出。

- **核心挑战**:平衡**召回**(抓住所有相关信息)和**精度**(剔除多余内容)。
- **一个低风险的做法**:**工具结果一旦处理完就清掉**,而不是整段对话都留着原始输出。工具返回的原始数据往往又大又只用一次——处理完就清,省下大量上下文。

这就是第二册第 7.3 节的服务端 compaction、第一册第 2 章的 `/compact` 背后的原理。

### 技术二:结构化记笔记(Agentic Memory)

agent 周期性地把笔记写到上下文窗口**之外**(文件、记忆库),需要时再取回。"以最小开销提供持久记忆。"

- 维护待办列表、跨复杂任务跟踪进度、记录战略洞察。
- Anthropic 的 memory 工具(第二册第 5 章)支持跨会话建立基于文件的知识库。

呼应第一册第 10 章的"文档接力"、第四册第 9.5 节的状态跟踪——**把状态从易失的上下文挪到持久的文件**。

### 技术三:子代理架构(Sub-Agent)

专门的子代理处理聚焦的任务,**各自有隔离的上下文窗口**,只把提炼过的摘要(通常 1000–2000 token)返回主 agent。

> "实现了清晰的关注点分离——详细的搜索上下文隔离在子代理内,主 agent 专注于综合和分析结果。"

这是第 6 章多智能体的基础,也是上下文管理的最高形态(第一册第 8 章讲过子代理的上下文隔离)。脏活在子代理的一次性窗口里干,主窗口只收干净的结论。

## 5.5 即时检索(Just-in-Time)

一个反直觉但重要的策略:**别预加载所有数据,运行时按需取。**

```
预加载(传统)             即时检索(JIT)
把所有可能用到的数据       只维护轻量标识符(文件路径、链接),
一次性塞进上下文           运行时动态取真正需要的
↓                        ↓
上下文爆炸、噪音大          上下文精简、按需展开
```

即时检索的好处:

- **模仿人类认知**——人不是把整个语料背下来,而是按需查阅。
- **渐进式展开**——通过探索性交互逐步揭示信息(呼应第三册 Skill 的渐进式加载、第一册第 5 章)。
- 但需要精心设计工具和启发式,避免浪费上下文在错误的检索上。

> **混合策略往往最好**:预取一些数据求速度,同时让 agent 在有益时自主探索。比如给 agent 一份目录结构(预取),让它自己决定读哪些文件(JIT)。

## 5.6 技术选型:按任务匹配

三种技术不是三选一,而是按任务特性匹配(常常组合用):

| 任务特性 | 主用技术 |
|---|---|
| 长对话流(聊天式) | 压缩 |
| 迭代式开发(跟踪进度) | 结构化记笔记 |
| 并行研究(宽度优先) | 多代理架构 |
| 大语料、按需访问 | 即时检索 |

很多长程 agent **三个都用**:压缩控制单会话的膨胀,记笔记跨会话持久化,子代理隔离脏活。

## 5.7 上下文工程的核心建议

Anthropic 的总结,也是本章的行动指南:

1. **无情地最小化 token**:策展"能最大化期望结果可能性的、最小的高信号 token 集"。不是塞得越多越好,是留得越准越好。
2. **从简单开始**:用"能工作的最简方案",按失败模式迭代。
3. **策略性地用示例**:给多样的、典范的示例,而非穷举边界情况(第四册第 2 章)——"示例是胜过千言万语的图画"。
4. **技术匹配任务**:压缩(对话流)/ 记笔记(迭代开发)/ 多代理(并行研究),按特性选。
5. **为演进做准备**:更聪明的模型"需要更少的规定性工程",随时间增加 agent 的自主性。

## 5.8 上下文工程与 harness

回到第 0 章:**管理上下文是 harness 的核心职责之一。** 本章的三大技术,在实践中是 harness 实现的:

- harness 决定何时触发压缩、清哪些工具结果。
- harness 提供记忆工具的存储后端、决定何时让 agent 记笔记。
- harness 编排子代理、收集它们的摘要。

你用现成的 harness(Claude Code、Managed Agents)时,这些大多自动处理;你自己写 harness 时,这些是你必须设计的。无论哪种,理解上下文工程的原理,你才能配置好、或实现好它。
