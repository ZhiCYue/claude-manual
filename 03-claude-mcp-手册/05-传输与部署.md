# 第 5 章 · 传输与部署

到这里你的服务器逻辑已经齐备。这一章讲传输层——服务器和宿主到底怎么"连上"。两种传输方式决定了你的服务器是本地工具还是云端服务。

---

## 5.1 回顾：传输层独立于业务

第 0 章 0.3 节讲过两层架构：数据层（JSON-RPC 消息）和传输层（怎么传这些消息）。关键结论：**同样的工具/资源/提示词代码，可以走不同的传输方式，几乎不用改业务逻辑**。

MCP 有两种传输：

| 传输 | 形态 | 典型场景 | 服务谁 |
|---|---|---|---|
| **stdio** | 本地子进程，标准输入输出 | 操作本地文件、本地数据库、本地设备 | 通常一个客户端 |
| **Streamable HTTP** | 远程服务，HTTP POST + 可选 SSE | SaaS、团队共享、云端服务 | 通常多个客户端 |

## 5.2 stdio 传输（本地）

前几章一直在用的就是 stdio。`mcp.run()` 默认就是它：

```python
if __name__ == "__main__":
    mcp.run()                  # 等价于 mcp.run(transport="stdio")
```

**工作原理**：宿主把你的服务器作为**子进程**启动（还记得第 1 章 `claude mcp add weather -- uv ... run weather.py` 吗？那条命令就是宿主用来启动子进程的）。宿主和服务器通过标准输入输出交换 JSON-RPC 消息——宿主写到服务器的 stdin，服务器写到 stdout。

**特点**：

- 性能最好，无网络开销。
- 服务器和宿主在同一台机器上。
- 生命周期由宿主管理——宿主启动它、用完关掉它。
- 适合：访问本地资源（文件、本地 DB）、个人工具、不需要共享的场景。

> ⚠️ **stdio 的一个坑**：服务器**绝对不能往 stdout 打印调试信息**——stdout 是 JSON-RPC 消息通道，任何额外输出都会破坏协议。调试输出要写到 **stderr** 或日志文件。这是 stdio 服务器最常见的 bug 来源（第 7 章详谈）。

## 5.3 Streamable HTTP 传输（远程）

要让服务器跑在云上、被多个用户/宿主共享，用 Streamable HTTP：

```python
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("weather")

@mcp.tool()
def get_forecast(city: str) -> str:
    """获取天气预报。"""
    return f"{city}：晴。"

if __name__ == "__main__":
    # 跑成 HTTP 服务，监听端口
    mcp.run(transport="streamable-http")
```

**工作原理**：服务器是一个长期运行的 HTTP 服务。客户端用 HTTP POST 发请求，服务器用 HTTP 响应回复；需要流式（如长任务进度）时用 Server-Sent Events（SSE）。

**特点**：

- 服务器跑在远程，独立于任何宿主。
- 一个服务器可同时服务多个客户端。
- 支持标准 HTTP 鉴权——Bearer token、API key、自定义头。MCP 推荐用 OAuth 获取 token。
- 适合：SaaS 产品、团队共享的内部服务、官方托管的远程服务（如 Sentry 的 MCP）。

## 5.4 两种传输怎么选？

```
你的服务器需要访问什么？
│
├── 本地资源（用户的文件、本机数据库、本地设备）
│   └── stdio —— 跟着宿主在用户机器上跑
│
├── 远程资源 / 要被多人共享 / 是个 SaaS
│   └── Streamable HTTP —— 部署在云上，多客户端连接
│
└── 既要本地又要远程？
    └── 写一份业务代码，跑两种传输（5.1 节：业务逻辑独立于传输）
```

经验法则：**个人用、访问本地东西 → stdio；团队共享、云端服务 → HTTP。**

## 5.5 远程部署

把 HTTP 服务器部署成真正的服务，和部署任何 Python web 服务一样。FastMCP 的 HTTP 服务器底层是 ASGI 应用，可以用 uvicorn 跑：

```bash
# 直接跑
python weather_server.py

# 或用 uvicorn（生产环境）
uvicorn weather_server:mcp.streamable_http_app --host 0.0.0.0 --port 8000
```

容器化（Dockerfile 片段）：

```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY . .
RUN pip install "mcp[cli]" httpx
EXPOSE 8000
CMD ["python", "weather_server.py"]
```

部署到任何能跑容器的地方（云服务器、Kubernetes、Serverless 平台）。客户端通过 URL 连接（第 6 章）。

## 5.6 远程服务器的鉴权

远程服务器对外暴露，**必须鉴权**。Streamable HTTP 支持标准 HTTP 鉴权：

- **Bearer token / API key**：客户端在请求头带 token，服务器验证。
- **OAuth**：MCP 推荐的方式——客户端通过 OAuth 流程获取 token。官方 SDK 提供 OAuth 支持。

最简单的 token 校验（生产环境用更完善的方案）：

```python
# 概念示意：在 HTTP 服务器前加鉴权中间件
# 实际用 SDK 的 auth 支持或标准 ASGI 中间件
from starlette.middleware.base import BaseHTTPMiddleware

class AuthMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        token = request.headers.get("Authorization", "").removeprefix("Bearer ")
        if not is_valid_token(token):
            from starlette.responses import JSONResponse
            return JSONResponse({"error": "unauthorized"}, status_code=401)
        return await call_next(request)
```

> **安全红线**：远程 MCP 服务器是公网入口，且 AI 会自动调用它的工具。鉴权、限流、输入验证、最小权限——一个都不能少。第 7 章是完整的安全章节。

## 5.7 在 API 里直连远程服务器

第二册讲过 Messages API 支持 `mcp_servers` 参数，让 Claude 直连远程 MCP 服务器（无需本地宿主）。你部署好的 HTTP 服务器可以这样被 API 直接使用：

```python
# 这是第二册的 API 侧用法，连你部署的远程服务器
response = client.beta.messages.create(
    model="claude-opus-4-8",
    max_tokens=4096,
    mcp_servers=[{
        "type": "url",
        "name": "weather",
        "url": "https://your-server.example.com/mcp",
    }],
    betas=["mcp-client-2025-11-20"],
    messages=[{"role": "user", "content": "北京天气怎么样？"}],
)
```

这就是 MCP 开放协议的回报：你写一次服务器，它既能接 Claude Code（stdio），又能被 API 直连（HTTP），还能接任何其他 MCP 宿主。
