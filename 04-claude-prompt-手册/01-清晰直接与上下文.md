# 第 1 章 · 清晰、直接与上下文

这是最基础也最重要的一章。**80% 的 prompt 问题在这里解决。** 在堆任何高级技巧之前，先把话说清楚。

---

## 1.1 核心原则：具体、明确

Claude 对清晰、明确的指令响应很好。**想要什么，就具体地说出来。** 尤其是想要"超出基础"的表现时，要明确要求，别指望模型从模糊的 prompt 里推断。

回顾第 0 章的对比：

**差：**
```
创建一个分析仪表盘
```

**好：**
```
创建一个分析仪表盘。包含尽可能多的相关功能和交互。
不要止步于基础——做一个功能完整的实现。
```

第二个之所以好，是因为它**明确要求了"上不封顶"的行为**。最新模型严格遵循指令——你说"创建仪表盘"，它就给你一个仪表盘；你想要"功能丰富、超出预期"的仪表盘，就得说出来。

## 1.2 用顺序步骤

当步骤的顺序或完整性重要时，用编号列表或项目符号把指令拆成清晰的步骤：

**差：**
```
帮我处理一下这些客户反馈
```

**好：**
```
处理这批客户反馈，按以下步骤：
1. 把每条反馈分类为：bug 报告 / 功能请求 / 使用问题 / 其他
2. 对 bug 报告，提取复现步骤（如果有）
3. 按出现频率统计，找出 top 5 最常见的问题
4. 输出一份按优先级排序的问题列表
```

清晰的步骤让模型知道做什么、按什么顺序、做到什么程度算完。

## 1.3 给动机和理由（最被低估的技巧）

**这是新手最容易忽略、收益却很高的一招。** 解释指令背后的动机——告诉 Claude 为什么这个要求重要——能让它更好地理解你的目标，产出更贴合的结果。

官方的经典对比：

**差：**
```
永远不要用省略号
```

**好：**
```
你的回复会被文本转语音引擎朗读，所以永远不要用省略号——
语音引擎不知道怎么读它。
```

为什么"好"的版本更好？因为 Claude 足够聪明，能从解释里**泛化**。知道了"会被朗读"这个原因，它不仅会避免省略号，还会避免其他朗读时有问题的东西（奇怪的符号、纯视觉的排版）。你给了原因，它就能举一反三。

**实用模式**：每当你下达一个看似武断的约束，加一句"因为……"。

```
用简体中文回复（因为这是面向中国用户的产品）
代码注释用英文（因为团队里有不说中文的成员）
金额一律保留两位小数（因为这是财务报表，精度有合规要求）
```

## 1.4 提供背景上下文

回到"聪明但全新的同事"隐喻：新同事缺的是背景。把任务相关的背景给足：

**差：**
```
这段代码有什么问题？
（贴一段代码）
```

**好：**
```
这是我们订单服务的支付回调处理函数。线上偶发"重复扣款"投诉，
但本地无法稳定复现。这个函数在高并发下可能被同一个回调触发多次。
帮我找出可能导致重复扣款的问题。
（贴代码）
```

第二个给了：这段代码是干什么的、出了什么现象、怀疑的方向。有了这些，模型能精准地往"并发/幂等"方向查，而不是泛泛地审一遍代码。

## 1.5 明确输出格式和约束

想要特定格式，就描述清楚：

```
输出一个 markdown 表格，列为：问题 | 严重程度 | 影响范围 | 建议修复
严重程度用 高/中/低 表示。
只列出确实存在的问题，不要为了凑数编造。
```

约束（"只列确实存在的""不要编造"）和格式（表格、列定义）都说清楚，模型就不会自由发挥。

## 1.6 "新同事检验法"实操

把第 0 章的黄金法则用起来。写完 prompt，逐句问自己：

- **这句话，一个不了解背景的人能看懂吗？** 看不懂 → 补上下文。
- **我有没有假设它知道某个"显然"的东西？** 有 → 写出来。
- **输出格式说清楚了吗？** 没有 → 加上。
- **有没有模糊的词？** "处理一下""优化优化""好看点"——换成可执行的具体描述。

一个改写练习，对照感受：

**改写前：**
```
帮我把这个 API 文档写好看点
```

**改写后：**
```
重写这份 API 文档，目标读者是第一次接入我们 API 的外部开发者。
要求：
1. 每个端点：一句话说明用途 + 请求示例 + 响应示例 + 常见错误码
2. 示例用 curl 和 Python 两种
3. 假设读者懂 HTTP 但不懂我们的业务概念，必要时解释业务术语
4. 语气专业但友好，避免内部黑话
```

"好看点"是模糊的许愿；改写后是可执行的规格。模型对后者的产出会稳定得多。

## 1.7 小结：清晰度清单

下达任务时过一遍：

- [ ] 目标具体（不是"处理一下"而是"分类成 X/Y/Z"）
- [ ] 想要超额表现时明确要求了（"尽可能丰富""超出基础"）
- [ ] 顺序重要时用了编号步骤
- [ ] 武断的约束都给了理由（"因为……"）
- [ ] 提供了任务背景（这是什么、为什么、怀疑方向）
- [ ] 输出格式和约束说清楚了
- [ ] 通过了"新同事检验"——没有模糊词、没有未言明的假设

这一章的技法朴素，但**投入产出比最高**。把它练扎实，再上后面的技巧。
