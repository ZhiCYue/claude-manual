# Prompt Engineering 进阶（第四册）

> 编写时间：2026-06，基于 Anthropic 官方 prompting best practices 及最新模型（Claude Fable 5 / Opus 4.8）的行为指南。
> 这是一册**横向能力**——前三册讲"用什么工具/接口/协议"，这册讲"怎么把话说好"，让模型真正按你的意图工作。无论你在用 Claude Code、写 API、还是造 MCP 服务器，都要把话说对。

## 这册讲什么、为什么"进阶"

Prompt engineering（提示工程）是用自然语言可靠地引导模型产出你想要结果的技艺。它**不是玄学，不是咒语**——是一套有原理、可系统化、可度量改进的工程方法。

为什么标"进阶"？因为基础的"把话说清楚"只是入场券。真正拉开差距的是：

- 知道**什么时候该调 prompt，什么时候该换模型/换方法**（不是所有问题都靠 prompt 解决）。
- 用**评估驱动**地迭代 prompt，而不是凭感觉改。
- 驾驭 **agentic 场景**的微妙行为——工具触发、并行调用、自主性与安全的平衡、子代理编排。
- 针对**最新模型的行为特性**精细调优——Fable 5 和 Opus 4.8 比老模型更字面化、更简洁、更自主，老 prompt 往往需要重调。

这册把基础技法讲扎实（第 1–8 章），把重头放在 agentic 和最新模型调优（第 9–10 章）——那才是"进阶"真正的价值所在。

## 全书主线：把 Claude 当聪明但全新的同事

这是贯穿全册的核心隐喻：

> **把 Claude 想象成一位绝顶聪明、但对你的规范和工作流毫无背景的新员工。你解释得越精确，结果越好。**

一个黄金检验法则：**把你的 prompt 拿给一个对任务背景了解很少的同事看，让他照着做。如果他会困惑，Claude 也会。**

所有技法——清晰、示例、XML 结构、角色、思考、链式——本质上都是在给这位新同事提供他需要的东西：明确的目标、足够的上下文、好的范例、清晰的结构。

## 学习路径建议（3 周）

```
第 1 周  00–04 章：基础四件套(清晰/示例/XML/角色) + 方法论。
         练习：选一个你现在用的 prompt，按"新同事检验法"重写，对比效果
第 2 周  05–08 章：思考、输出控制、链式、长上下文。
         练习：把一个复杂任务拆成"草稿→审查→修订"三步链式 prompt
第 3 周  09–10 章：agentic prompt 与最新模型调优(进阶核心)。
         练习：给一个 agent 写系统提示，调好工具触发、自主性、防过度工程
```

## 三条贯穿全册的硬规则

1. **不是所有问题都靠 prompt 解决**：延迟和成本常常换个模型更省事；prompt engineering 解决的是"可通过措辞引导"的问题（第 0 章）。
2. **评估先行**：没有明确的成功标准和测试方法，改 prompt 就是盲调。先建立"怎么算成功""怎么测"，再优化（第 0 章）。
3. **最新模型更字面、更自主**：Fable 5 / Opus 4.8 严格遵循指令、默认更简洁更主动。为老模型写的"CRITICAL: 你必须……"在新模型上会过度触发——该松绑了（第 10 章）。
