# 第 2 章 · 用示例 Multishot

示例是引导模型输出格式、语气、结构最可靠的手段之一。几个精心设计的示例（叫 few-shot 或 multishot prompting），能戏剧性地提升准确性和一致性。

---

## 2.1 为什么示例这么有效

回到"新同事"隐喻：教新人做一件事，最快的方式是什么？**给他看几个做好的样例。** 你说一百遍"格式要规范"，不如给他看三个规范的样例。

模型也一样。示例直接展示了你想要的**模式**——格式、语气、详略、结构——比任何文字描述都直观。当你发现"怎么说它都不照做"时，往往不是说得不够多，而是**该给示例了**。

## 2.2 一个直观对比

假设你要做情感分类，想要特定的输出格式。

**只有指令（容易跑偏）：**
```
判断这条评论的情感，输出情感和置信度。

评论：这家店服务还行，菜一般般，不会再来了。
```

模型可能输出"负面，70%"，也可能输出一段分析，也可能用英文……格式不稳定。

**加上示例（稳定）：**
```
判断评论的情感。参照以下示例的格式输出：

<examples>
<example>
评论：太棒了，超出预期，强烈推荐！
输出：{"sentiment": "正面", "confidence": 0.95}
</example>
<example>
评论：还行吧，没什么特别的。
输出：{"sentiment": "中性", "confidence": 0.80}
</example>
<example>
评论：太差了，再也不来，浪费钱。
输出：{"sentiment": "负面", "confidence": 0.98}
</example>
</examples>

评论：这家店服务还行，菜一般般，不会再来了。
输出：
```

现在模型清楚地知道：输出 JSON、字段叫什么、置信度是 0–1 的小数、情感用中文。格式锁定了。

## 2.3 好示例的三个标准

加示例时，让它们满足：

### 相关（Relevant）

示例要紧贴你的真实用例。做客服工单分类，示例就用真实的工单样子，别用天气问答。示例和实际输入越像，模型迁移得越准。

### 多样（Diverse）

示例要覆盖边界情况，且彼此差异足够大，避免模型学到你不想要的意外模式。

比如上面的情感分类，如果三个示例都是短句，模型可能以为"只处理短句"；如果都是正面，模型可能偏向输出正面。**有意覆盖：正面/负面/中性、长/短、明确/含糊**——这样模型学到的是"判断情感"这个任务本身，而不是某个表面模式。

特别要包含**边界和困难情况**：

```
<example>
评论：包装很用心，但东西收到是坏的。
输出：{"sentiment": "负面", "confidence": 0.75}
</example>
```

这个"先扬后抑"的例子教会模型处理混合情感——这正是纯指令难以说清的。

### 结构化（Structured）

用 `<example>` 标签包裹每个示例，多个示例外面再包 `<examples>`，让模型能把示例和指令区分开。XML 标签是第 3 章的主题，这里先记住：**示例要和指令有清晰的边界**，否则模型可能把示例内容误当成要处理的输入。

## 2.4 用几个示例？

**3–5 个通常最佳。** 太少（1 个）模式不够清晰，模型可能过拟合到那一个例子；太多则浪费上下文、收益递减。

从 3 个覆盖主要情况的示例起步，发现某类输入处理不好，就针对性地补一个那类的示例。

## 2.5 让 Claude 帮你做示例

两个省力技巧：

1. **让它评估你的示例**：
   ```
   这是我打算用的几个 few-shot 示例。请评估它们的相关性和多样性，
   指出覆盖不足的情况，并建议补充哪些示例。
   ```

2. **让它生成更多示例**：
   ```
   基于这 3 个示例的模式，再生成 5 个覆盖不同边界情况的示例。
   ```

这能快速把一组单薄的示例扩充得更健壮。

## 2.6 示例配合思考（进阶）

示例不仅能教格式，还能教**推理方式**。在 few-shot 示例里展示思考过程（用 `<thinking>` 标签），模型会把这种推理风格泛化到它自己的思考中：

```
<example>
问题：一个班 30 人，60% 是女生，女生里 1/3 戴眼镜。戴眼镜的女生有几人？
<thinking>
女生人数 = 30 × 60% = 18
戴眼镜的女生 = 18 × 1/3 = 6
</thinking>
答案：6 人
</example>
```

这教会模型"先拆解、分步算、再给答案"的模式。第 5 章会深入思考，这里先记住：**示例是教推理模式的好载体**。

## 2.7 什么时候示例不是最佳选择

示例很强，但不是万能：

- **格式极其复杂或动态**时，结构化输出（第 6 章的 `output_config.format`）比示例更可靠——它是强制的，示例只是引导。
- **任务逻辑复杂**时，光给输入输出对可能不够，需要配合清晰的指令（第 1 章）和思考（第 5 章）解释"为什么"。
- **示例会占大量上下文**时（每个示例很长），权衡成本——配合 prompt caching（第二册第 6 章）缓存固定的示例部分。

> **经验法则**：格式/风格/语气的一致性问题 → 优先用示例；硬性的格式保证 → 用结构化输出；复杂推理 → 示例 + 思考一起上。
