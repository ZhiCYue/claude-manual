# 第 6 章 · 多智能体协作

你特别想了解的"多 agent 协作自动完成全部任务",这一章讲透。它能把复杂研究任务的质量提升 90%,但烧 15 倍的 token。架构怎么搭、什么时候值这个钱、怎么编排——基于 Anthropic 构建多智能体研究系统的真实经验。

---

## 6.1 架构:编排者-工作者(规模化)

多智能体系统的主架构,就是第 2.4 节的**编排者-工作者**模式的规模化:

```
                    ┌─→ [子代理 1] 探索方面 A ─┐
人类查询 → [主 agent] ┼─→ [子代理 2] 探索方面 B ─┼→ [主 agent 综合] → 报告
          分析查询      └─→ [子代理 N] 探索方面 C ─┘
          制定策略       (各自独立的上下文窗口、工具、探索轨迹)
          派生子代理
```

主 agent(lead agent)**分析查询、制定策略、派生子代理**去同时探索不同方面。子代理并行工作,主 agent 综合发现。

**为什么这个架构强**:回顾第 5.4 节的子代理技术——每个子代理有**自己独立的上下文窗口和探索轨迹**,提供"关注点分离:不同的工具、prompt、探索轨迹",减少路径依赖。一个上下文窗口装不下的大任务,被切分到多个窗口里并行处理。

实测数据:token 用量本身解释了浏览任务性能 80% 的方差——**把工作分散到多个独立上下文窗口,本身就是性能的来源。**

## 6.2 成本:15 倍 token 的账(必须算)

**这是决定该不该上多 agent 的核心。** Anthropic 的真实数据:

| 系统 | token 用量(相对聊天) |
|---|---|
| 普通聊天 | 1× |
| 单 agent | 约 4× |
| **多 agent** | **约 15×** |

多 agent 烧 15 倍于聊天、约 4 倍于单 agent 的 token。代价巨大。

但回报也可能巨大:在内部研究评估上,**多 agent(Opus 4 主 + Sonnet 4 子)比单 agent Opus 4 高出 90.2%**;并行化把复杂查询的研究时间**缩短最多 90%**。

> **这笔账怎么算**:多 agent 用 15 倍 token 换 90% 的质量提升和 90% 的时间缩短。**只有当任务的价值能证明 15 倍 token 时,这笔账才划算。** 这是真正的工程判断,不是"多 agent 更先进所以用它"。

## 6.3 什么时候多 agent 值得,什么时候不值

| ✅ 多 agent 适合 | ❌ 多 agent 不适合 |
|---|---|
| 宽度优先、需要并行调查的查询 | 编码任务(可并行部分少) |
| 价值能证明 15 倍 token 的任务 | 需要 agent 间共享上下文的域 |
| 超出单上下文窗口的问题 | 需要 agent 间实时协调的场景 |
| 大量并行 + 众多复杂工具的场景 | |

**关键反例:编码通常不适合多 agent。** 因为编码任务可并行的部分少、且常需要共享上下文(改 A 文件要知道 B 文件怎么改)。这正是第 6.2 节数据里"多 agent 在研究上赢,但研究 ≠ 编码"的原因——研究是宽度优先、可并行、子任务独立的,天然适合多 agent;编码是深度优先、强耦合的,单 agent 往往更合适。

> **判断口诀**:任务能不能切成**互相独立、不需共享状态**的并行子任务?能 → 多 agent 可能值;不能(子任务强耦合)→ 单 agent。

## 6.4 编排的五条 prompt 原则

多 agent 系统成败,很大程度在于怎么 prompt 主 agent 和子代理。Anthropic 的五条核心经验:

### ① 教会清晰的委派

主 agent 必须给每个子代理:**一个目标、输出格式、工具和来源的指引、清晰的任务边界**。

早期含糊的指令(如"研究半导体短缺")导致多个子代理**重复劳动**——它们不知道彼此的边界,各自把整个题目做了一遍。

```
# ❌ 含糊,导致子代理重复劳动
"研究一下半导体行业"

# ✅ 清晰委派,边界分明
子代理1:"调研 2025-2026 半导体产能扩张,只看晶圆厂新建项目,输出按地区分类的产能数据表"
子代理2:"调研半导体需求侧,只看 AI 芯片和汽车芯片,输出需求增长预测"
子代理3:"调研供应链瓶颈,只看光刻设备和封装,输出关键瓶颈清单"
```

### ② 让努力程度匹配复杂度

在 prompt 里嵌入明确的指引,防止简单查询过度投入:

> 简单的事实查找:**1 个 agent,3–10 次工具调用**。
> 复杂研究:**10+ 个子代理**,职责分明。

不给这个指引,agent 可能给一个简单问题派一堆子代理,烧钱烧时间。

### ③ 工具设计是关键接口

回顾第 4 章——"agent-工具接口和人机接口一样关键"。在多 agent 里更甚:糟糕的工具描述把子代理引上完全错误的路;改进描述让任务完成时间减少 40%。

### ④ 搜索策略由宽到窄

教 agent:**从短而宽的查询开始,评估有什么,再逐步收窄**——而不是一上来就用过于具体的查询(那往往返回很少结果)。这是人类研究的方式,也是 agent 该学的。

### ⑤ 善用扩展思考

- **主 agent** 用思考来:规划方法、评估哪些工具合适、判断查询复杂度和子代理数量。
- **子代理** 用交错思考(interleaved thinking):在工具结果后评估质量、识别缺口、精炼下一个查询。

呼应第四册第 5 章——思考是 agent 规划和反思的引擎。

## 6.5 用 Managed Agents 实现多智能体

第二册第 9 章的 Managed Agents 内置了多智能体支持。你不用从零搭编排——声明一个 coordinator(协调者)和它能委派的 agent 名册:

```python
# 在 agent 配置上声明 multiagent(顶层字段,不在 session 上)
orchestrator = client.beta.agents.create(
    name="研究主管",
    model="claude-opus-4-8",
    system="你协调研究工作。把代码审查委派给 reviewer,把测试委派给 test agent。",
    tools=[{"type": "agent_toolset_20260401"}],
    multiagent={
        "type": "coordinator",
        "agents": [
            reviewer.id,                                    # 子代理(最新版本)
            {"type": "agent", "id": test_writer.id, "version": 4},
            {"type": "self"},                               # 也能派生自己的副本
        ],
    },
)
session = client.beta.sessions.create(agent=orchestrator.id, environment_id=env.id)
```

每个子代理在自己的**线程**(thread)里跑——独立的上下文、模型、系统提示、工具。它们共享容器和文件系统,但**不共享对话历史**。主 agent 通过跨线程消息委派和收集结果。最多 25 个并发线程,委派只有一层(子代理不能再派子代理)。

> **注意 Managed Agents 多 agent 的一个陷阱**:子代理共享文件系统但**不共享对话历史**。主 agent 要子代理做什么,必须在委派消息里说清楚(或写到磁盘),不能假设子代理"知道"主对话聊了什么。这正是第 6.4 节"清晰委派"原则的体现。

## 6.6 多 agent 的工程挑战

多 agent 比单 agent 难驾驭,几个真实挑战:

### 错误复合(Error Compounding)

agent 跨很多工具调用维护状态。**小故障会变成灾难**——传统软件里的小问题,在 agent 系统里能让它跑偏到完全不同的轨迹。一步失败,可能让 agent 探索完全错误的方向。

应对:重试逻辑 + 定期检查点 + 能从出错点恢复的系统(第 7 章详谈)。

### 同步瓶颈

当前主 agent 同步执行子代理——等每一批完成才继续。这在信息流上造成瓶颈。异步执行能让主 agent 实时引导子代理,但带来结果协调、状态一致性、错误传播的新挑战。

### 协调成本

子代理越多,协调越复杂。这也是"努力匹配复杂度"(第 6.4 节②)重要的原因——别为简单任务派一堆子代理,协调成本会吃掉收益。

## 6.7 "像你的 agent 一样思考"

Anthropic 的一条实用方法论:**用实际的 prompt 和工具搭建模拟,一步步看 agent 怎么工作。** 这能暴露失败模式:

- agent 在结果已经够用后还继续(不知道何时停)。
- 用啰嗦低效的查询。
- 选错工具。

看见这些,你才知道该怎么改 prompt、改工具、改编排。这呼应第四册第 0 章的评估驱动——多 agent 系统尤其需要观察真实行为来调。

## 6.8 多智能体决策清单

考虑上多 agent 时,逐项问:

- [ ] 任务能切成**独立、不需共享状态**的并行子任务吗?(不能 → 别上)
- [ ] 任务价值能证明 **15 倍 token** 吗?(不能 → 用单 agent)
- [ ] 是宽度优先的研究类任务,还是深度耦合的编码类?(编码 → 倾向单 agent)
- [ ] 委派指令是否清晰(目标/格式/边界/工具)?(含糊 → 子代理会重复劳动)
- [ ] 努力程度匹配了复杂度吗?(简单查询别派一堆子代理)
- [ ] 有错误恢复和检查点吗?(没有 → 错误会复合成灾难)
- [ ] 能观测每个子代理在做什么吗?(看不见 → 没法调试)

全过了,多 agent 才值得上。否则退回单 agent——它更便宜、更可预测、更好调试。
