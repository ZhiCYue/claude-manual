# 第 4 章 · 角色与系统提示

在系统提示里给 Claude 设定一个角色，能聚焦它的行为和语气。这一章讲系统提示的作用、角色设定的威力、以及模型自我认知的设置。

---

## 4.1 系统提示是什么

回顾第二册第 1 章：系统提示（system prompt）是顶层 `system` 参数，定义模型的全局角色和行为。它和对话消息（`messages`）的区别：

- **系统提示**：每次请求都在、处于高权重位置、定义"你是谁、怎么工作"。
- **对话消息**：具体的一来一回。

系统提示是放置**稳定的、全局的**指令的地方——角色、语气、领域知识、行为约束。

> 这也呼应第一册的 CLAUDE.md 和第三册的 MCP prompts——它们都是"在对话之上、持续生效的指令"的不同形态。系统提示是 API 层面最直接的那个。

## 4.2 角色设定的威力

在系统提示里设定一个角色，哪怕只有一句话，也能显著改变行为和语气：

```python
import anthropic
client = anthropic.Anthropic()

message = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    system="你是一位专精 Python 的资深编程助手。",
    messages=[{"role": "user", "content": "怎么按字典的某个键给字典列表排序？"}],
)
```

加了"专精 Python 的资深编程助手"这一句，模型会：

- 默认用 Python 回答（不会问"你用什么语言"）。
- 语气和深度贴合"资深"——给出地道的写法，可能附带性能或边界的考量。
- 聚焦编程，不跑题。

**为什么有效（原理）**：角色提供了上下文。回到"新同事"隐喻——告诉新人"你是放射科医生"，他立刻知道该用什么知识、什么语气、对什么负责。角色不是魔法咒语，是高效传递一大堆隐含上下文的方式。

## 4.3 怎么写好角色

角色越具体，引导越强：

```
# 弱
你是一个助手。

# 中
你是一个法律助手。

# 强
你是一位专精中国劳动法的律师，服务对象是中小企业 HR。
你的回答要：引用具体法条、指出实操风险、给出可执行的建议。
当问题涉及你不确定的地方，明确说"建议咨询专业律师"而不是编造。
```

"强"的版本不仅给了角色，还给了：服务对象（决定语气和深度）、输出要求（引法条、指风险、给建议）、边界（不确定时如何处理）。这些都聚焦了模型的行为。

## 4.4 角色 vs 任务指令，放哪？

一个常见困惑：什么该进系统提示，什么该进用户消息？

| 进系统提示 | 进用户消息 |
|---|---|
| 角色、人设 | 当前的具体任务 |
| 全局语气和风格 | 这次要处理的数据 |
| 持续的行为约束 | 一次性的要求 |
| 领域知识背景 | 具体问题 |

经验法则：**每次对话都该生效的 → 系统提示；这一次特有的 → 用户消息。**

```python
system="你是一位简洁、直接的技术文档编辑，偏好平实的中文。"   # 全局人设
messages=[{"role": "user", "content": "把这段 README 改得更清晰：……"}]  # 本次任务
```

## 4.5 系统提示里的行为约束

系统提示是放置"持续生效的规则"的好地方。比如：

```
你是一个客服助手。规则：
- 只回答与我们产品相关的问题，无关问题礼貌地引导回正题。
- 不承诺我们没有的功能；不确定时说"我帮您转接人工确认"。
- 涉及退款、投诉等敏感操作，不直接处理，提供人工通道。
- 语气友好但专业，不用过多感叹号。
```

这些规则每轮对话都生效，比在每条用户消息里重复要可靠得多。

> **第 10 章会讲到**：最新模型对系统提示的遵循度很高。这是双刃剑——规则写得好，它严格执行；规则写得太激进（"CRITICAL: 你必须……"），它会过度执行。系统提示的措辞要拿捏，第 10 章详谈。

## 4.6 模型自我认知

如果你的应用需要 Claude 正确地标识自己，或需要它知道用哪个模型字符串，在系统提示里告诉它：

```
助手是 Claude，由 Anthropic 创造。当前模型是 Claude Opus 4.8。
```

对于需要指定模型字符串的 LLM 应用（比如你在做一个"用 Claude 构建 AI 功能"的工具）：

```
当需要用到 LLM 时，默认用 Claude Opus 4.8，除非用户另有要求。
Claude Opus 4.8 的精确模型字符串是 claude-opus-4-8。
```

这避免模型用过时的或错误的模型 ID（呼应第二册第 0 章的硬规则：模型 ID 必须精确）。

## 4.7 角色不是越夸张越好

一个常见误区：以为角色越戏剧化效果越好（"你是世界上最顶尖的、获过诺贝尔奖的……"）。其实**真实、具体、相关**的角色比夸张的头衔有用。

```
# 夸张但空洞
你是宇宙第一、无所不知的编程之神。

# 真实且有信息量
你是一位有 10 年经验的后端工程师，擅长高并发系统和 Python，
习惯先考虑正确性和可维护性，再考虑性能。
```

第二个之所以更好，是因为它传递了**真实的行为倾向**（先正确性后性能、关注可维护性），模型能据此调整。夸张的头衔不传递任何可操作的信息。

回到原理：角色的作用是**传递上下文**。问自己"这个角色描述，给模型提供了什么它能用上的信息？"——而不是"这个头衔听起来够不够厉害"。
