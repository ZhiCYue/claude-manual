# 第 8 章 · 落地路线图

前面各章是零件:工作流、项目约定、配置、混合工具、协作规范、护栏、治理。这一章把它们排成一条**分阶段的落地路线**——从个人试用到组织治理,以及怎么应对常见阻力。

---

## 8.1 分阶段落地:别一步到位

团队采纳 AI 不该一刀切。分四个阶段,每阶段把上一阶段验证好了再往前:

```
阶段 1  个人采纳    ──→  阶段 2  项目落地  ──→  阶段 3  团队推广  ──→  阶段 4  组织治理
(几个人试用)         (一个项目改造)        (推广到团队)         (企业级)
```

**核心原则**(呼应第五册的"从简单开始"):**每阶段证明价值,再扩大。** 别在没验证个人价值前就推全组织;别在没建好工具中立护栏前就放开自主性。

## 8.2 阶段 1:个人采纳(几个人,1–2 周)

**目标**:几个有意愿的人先用顺,积累经验和信心。

做什么:

- 让 2–3 个有兴趣的开发者用 Claude Code(或各自喜欢的 AI 工具)做真实任务。
- 跑通个人工作流(第 1 章):探索→计划→执行→验证、会话卫生、按任务选模型。
- 不强制任何人,不改团队流程。

**不做什么**:别一上来就定一堆规范、改全队配置。先让早期采用者摸出"什么任务 AI 帮得上、什么帮不上"。

**验证**:这几个人是否真的更高效了?他们能讲出具体的收益和坑吗?

## 8.3 阶段 2:项目落地(一个项目,2–4 周)

**目标**:把一个项目改造成"AI 友好",建立配置即代码的样板。

做什么:

- 选一个合适的项目(不是最关键的核心系统,也不是没人维护的边角),做第 2 章的 AI 友好改造:整理结构、写 CONTRIBUTING.md、标准化命令。
- 建第 3 章的配置:工具中立层(.editorconfig/linter/pre-commit/CI)+ Claude Code 层(.claude/settings.json/CLAUDE.md/.mcp.json)进 git。
- 关键:**先把工具中立的护栏(CI/git hook)建起来**——这是后面推广的地基。

**验证**:新人(或 AI)在这个项目里上手是否更快?配置是否真的 clone 即用?

## 8.4 阶段 3:团队推广(整个团队,1–2 月)

**目标**:把样板推广到团队,处理混合工具现实,定协作规范。

做什么:

- 把阶段 2 的样板(AI 友好结构 + 双层配置)推广到团队的项目。
- **正视混合工具现实**(第 4 章):不强制统一工具,但确保所有"必须做到"的标准在工具中立层强制(CI、分支保护、git hook)。
- 定协作规范(第 5 章):AI 代码审查纪律、PR/commit 规范,写进 CONTRIBUTING.md。
- 建质量安全护栏(第 6 章):CI required checks、密钥扫描、分支保护。

**这是最关键也最容易出错的阶段**。常见错误:把团队标准只放进 Claude Code 配置,结果用别的工具的人不受约束(第 4 章反复警告的)。**确保标准在工具中立层。**

**验证**:团队代码质量是否稳住或提升(CI 通过率、返工率)?规范是否对所有人(不只 Claude Code 用户)生效?

## 8.5 阶段 4:组织治理(组织级,持续)

**目标**:企业级强约束、模型/成本治理、持续改进。

做什么:

- 上 managed policy(第 7 章):强制登录组织、最低版本、MCP 白名单、availableModels。
- 定模型分配和成本策略(第 7.2、7.3 节)。
- 文档化 onboarding,度量采纳和质量(第 7.4、7.5 节)。
- 建立规范的持续改进机制(评估驱动)。

**验证**:治理是否在不扼杀效率的前提下守住了安全合规底线?

## 8.6 常见阻力与应对

落地不是技术问题,大半是人的问题。常见阻力:

| 阻力 | 应对 |
|---|---|
| "AI 写的代码不可信" | 不强迫;用第 5 章的审查纪律证明 AI 代码和人代码一样要审、一样可控。从低风险任务起步建立信心。 |
| "我不用 Claude Code,这套跟我无关" | 讲清双层设计(第 4 章):团队标准在工具中立层,对你一样生效;你用什么工具是你的自由。 |
| "定这么多规范太繁琐" | 强调大部分是好工程本就该有的(CI、lint、审查),AI 只是契机。配置即代码让规范自动化,不是增加手工负担。 |
| "怕 AI 泄露代码/密钥" | 讲多重安全护栏(第 6 章):密钥扫描、最小权限、deny 规则、隔离环境。AI 友好的项目反而更安全。 |
| "成本会失控" | 讲成本管理(第 7.3 节):按任务选模型、effort 按需、监控用量、availableModels 限制。 |
| 早期采用者和保守派的分裂 | 分阶段(8.1):让早期采用者先证明价值,用事实而非说服推动保守派。 |

## 8.7 落地的反模式(别这么做)

| ❌ 反模式 | 为什么糟 | ✅ 改法 |
|---|---|---|
| 一步到位全组织铺开 | 没验证价值,阻力大、易翻车 | 分四阶段,每阶段验证 |
| 强制全队统一 AI 工具 | 不现实、阻力大、没必要 | 工具中立层定标准,工具自由(第 4 章) |
| 团队标准只放 Claude Code 配置 | 用别的工具的人不受约束 | 标准放工具中立层强制(第 4、6 章) |
| 先放开自主性再建护栏 | 出事的爆炸半径大 | 先建工具中立护栏(CI/hook),再放手 |
| AI 代码免审查 | 责任不清、质量失控 | AI 代码同等审查,提交者负责(第 5 章) |
| 规范定完就不管了 | 会过时、会漂移 | 评估驱动持续改进(第 7.5 节) |

## 8.8 一页纸路线图

```
阶段1 个人采纳    几个人试用,跑通个人工作流,验证价值          (第1章)
   ↓
阶段2 项目落地    一个项目改 AI 友好 + 双层配置进 git           (第2-3章)
   ↓             ★先建工具中立护栏(CI/git hook)作地基
阶段3 团队推广    推广样板,正视混合工具,定协作规范+护栏        (第4-6章)
   ↓             ★所有"必须做到"的标准放工具中立层强制
阶段4 组织治理    managed policy、模型/成本策略、onboarding、度量  (第7章)
   ↓
持续             评估驱动改进规范                              (第7.5节)
```

贯穿始终的两条:**每阶段验证再扩大;所有团队标准放工具中立层强制(不依赖任何单一 AI 工具)。**
