# Claude Code 融入团队开发：工作流、项目约定与协作规范（第六册）

> 编写时间：2026-06，基于 Claude Code v2.1.x 官方文档（团队/企业设置、模型配置、权限治理）及通用软件工程协作实践。
> 与第一册的区别：第一册讲 Claude Code 的**功能**（命令、Hooks、Skills……）；这一册讲怎么把它**落地进真实团队的日常开发**——个人工作流、项目约定、协作规范、治理。

## 这册讲什么、为什么需要它

你已经会用 Claude Code（第一册）。但"一个人用得溜"和"一个团队用得好"是两回事。团队场景会冒出一堆新问题：

- 项目结构怎么约定，才能让 AI（和人）都好上手？
- 哪些配置该进 git、该团队共享？
- **队友用别的 AI 工具/模型（Cursor、Copilot、GPT 系……）怎么办——这套规范还成立吗？**
- AI 生成的代码谁来审、怎么审、谁负责？
- 怎么在团队层面强制标准，而不是靠每个人自觉？
- 企业级怎么治理、控成本、带新人？

这一册系统地回答这些。它不是功能手册，是**团队落地与治理手册**。

## 全书主线：双层设计

整本书围绕一条核心原则展开——**把团队规范分成两层**：

```
┌─────────────────────────────────────────────────────┐
│  工具中立层（Tool-Neutral Layer）                       │
│  项目结构 · 文档约定 · CI · git pre-commit · linter ·   │
│  PR 规范 · 分支保护 · AI 代码审查纪律                    │
│  → 对所有人生效,无论用什么 AI 工具/模型                   │
├─────────────────────────────────────────────────────┤
│  Claude Code 配置层                                    │
│  settings.json · hooks · CLAUDE.md · commands · agents │
│  → 给用 Claude Code 的成员额外的自动化便利               │
└─────────────────────────────────────────────────────┘
```

**最重要的一条推论**（直接回答"队友用别的工具怎么办"）：

> **想让一条规则对团队所有人生效,就在工具中立层强制它（CI、git hooks、分支保护、required checks）。** Claude Code 的 hooks 和权限只约束用 Claude Code 的人——别指望它能拦住一个用 Cursor 的同事提交不合规的代码。Claude Code 配置层是"锦上添花的自动化"，不是"团队标准的执行者"。

这条原则会贯穿每一章。混合工具团队（这是大多数真实团队的现状）尤其要分清这两层。

## 三条贯穿全册的硬规则

1. **要对所有人生效的规则,放工具中立层强制**：CI / git hooks / 分支保护 / linter。Claude Code 配置只是给 Claude Code 用户的便利层（第 0、4、6 章）。
2. **AI 生成的代码,人负责**：无论代码来自 Claude、GPT 还是别的，提交它的人对它负全责，必须经人审查、有测试。这条与模型/工具无关（第 5 章）。
3. **配置即代码**：能进 git 的团队约定都进 git，让规范可版本化、可审查、可复制——而不是散落在每个人脑子里或本地（第 3 章）。
