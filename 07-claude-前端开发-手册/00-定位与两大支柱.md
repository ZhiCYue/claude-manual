# 第 0 章 · 定位与两大支柱

这一章定位本册、建立两大支柱的心智模型,并讲清它和前六册的关系。

---

## 0.1 这册的定位

前六册是通用的 AI 辅助开发能力。这一册把它们**落到前端这个具体领域**,并加一个新主题:规格驱动开发。

```
通用能力(前六册)              本册(落到前端 + 新方法)
─────────────────────        ──────────────────────────
① Claude Code 使用      ──→   前端项目里怎么用
② API/Agent SDK                (本册不重复,需要时引用)
③ MCP                   ──→   前端用得上的 MCP(如 Playwright)
④ Prompt Engineering    ──→   前端任务怎么给指令
⑤ Agent 架构            ──→   前端的"计划→执行→验证"
⑥ 团队开发(双层设计)    ──→   前端的工具中立质量门
                              + 新增:规格驱动开发(SDD)/OpenSpec
```

所以本册不是孤立的——它假设你读过前几册(尤其第六册的双层设计),在那个基础上谈前端的具体实践。

## 0.2 为什么 AI 辅助前端需要专门讲

前端有几个特点,让"AI 辅助"既特别有用、也特别容易出问题:

- **可视化**:前端有 UI,AI 能"看到"结果(截图、浏览器)——这是地面真相的金矿(第五册第 3 章),但你得给它这个能力。
- **生态变化快**:构建工具、框架、React 本身都在快速演进,AI 的知识可能过时,你得知道当前该用什么(第 1、3 章)。
- **容易"形似神不似"**:UI 看起来对、实则交互/边界/可访问性有问题——AI 尤其容易这样,需要 spec 和验证兜住(第 6–8 章)。
- **组件化、约定多**:项目的组件约定、状态管理模式,AI 猜不出,得写下来(第 4 章)。

## 0.3 两大支柱

把 AI 辅助前端做好,靠两根柱子。理解它们,全册的结构就清楚了。

### 支柱一:好的工程环境(让 AI 好干活)

AI 在一个好的前端工程环境里才能干得准。这个环境包括:

- **对的构建工具**(第 1–2 章):Vite/Next/webpack 选对,开发体验和 AI 的迭代速度都受益。
- **现代的 React 模式**(第 3 章):React 19 的范式,AI 写出来的代码才不过时。
- **AI 友好的项目结构**(第 4 章):清晰的结构、成文的约定,AI 知道东西放哪、该怎么写。
- **测试作地面真相**(第 4、5 章):Vitest/Playwright 让 AI 能验证"改对没改对"。
- **工具中立的质量门**(第 5 章):lint/类型/测试在 CI 强制,承接第六册——对所有人(所有 AI 工具)生效。

这根柱子本质上是前几册的延续:**为 AI 设计好环境。**

### 支柱二:规格驱动(让 AI 不跑偏)

这是本册的新内容,也是回答"怎么让 AI 实现功能时不跑偏"的核心。

AI 写前端最大的风险是**不可预测**——你说"加个登录页",它可能加得跟你想的不一样。规格驱动开发(Spec-Driven Development, SDD)的解法:

> **写任何代码之前,先就"要建什么"达成一致(写成 spec),AI 再照着实现。**

这让 AI 的产出从"我猜你想要 X"变成"我们审过、白纸黑字定了 X"。第 6 章讲原理,第 7 章讲 OpenSpec 这个落地工具,第 8 章把两根柱子合成完整工作流。

## 0.4 两大支柱怎么配合

两根柱子不是独立的,合起来才是 AI 辅助前端最可靠的形态:

```
需求
  ↓
【支柱二】OpenSpec propose → 生成 spec → 你审查 spec   ← 让 AI 不跑偏:先定对要建什么
  ↓
【支柱二】OpenSpec apply → Claude Code 照 spec 实现
  ↓
【支柱一】在好环境里实现:现代构建、清晰结构、AI 友好的约定   ← 让 AI 好干活
  ↓
【支柱一】测试/浏览器验证(地面真相)→ 工具中立质量门(CI)
  ↓
【支柱二】OpenSpec archive → spec 与代码同步归档
```

**一句话**:**AI 在一个好环境里(支柱一)、照着审过的 spec(支柱二)干活。** 缺了支柱一,AI 没有地面真相、不知约定,干得糙;缺了支柱二,AI 不知道确切要建什么,跑偏。两根都要。

## 0.5 承接第六册:双层设计在前端

第六册的双层设计(工具中立层 + AI 工具配置层)在前端同样成立,本册第 5 章会详谈。提前点出关键:

- **前端的工具中立层**:ESLint / Prettier / TypeScript / Vitest / Playwright 配置 + git pre-commit + CI。对所有人(用 Claude Code / Cursor / 裸写)生效。
- **规格驱动也是工具中立的**:OpenSpec 支持 25+ AI 助手,它的 spec 是一份所有工具都能读的共享契约——正是第六册"工具中立的唯一来源"在前端的体现(第 7 章会点明)。

## 0.6 本册的诚实前提

前端生态变化极快。本册基于 **2026 年 6 月**的现状编写(React 19、Vite 主导、Next.js App Router 稳定、webpack 偏 legacy)。两个提醒:

1. **版本和具体配置会变**——读到本册时,先确认当前最新(官方文档永远是准绳)。本册讲的**原则和选型思路**比具体版本号更持久。
2. **AI 的前端知识可能过时**——这正是第 1、3 章存在的意义:你得知道当前该用什么,才能判断 AI 给的方案是不是过时的(比如它可能还在用停止维护的 CRA、或过时的状态管理模式)。
