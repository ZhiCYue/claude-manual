# 第 1 章 · 个人日常工作流

团队落地的起点是每个人自己用得顺。这一章讲怎么把 Claude Code 融进你一天的真实开发——什么任务交给它、什么自己做、怎么协作而不是甩锅给 AI。

---

## 1.1 一天的开发,Claude Code 落在哪

不是"打开 Claude Code 让它写完整个功能"。真实的融入是**穿插在你本来的开发流里**,在合适的环节用对的方式。一个典型的一天:

```
晨会后,认领一个任务
   ↓
① 理解现状        → 让 Claude Code 调研:"这个功能现在怎么实现的?涉及哪些文件?"
   ↓                （它只读地探索,你快速建立全局认知）
② 出方案          → Plan Mode:"我要加 X,给实施计划,考虑对 Y 的影响"
   ↓                （你审计划,这是纠错成本最低的点）
③ 实现            → 批准计划,它按计划改;你盯着、随时 Esc 纠偏
   ↓
④ 验证            → "跑测试确认" / "用浏览器实际操作一遍"
   ↓                （让它自我验证,而非你全靠肉眼）
⑤ 自审 + 提交      → 自己过一遍 diff → 写 PR → 进入团队审查流程(第 5 章)
```

这套"探索→计划→执行→验证"就是第一册第 10 章的黄金四步法,这里是它在日常节奏里的落点。

## 1.2 什么交给 AI,什么自己做

一个实用的判断框架。不是"能交给 AI 的都交",而是按**任务性质**分:

| 适合交给 Claude Code | 自己主导(AI 辅助) |
|---|---|
| 机械重复(批量改名、补类型、补测试) | 核心架构决策 |
| 探索陌生代码、搜索调用链 | 涉及业务取舍的判断 |
| 写样板、脚手架、初稿 | 安全敏感、难撤销的改动 |
| 写/修测试(明确的验收标准) | 你说不清想要什么的探索 |
| 解释代码、生成文档初稿 | 最终的质量把关和负责 |

**核心原则**(贯穿全册的第 2 条):**AI 写的,你负责。** 你按下提交键的那一刻,这段代码就是你的——无论是 AI 写的还是你写的。所以"自己做"那列的本质是:**需要你负责、需要你判断的事,AI 可以辅助,但决定权和责任在你。**

## 1.3 给好任务的几个习惯

呼应第四册(Prompt Engineering),团队日常里最高频的几个:

- **给目标和验收,不给微观步骤**:"把登录接口的报错改成中文,改完确保 `make test` 全绿"——比"打开 auth.py 第 45 行改字符串"好。
- **给现场**:报错原文、复现步骤、相关文件 `@` 出来。
- **划边界**:"只改 `src/api/` 下的文件,别动 migrations"。
- **要自我验证**:"改完跑 X 确认"——把循环的终止条件从"我觉得好了"变成"测试证明好了"。

> 这些是第四册的核心。团队场景里它们更重要——因为你的产出要进团队审查,措辞不清导致的返工是团队成本,不只是你的。

## 1.4 会话卫生:一任务一会话

第一册第 2 章的纪律,在高强度的团队日常里尤其值钱:

- **一个任务一个会话**,做完 `/clear`。旧任务的残留会拖慢、稀释下一个任务。
- 长任务用 `/compact` 压缩(带保留指示),别让上下文爆掉再被动压缩。
- 随时 `/context` 看占用,建立"窗口体感"。

为什么团队场景更重要:你一天可能切换好几个任务/工单,干净的会话边界让每个任务都从清爽的上下文开始,产出质量更稳。

## 1.5 模型按任务选(个人层面)

第一册第 2 章和官方模型配置:不同任务用不同模型,省时省额度。

```
/model opus        # 架构设计、疑难 bug、大重构 —— 要最强推理
/model sonnet      # 日常编码、常规修改 —— 快且省
/model haiku       # 机械批量任务 —— 最快最省
/model opusplan    # 计划阶段用 opus,执行阶段自动切 sonnet —— 两全
```

- `opusplan` 是个聪明的默认:复杂任务用 Opus 规划、Sonnet 执行,质量和成本兼顾。
- `/effort` 调思考深度:简单任务 `low`/`medium` 省 token,难任务 `high`/`xhigh`。
- 一次性深推理:prompt 里加 `ultrathink` 关键词,只这一轮加深。

> 团队层面的模型分配策略(谁能用 opus、怎么控成本)是第 7 章的治理话题。这里是个人层面:**按任务难度选模型,别一律用最贵的。**

## 1.6 打断与回退:别干等它做错

第一册第 1 章的两个救命操作,团队日常高频:

- **`Esc`**:它方向跑偏时立即打断,补充或纠正。不丢上下文。新人最该克服的心理障碍——别干等它把错的事做完。
- **`Esc Esc` / `/rewind`**:回退到之前的检查点(含代码改动)。让它大胆尝试,跑偏了零成本回退。

## 1.7 个人工作流自检

把这一章落成习惯:

- [ ] 复杂任务先 Plan Mode 出方案,审完再执行
- [ ] 任务有明确验收标准,且要求 AI 自我验证
- [ ] 一任务一会话,做完 `/clear`
- [ ] 按任务难度选模型(别一律最贵)
- [ ] 跑偏立即 `Esc`,不干等
- [ ] **提交前自己过一遍完整 diff**——你对这段代码负责(第 5 章会讲团队审查)

> 个人用顺了,才谈得上团队落地。但注意:**个人习惯不是团队标准。** 你自觉"提交前跑 lint"很好,但要让全队都做到,得靠工具中立层强制(第 0 章双层原则,第 6 章详谈)——这是个人和团队的分水岭。
