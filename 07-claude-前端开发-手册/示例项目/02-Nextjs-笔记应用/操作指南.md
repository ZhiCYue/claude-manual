# 实战示例二 · Next.js 笔记应用（操作指南）

一个用**当前全栈主流栈**(Next.js App Router + React 19 Server Components + Server Actions)做的笔记应用。演示 React 19 的"全栈组件模型"——服务端直接取数、Server Action 改数据,这是 SPA 做不到的。

> 成品功能:列出笔记、添加笔记。数据在服务端(这里用进程内的模拟 db),通过 Server Component 渲染、Server Action 修改。

---

## 第 0 步:你将得到什么

```
notes-app/
├── package.json
├── AGENTS.md                  # 项目约定(第4章)
├── lib/
│   └── db.ts                  # 模拟数据层(进程内)
└── app/                       # App Router(约定式路由)
    ├── page.tsx               # 首页:Server Component,直接取数渲染(第3.2节)
    └── notes/
        ├── actions.ts         # Server Actions(第3.3节)
        └── NoteForm.tsx       # Client Component,用 useActionState
```

本文件夹已提供这些源码,可直接复制。

## 第 1 步:脚手架(主流命令)

```bash
# Next.js 官方脚手架 —— 2026 全栈 React 的主流(第1章)。App Router 现在是默认。
npx create-next-app@latest notes-app
# 选择:TypeScript √、App Router √、Tailwind(可选)
cd notes-app
```

> 对照第 1 章:要 SSR/SEO/全栈一体 → Next.js。它内部用 Turbopack(不再是慢 webpack),开发体验好。

## 第 2 步:理解 React 19 的全栈心智模型(第 3 章)

这是本示例和 SPA 示例最大的不同。回顾第 3.1 节:

> React 19 把心智从"跑在浏览器的 JS 库"变成"横跨服务端和客户端的全栈组件模型"。

具体到 Next.js App Router:

- **组件默认是 Server Component**——在服务端渲染,**直接 `await` 取数**,不进客户端包(第 3.2 节)。
- 需要交互(state、事件)的组件才标 `"use client"` 变 Client Component。
- 改数据用 **Server Action**(`"use server"`),客户端表单直接 `action={...}` 调用(第 3.3 节)。

## 第 3 步:Server Component 直接取数(第 3.2 节)

`app/page.tsx`(本文件夹已提供)是个 **Server Component**——注意它直接 `await` 取数据,**没有 useEffect、没有 loading state**:

```tsx
// 这是 Server Component(App Router 默认)。在服务端执行。
export default async function Home() {
  const notes = await getNotes();   // 直接 await,服务端取数
  return (
    <main>
      <h1>笔记</h1>
      <NoteForm />               {/* Client Component,处理交互 */}
      <ul>{notes.map((n) => <li key={n.id}>{n.text}</li>)}</ul>
    </main>
  );
}
```

对比 SPA(示例一)用 TanStack Query + useEffect 那套——全栈框架里,服务端组件直接取数,客户端包更小、首屏更快。

## 第 4 步:Server Action 改数据(第 3.3 节)

`app/notes/actions.ts`(已提供)是 **Server Action**——标 `"use server"`,在服务端执行,改完用 `revalidatePath` 让页面刷新:

```typescript
"use server";
export async function addNote(formData: FormData) {
  const text = String(formData.get("text") ?? "").trim();
  if (!text) return;
  await createNote(text);
  revalidatePath("/");   // 让首页重新渲染,显示新笔记
}
```

`NoteForm.tsx`(Client Component)用 `useActionState` 调它,自动管 pending 状态(第 3.3 节):

```tsx
"use client";
const [, formAction, isPending] = useActionState(/* ... */);
// <form action={formAction}> + 按钮 disabled={isPending}
```

## 第 5 步:用规格驱动建功能(第 6-7 章)

和示例一一样,较大功能走规格驱动。比如要加"删除笔记":

```
/opsx:propose "删除笔记"
```

审 spec 时确认:用 Server Action 删除、删后 revalidate、有确认交互。审满意后 `/opsx:apply`。

> 全栈应用尤其受益于规格驱动:涉及服务端逻辑、数据变更,先定清楚 spec 再让 AI 写,避免它在服务端/客户端边界上跑偏。

## 第 6 步:测试与质量门(第 4-5 章)

- **e2e(Playwright)**:跑通"添加笔记→出现在列表"的真实路径——全栈应用的地面真相。
- **质量门**:lint/typecheck/build/e2e 进 CI(用第七册 `模板/ci.yml`),对所有人强制。

让 Claude Code 改完"用 Playwright 实际打开页面验证添加流程"。

## 第 7 步:跑起来

```bash
npm run dev        # 打开 http://localhost:3000
```

你会看到笔记应用:输入笔记、提交,页面通过 Server Action + revalidate 刷新显示。注意整个取数/改数据都在服务端,客户端只有表单交互那一小块 JS。

## 这个示例教会你什么

演示了 SPA 示例做不到的全栈模型:

```
Server Component 直接取数(第3.2) → 客户端包更小、首屏快
Server Action 改数据(第3.3) → 客户端表单直接调服务端函数
useActionState 管表单状态(第3.3) → 不用手写 loading/error
```

**何时选这条路(对比示例一)**:要 SEO、首屏性能、服务端逻辑、全栈一体 → Next.js(本示例);纯客户端、SEO 不重要、要最大控制 → Vite SPA(示例一)。选型见第 1.3 节决策树。

**练习**:用规格驱动给它加"编辑笔记"——`/opsx:propose "编辑笔记"`,审 spec(用 Server Action、乐观更新可考虑 useOptimistic),apply。
