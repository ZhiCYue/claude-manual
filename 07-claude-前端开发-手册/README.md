# AI 辅助前端开发实践：工程规范、构建选型与规格驱动开发（第七册）

> 编写时间：2026-06，基于当前前端生态实测核对（React 19、Vite、Next.js、webpack 5 的 2026 年现状）及 OpenSpec 官方文档。
> 定位：把前五册的 AI 辅助开发能力、第六册的团队工程实践，**落到前端这个具体领域**，并引入"规格驱动开发"这个让 AI 不跑偏的关键方法。

## 这册讲什么

你已经会用 Claude Code、会调 API、会写 prompt、会搭 agent、会带团队（前六册）。这一册回答一个具体而高频的问题：

> **在真实的前端项目里（React 技术栈），怎么把 AI 辅助开发做好？怎么选构建工具？怎么让 AI 实现功能时不跑偏？**

它覆盖你问的三块，并织进一条主线：

1. **前端工程规范** —— 让项目对 AI(和人)友好,承接第六册的双层设计。
2. **构建工具与框架选型** —— Vite / Next.js / webpack 的 2026 现状与决策(含 webpack + React 的实战)。
3. **规格驱动开发(SDD)与 OpenSpec** —— 先定 spec 再写码,让 AI 的产出确定、可审查。

## 全书主线：AI 辅助前端的两大支柱

把"AI 帮你写前端"做好,靠两根支柱:

```
┌──────────────────────────┬──────────────────────────┐
│  支柱一:好的工程环境       │  支柱二:规格驱动           │
│  (让 AI 好干活)           │  (让 AI 不跑偏)           │
├──────────────────────────┼──────────────────────────┤
│  · 清晰的项目结构          │  · 写码前先定 spec         │
│  · 现代构建工具            │  · spec 可审查、进 git     │
│  · 类型 + 测试(地面真相)   │  · AI 照 spec 实现         │
│  · 工具中立的质量门         │  · OpenSpec 把这套自动化    │
│  (第 1–5 章)              │  (第 6–8 章)              │
└──────────────────────────┴──────────────────────────┘
```

- **支柱一**承接前几册:好的前端工程环境(结构、工具、测试、质量门)让 AI 有地面真相、有清晰约定,干得准。
- **支柱二**是这册的新内容:规格驱动开发——把第一册的 Plan Mode、第五册的"探索→计划→执行"做成**持久化、可审查、工具中立**的工作流,OpenSpec 是它的落地工具。

两根柱子合起来:**AI 在一个好环境里、照着审过的 spec 干活**——这是 AI 辅助前端最可靠的形态。

## 三条贯穿全册的硬规则

1. **新建 React 项目,默认 Vite 或 Next.js**：webpack 已偏 legacy(CRA 停止维护),用它需要明确理由(已有项目、微前端、深度定制)。第 1 章。
2. **测试是 AI 的地面真相**：前端的 Vitest/Playwright 不只是质量保证,更是让 AI 能自我验证"改对没改对"的依据(承接第五册第 3 章)。第 4、5 章。
3. **复杂功能先定 spec 再让 AI 写**：把"我觉得你想要 X"变成"我们审过、写成 spec 的 X"。OpenSpec 把这套自动化,且工具中立(混合工具团队都能用)。第 6–8 章。
