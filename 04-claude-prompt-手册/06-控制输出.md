# 第 6 章 · 控制输出

模型能力很强，但默认的输出格式、详略、风格未必合你的需求。这一章讲怎么精确控制输出——格式、详略、markdown、并讲清为什么 prefill 已成历史、用什么替代。

---

## 6.1 说"做什么"，而非"别做什么"

控制格式最有效的第一原则：**告诉模型你想要什么，而不是你不想要什么。**

| ❌ 说"别做" | ✅ 说"要做" |
|---|---|
| 不要用 markdown | 用流畅连贯的散文段落写 |
| 别太啰嗦 | 用 2–3 句话简明回答 |
| 不要用项目符号 | 把要点融进完整的句子里 |

为什么？"别用 markdown"让模型去想"markdown 是什么、我怎么避开"，反而可能强化对 markdown 的注意；"用流畅的散文段落"直接给了目标，模型照着做就行。**正面描述目标，比负面描述禁区更可靠。**

## 6.2 用 XML 标签标定输出格式

第 3 章讲过，可以让模型把输出放进指定标签：

```
把回复的散文部分写在 <smoothly_flowing_prose> 标签里。
```

这既指定了格式（散文），又给了你可靠提取的边界。

## 6.3 让 prompt 的风格匹配你想要的输出

一个微妙但有效的技巧：**你 prompt 的格式风格会影响模型输出的风格。**

如果你的 prompt 里全是 markdown 项目符号，模型倾向于也用项目符号回。如果你还在为输出格式发愁，试试**把 prompt 写成你想要的输出的样子**——想要纯散文输出，就把 prompt 里的 markdown 去掉，用平实的段落写。

## 6.4 控制 markdown 和详略

最新模型默认比较简洁，且偏爱 markdown 结构。要精细控制，给明确指引。比如想要少 markdown、多散文：

```
<避免过度markdown和项目符号>
写报告、文档、技术说明、分析或任何长文时，用清晰流畅的散文，完整的段落和句子。
用标准的段落分隔来组织，markdown 主要保留给 `行内代码`、代码块和简单标题。
避免用 **加粗** 和 *斜体*。

不要用有序列表（1. ...）或无序列表（*），除非：a) 你在呈现真正离散的条目，列表是最佳选择；
或 b) 用户明确要求列表或排名。

不要用项目符号罗列，把它们自然地融进句子里。这条尤其适用于技术写作。
用散文而非过度格式化能提升用户满意度。绝不要输出一连串过短的项目符号。

目标是可读、流畅的文本，引导读者自然地理解，而不是把信息切碎成孤立的点。
</避免过度markdown和项目符号>
```

这个片段可以直接放进系统提示。反过来，想要更多结构化（表格、列表）也同理——明确描述你要的结构。

## 6.5 控制 LaTeX 数学输出

最新模型默认用 LaTeX 渲染数学公式。如果你的展示环境不支持 LaTeX、要纯文本：

```
回复用纯文本，不要用 LaTeX、MathJax 或任何标记记号（如 \( \)、$、\frac{}{}）。
所有数学表达式用标准文本字符写（除法用 "/"、乘法用 "*"、指数用 "^"）。
```

## 6.6 详略与沟通风格

最新模型比老模型更简洁、更自然——这意味着它可能**跳过工具调用后的总结**，直接进行下一步。如果你想要更多可见性（看到它在想什么、做了什么）：

```
完成涉及工具调用的任务后，简要总结一下你做了什么。
```

反过来，如果它太啰嗦，正面描述你要的简洁度（第 6.1 节的原则）：

```
回复要简洁、聚焦。跳过非必要的背景，示例保持最少。
```

> **第 10 章会深入**：Fable 5 / Opus 4.8 的沟通风格各有特点，详略需要针对性调。这里先掌握通用手法：详略问题用正面描述目标来调，必要时配示例。

## 6.7 结构化输出（强制保证）

前面的技法都是"引导"。当你需要**保证**输出符合某个格式（机器要解析的 JSON），用结构化输出——它是强制的。回顾第二册第 2 章：

```python
response = client.messages.parse(
    model="claude-opus-4-8",
    max_tokens=1024,
    messages=[{"role": "user", "content": "提取联系人信息：……"}],
    output_format=MyPydanticModel,    # 保证输出符合这个 schema
)
contact = response.parsed_output      # 验证过的对象
```

**引导 vs 保证**：要 JSON 但偶尔格式飘了能容忍 → prompt 引导 + 重试；机器解析、绝不能错 → 结构化输出。

## 6.8 告别 prefill（重要的现代变化）

**这是从老资料迁移过来的人必须知道的变化。**

老的 prompt engineering 教材有一招叫"prefill"——在 assistant 轮预填几个字，强制模型沿着某个格式继续（比如预填 `{"` 强制它输出 JSON）。

**从 Claude 4.6 起，prefill 不再支持。** 给这些模型（含 Fable 5、Opus 4.6/4.7/4.8、Sonnet 4.6）发带预填 assistant 消息的请求会返回 **400 错误**。

为什么取消？因为模型的智能和指令遵循已经强到不需要 prefill 了。各种 prefill 的老用法，现在这样替代：

| prefill 的老用法 | 现代替代 |
|---|---|
| 强制 JSON/特定格式 | **结构化输出**（`output_config.format`），或直接要求模型遵循格式 + 重试 |
| 强制分类标签 | 带 enum 字段的工具，或结构化输出 |
| 跳过前言（"以下是总结："） | 系统提示里说"直接回答，不要前言。不要以'以下是''根据'开头" |
| 绕过不当拒绝 | 现在很少需要——模型拒绝得更恰当了，清晰的用户消息即可 |
| 续写中断的回复 | 把续写移到用户消息："你之前的回复在 `[原文]` 处中断了，从那里继续" |
| 周期性注入上下文 | 注入到用户轮，或通过工具/上下文压缩时注入 |

> **行动项**：如果你有用 prefill 的老 prompt，迁移它。最常见的"强制 JSON"用结构化输出替代——更可靠，且不会 400。

## 6.9 输出控制速查

| 想要 | 怎么做 |
|---|---|
| 特定格式（引导） | 正面描述目标 + XML 标签标定 + prompt 风格匹配 |
| 特定格式（保证） | 结构化输出 `output_config.format` |
| 少 markdown / 纯散文 | 第 6.4 节的片段放系统提示 |
| 纯文本数学 | 第 6.5 节的 LaTeX 禁用指令 |
| 更详细 / 更简洁 | 正面描述想要的详略，配示例 |
| 跳过前言 | "直接回答，不要前言" |
| 续写中断 | 把中断点和续写要求放用户消息 |
| ~~prefill~~ | 已废弃，用上面的替代方案 |
