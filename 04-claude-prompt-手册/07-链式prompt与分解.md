# 第 7 章 · 链式 prompt 与分解

有些任务复杂到单个 prompt 搞不定，或者你需要在中间步骤插手检查。这时把任务拆成**串联的多次调用**——每一步一个聚焦的 prompt。这一章讲怎么拆、以及最有用的"自我修正"模式。

---

## 7.1 为什么要拆

一个 prompt 同时要"分析 + 决策 + 执行 + 检查"，模型容易顾此失彼——分析得潦草、或者跳过检查。把它拆成几步，每步只干一件事，每步都能做到位。

好处不止于此：

- **可检查中间结果**：每步是独立的 API 调用，你能 log、评估、在任意点分支。
- **可强制流程结构**：保证"先做 A 再做 B"，而不是模型自由发挥顺序。
- **每步更聚焦**：单一职责的 prompt 比万能 prompt 质量更稳。

## 7.2 什么时候该拆，什么时候不该

**重要的现代判断**：最新模型有 adaptive thinking 和子代理编排能力，**大部分多步推理它能在内部完成**。你不必把所有多步任务都手动拆成链。

| 该用链式 | 不必用链式（让模型内部处理） |
|---|---|
| 需要检查/记录中间输出 | 纯粹的多步推理（思考能搞定） |
| 要强制特定的流水线结构 | 模型能自主规划的探索性任务 |
| 中间步骤要人工审批 | 简单的"分析然后输出" |
| 不同步骤用不同模型/配置 | |

换句话说：**因为"我需要在中间插手"而拆，不是因为"任务有多步"而拆。** 任务有多步、但你不需要看中间，交给模型的思考（第 5 章）更省事。

## 7.3 最常见的链式模式：自我修正

最有价值的链式模式是**自我修正**——生成草稿 → 让模型对照标准审查 → 让模型根据审查修订。三步，每步一个独立调用：

```
步骤 1（生成）：
  prompt: "写一篇关于 X 的产品介绍，面向非技术用户，500 字。"
  → 得到草稿 draft

步骤 2（审查）：
  prompt: "审查下面这篇产品介绍，对照标准：
           - 非技术用户能否看懂（无术语黑话）？
           - 是否突出了核心价值？
           - 长度是否合适？
           列出具体问题。
           <draft>{draft}</draft>"
  → 得到审查意见 review

步骤 3（修订）：
  prompt: "根据审查意见修订这篇介绍。
           <draft>{draft}</draft>
           <review>{review}</review>"
  → 得到终稿
```

为什么这个模式好？呼应第 5.5 节和第一册第 0 章的原则：**模型审查和批判既有内容的能力，强于一次写完美。** 把"写"和"审"分成两步独立调用，审查时模型没有"我刚写的"这个包袱，更客观——就像人请别人校对而非自己校对。

## 7.4 链式的代码骨架

链式本质上就是普通的代码流程，每步一个 API 调用：

```python
import anthropic
client = anthropic.Anthropic()

def call(prompt: str) -> str:
    resp = client.messages.create(
        model="claude-opus-4-8",
        max_tokens=4096,
        messages=[{"role": "user", "content": prompt}],
    )
    return next(b.text for b in resp.content if b.type == "text")

# 三步链
draft = call("写一篇关于智能水杯的产品介绍，面向非技术用户，500 字。")

review = call(f"""审查这篇产品介绍，对照标准：
- 非技术用户能否看懂？
- 是否突出核心价值？
- 长度合适吗？
列出具体问题。

<draft>{draft}</draft>""")

final = call(f"""根据审查意见修订这篇介绍。

<draft>{draft}</draft>
<review>{review}</review>""")

print(final)
```

注意每步用 XML 标签（第 3 章）把上一步的产物清晰地传入下一步。

## 7.5 其他链式模式

### 流水线（每步不同职责）

```
原始数据 → [提取] → 结构化数据 → [分析] → 洞察 → [生成报告] → 终稿
```

每个箭头是一次调用。适合"提取 → 分析 → 呈现"这类有明确阶段的任务。

### 分支（按中间结果走不同路）

```
分类工单 → 若是 bug → [走 bug 处理链]
           若是咨询 → [走咨询回复链]
```

第一步分类，你的代码根据结果调用不同的后续链。

### 扇出再合并（并行 + 汇总）

```
        ┌→ [分析维度 A] ┐
大任务 ─┼→ [分析维度 B] ┼→ [汇总] → 结论
        └→ [分析维度 C] ┘
```

三个维度并行调用（互不依赖），再用一个调用汇总。注意：这种"并行子任务再汇总"在 agent 场景里，最新模型可以用**子代理**自动完成（第 9 章），不必手动编排。

## 7.6 链式 vs 让模型内部处理（再强调）

随着模型变强，很多过去需要手动链式的场景，现在交给模型的思考和子代理能力更省事。判断的关键还是那句：

> **你需要在中间插手吗？** 需要看/改/审批中间结果 → 链式；只要最终结果 → 让模型内部一气呵成。

不要因为"任务复杂"就条件反射地拆链——先试试一个好的单 prompt（配思考），不行再拆。这呼应第 0 章的方法论：**先用简单的，简单的不够再上复杂的。**

## 7.7 链式与上下文管理

长链式任务里，每步的产物会累积。如果产物很大（长文档、大量数据），注意：

- 不必每步都把所有历史传下去——只传下一步真正需要的。
- 固定不变的部分（如一份参考文档）可以用 prompt caching 缓存（第二册第 6 章）。
- 超长的多步任务，考虑把状态存到文件而非全塞进 prompt（第 9 章的状态跟踪、第一册第 10 章的"文档接力"）。
