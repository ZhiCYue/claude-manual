# 第 10 章 · 针对最新模型调优

这是全册的"前沿"一章。不同模型有不同的行为性格——Fable 5 和 Opus 4.8 比老模型更字面、更简洁、更自主。**为老模型调好的 prompt，搬到新模型上往往需要重调。** 这一章讲怎么针对最新模型精调。

---

## 10.1 核心认知：新模型更字面、更自主、更简洁

最新模型（Opus 4.5/4.6/4.7/4.8、Fable 5）相比前几代，有几个一致的行为转变：

1. **更字面地遵循指令**——它严格按你说的做，不会自作主张地泛化或推断你没说的需求。
2. **更主动自主**——更愿意动手、更倾向于用工具和子代理、长程任务更持久。
3. **更简洁更自然**——默认更简洁、更口语化、更少机器味，可能跳过冗长的总结。
4. **对系统提示遵循度更高**——你写的规则它认真执行。

这四点都是双刃剑。**好处**：你能精确控制它。**代价**：为老模型写的 prompt（往往是为了"对抗"老模型的迟钝、偷懒、过度啰嗦）现在会**过度生效**。

> **第三册第 10 章和迁移指南反复强调的一句话**：迁移到新模型，最常见的工作不是改 API（那是机械的），而是**重调 prompt**——把过激的措辞松绑、把为老模型加的脚手架拆掉。

## 10.2 从老模型迁移：该拆的脚手架

如果你有为老模型（4.5 之前）调好的 prompt，迁移到 4.6+/Fable 5 时检查这些：

### 把过激的工具/技能语言松绑

```
# ❌ 老模型需要这种强度来对抗"不爱用工具"
CRITICAL: 你必须使用此工具！如有疑问，务必使用！

# ✅ 新模型这样就够，过激语言会导致过度触发
需要时使用此工具。
```

具体替换：

- "默认使用 [工具]" → "当 [工具] 能增进对问题的理解时使用它"
- "如有疑问，使用 [工具]" → **直接删掉**（新模型会恰当触发，这句会导致过度触发）
- "CRITICAL: 你必须……" → "……"（去掉强调，正常陈述）

### 拆掉"反偷懒"脚手架

老模型常需要"务必详尽""不要偷懒""超出基础"这类话来对抗惰性。新模型默认就主动，这些话可能导致过度工作、过度思考。**A/B 测试一下去掉它们**——很可能去掉后更好。

### 拆掉强制进度汇报的脚手架

```
# ❌ 老模型可能需要这样才汇报进度
每 3 次工具调用后，总结一下进度。

# 新模型：删掉，它自己会做（甚至可能做得过多，见 10.4）
```

### 迁移走 prefill 和 budget_tokens

- prefill 在 4.6+ 会 400 → 用结构化输出（第 6.8 节）
- `budget_tokens` 在 Fable 5/Opus 4.7/4.8 会 400 → 用 adaptive thinking + effort（第二册第 2 章）

## 10.3 effort 是要测的维度，不是固定值

在新模型上，**effort 比以往任何时候都重要**，而且不该反射性地拉满。

老模型时代很多人反射性地用最高档求智能。但 Opus 4.8 智能上限更高，**起点设 `high`、然后迭代**，而不是默认 `xhigh`。

```
在你自己的评估集上扫一遍 medium / high / xhigh，
按"智能 ↔ 延迟 ↔ 成本"的权衡逐路由选择——这个关系不是单调的：
高 effort 在 agent 任务上常常减少总轮数和总成本，
而某些任务 medium 用更少时间给出同样好的结果。
```

把 `max` 留给极难、对延迟不敏感的情况。**别凭感觉设 effort，用评估扫**（呼应第 0 章方法论）。

## 10.4 控制详略与沟通风格

新模型默认简洁，但 Fable 5 和 Opus 4.8 各有特点，需要针对性调。

### 如果它太简洁（跳过了你想看的总结）

```
完成涉及工具调用的任务后，简要总结你做了什么。
```

### 如果它太啰嗦（叙述过多）

新模型（尤其 Opus 4.8、Fable 5）在长工具调用会话里可能叙述较多。想要更安静：

```
工具调用之间默认保持安静。只在你有发现、改变方向、或遇到障碍时才写文字——
每种情况一句话。不要叙述常规动作（"现在我要……""让我检查……""正在看……"）。
完成时：一两句话说明结果。不要逐一复述每个文件或测试——用户一直在看。
```

### Fable 5 的"接地"沟通风格

Fable 5 在长 agent 会话深处，可能产生用户难以跟随的文本（密集的箭头链缩写、实现级细节、引用用户没看到的思考）。一个沟通风格附加段能显著缓解：

```
工具调用之间用简洁的速记没问题（那是你在思考，简短是好的）。
但你的最终总结不同：它面向一个没看过任何过程的读者。
开头先给结果：一句话说明发生了什么或你发现了什么。然后是支撑细节。
要在"短"和"清楚"之间选，选清楚。写完整的句子，把术语拼出来而非缩写，
不要用箭头链或你之前自创的标签——读者没有上下文解码它们。
```

## 10.5 Fable 5 的几个专属调优点

Fable 5 是最强模型，有一些独特的行为需要 prompt 配合（详见已加载的模型指南，这里提炼要点）：

### 长程任务：把完整任务规格一次说清

```
（在第一个、措辞充分的初始轮里给出完整任务规格，并用 high/xhigh effort）
```

Fable 5 的长程连贯性部分来自每步更多的推理。配合清晰的前期目标，这种更智能的规划往往同时更高效、更准确。

### 防过度规划

```
当你有足够信息行动时，就行动。不要重新推导对话里已确立的事实，
不要重新争论用户已经做出的决定。如果在权衡选择，给推荐而非详尽罗列。
（这条不适用于思考块）
```

### 防高 effort 下的过度整理

```
不要添加超出任务所需的功能、重构或抽象。修个 bug 不需要周边清理，
一次性操作通常不需要辅助函数。做能良好工作的最简实现。
```

### 给它记忆面

Fable 5 在能把学到的东西写到某处供日后参考时表现明显更好（哪怕一个 .md 文件）。告诉它写哪、用什么格式、日后查阅它。

### `reasoning_extraction` 拒绝

试图让 Fable 5 在回复文本里吐露内部推理的请求，可能被拒（`stop_details.category: "reasoning_extraction"`）。需要推理可见性的应用，应读 `display: "summarized"` 的思考摘要，而非 prompt 它吐露推理。

## 10.6 前端设计：避免"AI 味"

新模型擅长前端，但没引导时会默认收敛到通用模式，产生用户称之为"AI slop"（AI 流水线货）的审美。要做出独特、惊艳的前端：

```
<前端审美>
你倾向于收敛到通用的、"在分布内"的输出。在前端设计上，这造成"AI slop"审美。
避免这个：做有创意、有辨识度、令人惊喜的前端。

聚焦：
- 排版：选美观、独特、有趣的字体。避免 Arial、Inter 这类通用字体，
  选有辨识度的、能提升美感的。
- 配色与主题：committed 到一个连贯的审美。用 CSS 变量保持一致。
  主色配锐利的点缀色，胜过怯懦的、均匀分布的配色。
- 动效：用动画做效果和微交互。一次精心编排的页面加载（错落的渐显）
  比散落的微交互更有冲击力。
- 背景：营造氛围和层次，而非默认纯色。

避免通用的 AI 生成审美：
- 滥用的字体（Inter、Roboto、Arial、系统字体）
- 陈词滥调的配色（尤其白底紫色渐变）
- 可预测的布局和组件模式

有创意地诠释，做出真正为这个语境设计的意外选择。在明暗主题、不同字体、
不同审美间变化。你仍倾向于收敛到常见选择——避免它，跳出框框思考至关重要！
</前端审美>
```

## 10.7 调优方法论：别照搬，要评估

这一章给了大量针对性的 prompt 片段，但最重要的是回到第 0 章的方法论：

> **这些是起点，不是定论。** 模型行为会随版本演进，你的场景也独一无二。把这些片段当假设，用你自己的评估集验证——A/B 测试加了/去了某段引导后的效果，量化对比，保留有效的。

具体到迁移：

1. **先用现有 prompt 跑新模型**，看哪里行为变了（更主动？更简洁？过度触发？）。
2. **针对观察到的变化调**——过度触发就松绑措辞，太简洁就要总结，过度工程就加约束。
3. **一次改一处，跑评估集确认**（第 0 章纪律）。

> 迁移到新模型的真功夫，是**把为旧模型的弱点打的补丁拆掉**。新模型不需要那些补丁，留着它们反而帮倒忙。

## 10.8 最新模型调优速查

| 现象（新模型上） | 调法 |
|---|---|
| 工具/技能过度触发 | 松绑措辞，删"如有疑问就用"，去掉"CRITICAL/必须" |
| 过度思考、拖慢 | 降 effort；加"拿不准直接回答" |
| 太简洁、跳过总结 | "完成后简要总结你做了什么" |
| 太啰嗦、叙述过多 | 加"默认安静，只在有发现/改方向/遇阻时写" |
| 过度工程、加多余抽象 | 第 9.6 节的"避免过度工程"片段 |
| 危险操作不确认 | 第 9.3 节的"自主与安全"片段 |
| Fable 5 长会话文本难懂 | 第 10.4 节的"接地沟通风格"片段 |
| 前端"AI 味" | 第 10.6 节的前端审美片段 |
| 从老模型迁移 | 拆补丁：松绑过激语言、去反偷懒脚手架、迁移 prefill/budget_tokens |
