# 第 3 章 · 动态工作流与自主 Agent

工作流是你提前写死路径,自主 agent 是模型当场决定路径。这一章讲"dynamic workflow"——模型驱动的动态流程——以及自主 agent 的解剖和它需要的护栏。

---

## 3.1 从固定工作流到动态工作流

第 2 章的五模式里,前三个(链式、路由、并行)路径基本固定。但**编排者-工作者**和**评估者-优化者**已经有了"动态"的影子——编排者运行时才决定派几个 worker,评估者循环到合格才停,轮数不定。

把这种"动态"推到极致,就是**自主 agent**:整个路径都由模型决定,没有预设的步骤序列。所谓 "dynamic workflow",就是这个光谱上偏向自主的那一端——**模型根据环境反馈,动态地编排自己的下一步**。

```
固定工作流 ←———————————————————→ 自主 agent
  路径写死    路径部分动态(编排者)   路径完全由模型决定
```

## 3.2 自主 Agent 的循环解剖

回顾第一册第 0 章和第二册第 4 章的 agent loop,这里从架构视角正式解剖。自主 agent 这样运转:

```
① 人类给出方向(或先讨论澄清任务)
        ↓
②[循环开始] 模型基于当前状态规划行动
        ↓
③ 执行模型选择的工具调用
        ↓
④ 从环境获得"地面真相"(ground truth)
   —— 工具结果、代码执行输出、报错
        ↓
⑤ 评估进度、调整方法
        ↓
   需要信息/判断时 → 回到人类问
        ↓
⑥ 完成,或撞到停止条件(迭代上限)→ 退出
   否则 → 回到 ②
```

**关键认识**:

1. **每一步都从环境获得反馈。** 这是自主 agent 能工作的根本——它不是凭空规划,而是看真实的工具结果、真实的报错,据此调整。代码跑失败了,它看到错误信息,改方法重试。这个"环境反馈回路"是 agent 和"一次性生成一长串计划"的本质区别。

2. **实现其实很简单。** 自主 agent 的本质就一句话:**"LLM 在循环里,基于环境反馈使用工具。"** 别被"自主 agent"这个词吓到——它不是什么复杂的架构,是一个简单的循环 + 好的工具 + 好的环境反馈。

3. **停止条件很重要。** agent 会一直转,所以 harness 必须有停止条件:任务完成、迭代上限、预算上限。没有它,agent 可能无限循环或失控烧钱(第二册的 `--max-turns`、第二册第 2 章的 task budget)。

## 3.3 "地面真相"为什么是关键

第 ④ 步的"环境反馈"是自主 agent 可靠性的命脉。**好的 agent 系统,核心是给模型提供高质量、真实的反馈。**

- 让 agent 写代码,给它能跑测试的环境——测试结果是地面真相,它据此知道对没对。
- 让 agent 改 UI,给它能截图的工具(如 Playwright MCP,第三册/第一册)——截图是地面真相,它据此知道改对没改对。
- 让 agent 查数据,给它能验证的来源——多源交叉是地面真相,它据此知道信息可靠没。

呼应第一册的原则:**让 agent 自我验证。** "改完跑测试""改完截图对比"——这些不是装饰,是给 agent 提供地面真相,把它的循环终止条件从"我觉得好了"变成"环境证明好了"。第 4 章讲怎么设计这种环境接口。

## 3.4 何时信任自主性

自主 agent 强大,但它把控制权交给了模型。什么时候敢这么做?

**敢用自主 agent 的前提**:

- **你信任模型的决策**——在这个任务域,模型确实有能力(第 1.5 节四问之"可行性")。
- **错误可恢复**——有测试、有审查、能回滚(四问之"错误成本")。
- **有护栏**——见下一节。
- **在沙箱里测过**——别在生产环境第一次放手。

**不敢用、退回工作流的信号**:

- 任务其实步骤固定(那用工作流更可预测)。
- 错误不可逆且代价高(那要人在回路)。
- 模型在这个域不够可靠(那别赌)。

## 3.5 自主 Agent 的护栏

放手让 agent 自主,不等于不管。护栏是让自主**安全**的关键:

### ① 权限与审批(harness 层)

第一册第 4 章的权限模式、Hooks,在这里是护栏。危险操作(删除、推送、对外发布)要门控。第四册第 9.3 节的"自主与安全平衡"prompt 也是护栏的一部分。

### ② 沙箱隔离

让 agent 在隔离环境跑(容器、一次性 VM、无敏感数据的环境),爆炸半径被限死。Managed Agents 的每会话容器(第二册第 9 章)就是这个思路。

### ③ 停止条件

迭代上限、预算上限、超时。防止失控。

### ④ 可观测性

能看见 agent 在做什么(透明性,第 0.3 节)。看不见的自主是危险的自主。第 7 章详谈。

### ⑤ 人在回路的检查点

关键节点让人审批。不是每步都问(那就不自主了),而是在"难以撤销"的节点问。

> **护栏的设计哲学**(呼应第一册第 4.5 节):给 agent 更大自主权的正确方式,不是关掉所有检查,而是**把边界规则写清楚**(精确的权限 + 护栏 + 沙箱),然后在边界内放手。

## 3.6 动态工作流的一个完整例子

把这章串起来——一个自主编码 agent 怎么动态地完成"修复一个 bug":

```
人类:"tests/test_payment.py 里有个测试失败,修复它"

agent 循环(路径完全由模型动态决定,每次可能不同):
  轮1  规划:先看测试为什么失败 → 调 bash 跑测试
       环境反馈:"AssertionError: 重复扣款"
  轮2  规划:看被测函数 → 调 read 读 payment.py
       环境反馈:函数内容(发现没有幂等检查)
  轮3  规划:加幂等检查 → 调 edit 改代码
       环境反馈:编辑成功
  轮4  规划:验证 → 调 bash 重跑测试
       环境反馈:"1 passed"
  轮5  规划:完成 → 输出总结,退出
```

注意:**没有一步是提前写死的。** 模型看测试失败 → 决定读函数;看到没幂等 → 决定加检查;改完 → 决定重跑验证。每一步都是基于上一步的环境反馈动态决定的。这就是 dynamic workflow / 自主 agent 的实质。

而护栏在背后:edit 操作如果碰受保护文件会被 Hook 拦(第一册第 6 章);跑了 5 轮还没好会撞迭代上限;整个过程你能在界面看见(透明)。**自主在护栏内发生。**
