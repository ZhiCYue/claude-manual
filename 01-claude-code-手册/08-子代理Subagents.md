# 第 8 章 · 子代理（Subagents）

## 8.1 是什么

子代理是主 Claude 通过 `Agent` 工具派生出的"分身"：

- 拥有**全新的、独立的上下文窗口**（不继承主对话历史）
- 可以**单独配置**系统提示、可用工具、模型
- 独立跑完自己的 Agent Loop，最后只把**最终报告**返回主对话

```
主对话（你的窗口）
  │
  ├─→ 派生子代理："全库找出所有处理支付回调的代码路径"
  │      子代理窗口：Grep ×8、Read ×15、共消耗 6 万 token
  │      ↓
  ←── 只返回 800 token 的结论报告
  │
  主窗口只增加了 800 token，而不是 6 万
```

## 8.2 为什么需要它（原理：上下文经济学）

回顾 0.4 节：上下文是稀缺资源，且"脏数据"会稀释注意力。搜索调研类任务的特点是**过程废料极多、有效结论极少**——翻 20 个文件可能只为确认一句话。

子代理的本质是**上下文的进程隔离**：脏活在分身的一次性窗口里干，干完窗口即焚，主对话只收蒸馏后的结论。这有三个收益：

1. 主窗口保持干净，主任务的推理质量不下降；
2. 多个子代理可**并行**，互不污染；
3. 子代理可以用更便宜的模型干粗活（如用 Haiku 做大规模搜索）。

代价也要知道：子代理是**冷启动**的——它不知道主对话聊过什么，所以任务描述必须自包含；另外派生本身有开销，三十秒能干完的小事别动用分身。

## 8.3 内置子代理

| 名字 | 能力 | 用法 |
|---|---|---|
| **Explore** | 只读搜索专家（不能改任何东西） | 「用 Explore 找出所有调用了 deprecated API 的位置」 |
| **Plan** | 架构规划（产出实施方案） | 复杂任务的方案设计 |
| **general-purpose** | 全能型，多步骤研究和执行 | 默认派生类型 |

日常触发方式就是自然语言：

```
> 用 Explore 子代理调查：这个仓库里所有发送邮件的代码路径，
  包括直接调用和通过消息队列间接触发的，要求列出文件:行号
```

## 8.4 自定义子代理

### 交互式创建

```
/agents     # 列出、创建、编辑子代理，可选生成方式（描述需求让 Claude 代写）
```

### 手写文件

存放位置：`项目/.claude/agents/`（团队共享）或 `~/.claude/agents/`（个人全局）。

`.claude/agents/code-reviewer.md`：

```markdown
---
name: code-reviewer
description: 资深代码审查员。在一段代码编写完成后应主动使用它进行审查。
tools: Read, Grep, Glob, Bash
model: opus
---

你是一位以严格著称的资深代码审查员。

工作流程：
1. 运行 `git diff HEAD` 查看未提交的全部改动（若无改动则审查最近一次 commit）
2. 对每个改动文件，结合其上下文代码阅读理解
3. 重点检查（按优先级）：
   - 安全：注入、越权、敏感信息泄露
   - 正确性：边界条件、并发问题、错误处理是否吞异常
   - 可维护性：命名、重复代码、函数长度
4. 不要泛泛而谈，每个问题必须给出 文件:行号 和具体的修改建议

输出格式：按 [严重] [建议] [可选] 三级分组的清单。
```

`.claude/agents/test-runner.md`：

```markdown
---
name: test-runner
description: 测试修复专家。当测试失败需要排查修复时主动使用。
tools: Read, Edit, Bash, Grep, Glob
---

你负责让测试变绿，且**绝不通过削弱测试来达成**（禁止删测试、改断言迁就实现、加 skip）。

流程：跑测试 → 读失败信息 → 定位是实现的 bug 还是测试本身过时 →
修复 → 重跑 → 全绿后输出：改了什么、为什么、是否暴露了更深的问题。
```

### frontmatter 字段语义

| 字段 | 作用 | 要点 |
|---|---|---|
| `description` | **决定自动委派**：主 Claude 据此判断何时把任务交给它 | 写明触发场景；加"主动使用"字样会提高委派积极性 |
| `tools` | 能力白名单 | 审查员不给 Edit → **物理上**不可能改你的代码 |
| `model` | 该代理用什么模型 | 粗活配 haiku，精活配 opus |
| 正文 | 它的系统提示 | 它看不到主对话，正文要交代完整的角色和流程 |

## 8.5 并行与后台模式

```
> 同时做三件事：
  1. 用一个子代理审计 src/api/ 的输入校验
  2. 用另一个子代理检查全部 SQL 拼接是否有注入风险
  3. 用第三个子代理盘点过期依赖
  都完成后汇总成一份安全报告
```

三个分身并行各查各的，主对话最后只收三份报告。长任务还可以让子代理**后台运行**——主对话继续和你讨论别的，分身完成时会回来汇报。

## 8.6 使用准则

**适合**：

- 搜索、调研、盘点（废料多结论少）
- 独立的审查/测试任务（隔离还带来客观性——审查者不受"写代码时的思路"污染）
- 可并行的批量工作

**不适合**：

- 强依赖主对话上下文的任务（分身不知道你们刚才聊了什么）
- 需要和你来回交互确认的任务（分身不能中途问你）
- 三十秒的小事（派生开销不划算）

**写好委派任务的要诀**：当成给外包工程师写工单——背景、目标、范围、期望的输出格式，一次性写全。

```
❌ 「让子代理看看登录相关的代码」
✅ 「派一个 Explore 子代理：在本仓库找出登录流程的全部代码路径
    （从路由入口到 session 写入），包括第三方 OAuth 分支；
    输出按调用顺序排列的 文件:行号 列表，每项一句话说明职责」
```
