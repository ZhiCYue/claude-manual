# 第 0 章 · 总览与核心原理

这一章不教任何操作，只建立一个准确的「心智模型」。它是全书最重要的一章——后面每个高级功能，都是这一章原理的直接推论。

---

## 0.1 Claude Code 是什么

Claude Code 是 Anthropic 官方的**终端 AI 编程代理（Coding Agent）**。

和网页聊天版的本质区别：聊天版只能「说」，Claude Code 能「做」。它运行在你的终端里，拥有对你机器的（受控的）操作能力：

- 读写任意文件
- 执行 shell 命令（git、npm、pytest、docker……）
- 全库搜索代码
- 访问网络（搜索、抓取文档）
- 通过 MCP 扩展接入数据库、浏览器、第三方服务

你描述**目标**，它自己规划**步骤**、调用工具、检查结果、修正错误，循环往复直到完成。

## 0.2 系统的三层结构

理解 Claude Code，先分清三个角色：

```
┌─────────────────────────────────────────────┐
│  你（终端里的人类）                            │
│   · 下达目标、审批权限、验收结果                 │
├─────────────────────────────────────────────┤
│  Harness（Claude Code 程序本身）               │
│   · 管理对话、真正执行工具、检查权限、             │
│     运行 Hooks、加载 CLAUDE.md                │
├─────────────────────────────────────────────┤
│  模型（Fable / Opus / Sonnet…，跑在云端）       │
│   · 只做一件事：根据看到的全部文本，               │
│     输出下一段文本（其中可能包含工具调用请求）       │
└─────────────────────────────────────────────┘
```

三个关键推论：

1. **模型本身碰不到你的电脑。** 它输出的"工具调用"只是一段结构化文本，例如「我想调用 Bash，参数是 `npm test`」。真正执行的是 harness。
2. **所以权限控制是可靠的。** harness 在执行前检查规则、询问你——这是程序逻辑，不依赖模型"自觉"。第 4 章的权限和第 6 章的 Hooks 都建立在这一点上。
3. **模型每轮看到的是纯文本。** 文件内容、命令输出、报错信息，最终都变成文本进入它的"视野"。这个视野就是下一节的主角：上下文窗口。

## 0.3 Agent Loop：一次任务的完整解剖

假设你输入：

```
> 修复 tests/test_parser.py 里失败的那个测试
```

实际发生的循环（简化的真实流程）：

```
轮 1  模型：「我先看看测试为什么失败」
      → 工具调用：Bash("pytest tests/test_parser.py -x")
      ← harness 执行，返回输出：
        「FAILED test_parse_date - AssertionError: expected '2026-06-12', got None」

轮 2  模型：「测试期望解析出日期但得到 None，看一下被测函数」
      → 工具调用：Read("src/parser.py")
      ← harness 返回文件内容（带行号）

轮 3  模型：「第 42 行的正则漏了对 '-' 分隔符的支持，修复它」
      → 工具调用：Edit("src/parser.py", old="r'\d{4}/\d{2}/\d{2}'",
                                        new="r'\d{4}[-/]\d{2}[-/]\d{2}'")
      ← harness 应用编辑，返回成功

轮 4  模型：「重新跑测试验证」
      → 工具调用：Bash("pytest tests/test_parser.py")
      ← 「1 passed」

轮 5  模型：不再调用工具，输出总结文字 →「已修复：parser.py 的日期
      正则原本只支持 / 分隔……」 → 控制权交还给你
```

注意几点：

- **每一"轮"都是一次完整的模型调用**，模型每次都重新读取整段对话历史（含所有工具输出）。
- **循环的终止条件是模型自己决定的**——当它认为目标达成，就停止调用工具、输出总结。
- **验证（轮 4）不是装饰**。让它"改完跑一遍测试"是你能下达的最有价值的附加指令，因为它把循环的终止条件从"我觉得改好了"变成"测试真的通过了"。

## 0.4 上下文窗口：最重要的稀缺资源

模型每轮"看到"的全部内容，称为上下文窗口（context window），上限约 **20 万 token**（约等于 15 万英文单词或一本中篇小说）。它的构成：

```
┌──────────────────────────────┐
│ 系统提示词（Claude Code 内置指令）│  ~2 万 token，固定开销
│ 工具定义（每个工具的说明书）       │  随 MCP 增多而增长
│ CLAUDE.md（你的项目记忆）        │  你可控 ← 第 3 章
│ ───────────────────────────  │
│ 对话历史：                      │
│   你的消息 + 模型回复            │
│   + 每一次工具调用的完整输出      │  ← 增长最快的部分
│     （读过的文件、命令输出…）     │
└──────────────────────────────┘
```

三个工程后果：

1. **越长越贵越慢**：每轮都要重新处理全部历史。
2. **越长越"笨"**：无关的旧信息稀释注意力。一个塞满了上个任务残留物的窗口里，模型对当前任务的把握会明显下降。
3. **满了就要压缩**：接近上限时自动触发 compact（把历史压缩为摘要），压缩必然丢失细节。

会话中随时用 `/context` 命令可视化查看各部分占用——强烈建议入门阶段经常看，建立直觉。

> **由此推出全书一半的高级技巧**：
> - 一个任务一个会话，做完就 `/clear`（第 2 章）
> - 把长期信息放进文件（CLAUDE.md / 计划文档），不放对话里（第 3、10 章）
> - 搜索调研类脏活交给子代理，在"分身"的窗口里干，主窗口只收结论（第 8 章）

## 0.5 内置工具清单

模型可调用的主要内置工具。看懂它们，你就能看懂 Claude Code 工作时屏幕上滚动的每一行：

| 工具 | 作用 | 备注 |
|---|---|---|
| `Read` | 读文件（支持图片、PDF、Jupyter） | 带行号返回 |
| `Write` | 新建/整体覆盖文件 | |
| `Edit` | 精确字符串替换式修改 | 必须先 Read 过该文件 |
| `Bash` | 执行 shell 命令 | 权限检查最严格的工具 |
| `Glob` | 按文件名模式找文件 | 如 `src/**/*.ts` |
| `Grep` | 按正则全库搜索内容 | 基于 ripgrep，极快 |
| `WebSearch` / `WebFetch` | 搜网页 / 抓取网页内容 | 查最新文档 |
| `Agent` | 派生子代理 | 第 8 章 |
| `TodoWrite` | 维护任务清单 | 你在界面上看到的 checklist |
| `AskUserQuestion` | 向你提选择题 | 关键决策时弹出 |

## 0.6 安全模型小结

Claude Code 的安全是**纵深防御**，从外到内：

1. **权限规则**（第 4 章）：harness 层面的白名单/黑名单，`deny` 规则连模型被恶意提示词诱导都拦得住。
2. **Hooks**（第 6 章）：你的自定义程序逻辑，可在任何工具执行前否决。
3. **询问机制**：默认模式下每个敏感操作都要你按一次确认。
4. **检查点**：文件修改自动存档，`/rewind` 可回退（第 2 章）。
5. **模型对齐**：模型本身被训练为拒绝危险操作——但这是最后一道防线，不是唯一一道。

理解这个顺序后你就明白：**给 AI 更大自主权的正确方式，不是关掉所有确认，而是把边界规则写清楚**（精确的 allow/deny + Hooks），然后在边界内放手。
