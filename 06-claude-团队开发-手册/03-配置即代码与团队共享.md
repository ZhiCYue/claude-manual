# 第 3 章 · 配置即代码与团队共享

团队规范不该散落在每个人的脑子里和本地设置里。这一章讲怎么把配置当代码管理——哪些进 git、怎么共享、两层(工具中立 + Claude Code)各有哪些。

---

## 3.1 原则:配置即代码

第三条硬规则:**能进 git 的团队约定都进 git。** 好处:

- **可版本化**:配置变更有历史,能审查、能回滚。
- **可共享**:新人 clone 即继承全套环境,不用口口相传。
- **可审计**:谁改了什么规则,git 看得见。
- **单一来源**:不会出现"每个人本地设置不一样"导致的不一致。

反面:配置散落在个人本地 / 口头约定 / 某人脑中——新人上手慢、标准漂移、出了问题没法追溯。

## 3.2 两层配置全景

回顾第 0 章双层设计。团队项目里,两层各有一组进 git 的配置:

### ① 工具中立层(对所有人生效)

| 文件 | 作用 | 谁受益 |
|---|---|---|
| `.editorconfig` | 缩进、换行、编码统一 | 所有编辑器 |
| `.eslintrc` / `ruff.toml` / `.prettierrc` 等 | linter / formatter 规则 | 所有人 |
| `.pre-commit-config.yaml` / `.husky/` | git pre-commit hooks(本地强制) | 所有提交者 |
| `.github/workflows/` | CI 流水线 | 所有 PR |
| `CONTRIBUTING.md` / `AGENTS.md` | 团队约定(第 2 章) | 所有人、所有 AI 工具 |
| `Makefile` / `scripts/` | 标准化命令 | 所有人 |
| `.gitignore` | 忽略规则 | 所有人 |

### ② Claude Code 配置层(对 Claude Code 用户生效)

| 文件 | 作用 | 进 git? |
|---|---|---|
| `.claude/settings.json` | 团队共享的权限、hooks、插件 | ✅ 进 git |
| `.claude/settings.local.json` | 个人本地设置 | ❌ gitignore |
| `CLAUDE.md` / `.claude/CLAUDE.md` | 项目记忆 | ✅ 进 git |
| `.mcp.json` | 项目级 MCP 服务器 | ✅ 进 git |
| `.claude/commands/` | 团队自定义命令 | ✅ 进 git |
| `.claude/agents/` | 团队子代理 | ✅ 进 git |

> 注意 `.claude/settings.local.json` 要在 `.gitignore` 里——它是个人本地的(放个人的额外权限、本地 MCP 等),不该共享。Claude Code `/init` 时通常会帮你加上。

## 3.3 Claude Code 配置层:团队共享什么

### settings.json:团队权限基线

把团队都认可的安全操作放进 `.claude/settings.json`(进 git),让团队成员零打扰地用常用命令,同时锁死危险操作:

```json
{
  "permissions": {
    "allow": [
      "Read(**)",
      "Edit(src/**)",
      "Edit(tests/**)",
      "Bash(make test)",
      "Bash(make lint)",
      "Bash(git status)",
      "Bash(git diff *)"
    ],
    "ask": [
      "Bash(git push *)"
    ],
    "deny": [
      "Read(.env*)",
      "Read(secrets/**)",
      "Bash(rm -rf *)"
    ]
  }
}
```

团队成员 clone 后,这套权限自动生效——常用操作免询问,推送要确认,机密文件碰不到。

> **但记住双层原则**(第 0、6 章):`deny: Read(.env*)` 只拦得住用 Claude Code 的人。要真正保护密钥不进仓库,还得靠工具中立层(CI 的 secret 扫描 + git hook)——那才对所有人生效。Claude Code 的 deny 是便利和第一道防线,不是唯一防线。

### 共享自定义命令:固化团队流程

把团队的固定流程做成 `.claude/commands/`(进 git),全队共享。比如团队的"开 PR"流程:

```markdown
<!-- .claude/commands/pr.md -->
---
description: 按团队规范创建 PR
allowed-tools: Bash(git *), Bash(gh *)
---
当前改动：!`git status --short`

按团队规范创建 PR：
1. 确保在功能分支(不是 main),没有就先建 fix/feat 分支
2. 跑 `make test` 和 `make lint`,不过不准提交
3. commit message 遵循 @CONTRIBUTING.md 的规范
4. 用 `gh pr create`,PR 描述用中文,包含:改了什么、为什么、怎么测的
```

全队用同一个 `/pr`,PR 流程就标准化了(对 Claude Code 用户)。

### 共享 MCP 配置:统一工具接入

`.mcp.json`(进 git,第三册第 6 章)让团队共享同一套 MCP 服务器。新人 clone 即接入团队的内部工具(工单系统、数据库只读等),不用各自配置。

```json
{
  "mcpServers": {
    "tickets": {
      "command": "python",
      "args": ["scripts/mcp/tickets_server.py"],
      "env": { "TICKETS_API": "${TICKETS_API_URL}" }
    }
  }
}
```

注意用 `${变量}` 引用环境变量,别把密钥写进进 git 的文件。

## 3.4 一个完整的团队项目配置布局

把两层配置放在一起,一个组织良好的团队项目长这样:

```
project/
├── .editorconfig                    # 工具中立:编辑器统一
├── .pre-commit-config.yaml          # 工具中立:git hook(所有人强制)
├── ruff.toml                        # 工具中立:linter 规则
├── Makefile                         # 工具中立:标准命令
├── CONTRIBUTING.md                  # 工具中立:团队约定(实质内容)
├── .github/workflows/ci.yml         # 工具中立:CI(对所有 PR 强制)
├── .gitignore                       #   含 .claude/settings.local.json
├── CLAUDE.md                        # Claude 层:@import CONTRIBUTING + 补充
├── .mcp.json                        # Claude 层:团队 MCP
└── .claude/
    ├── settings.json                # Claude 层:团队权限/hooks(进 git)
    ├── settings.local.json          # Claude 层:个人本地(gitignore)
    ├── commands/                    # Claude 层:团队命令(进 git)
    └── agents/                      # Claude 层:团队子代理(进 git)
```

新人 `git clone` 一下,**两层配置全继承**:工具中立层让他的 CI/lint/约定到位(无论他用什么 AI 工具),Claude Code 配置层让用 Claude Code 的他额外获得权限基线、团队命令、MCP 接入。

## 3.5 配置变更的治理

配置是代码,就该走代码的流程:

- **配置改动走 PR**:改 `.claude/settings.json` 的权限、改 CI、改 CONTRIBUTING,都该有人审。别让某个人直接 push 改了全队的权限规则。
- **谨慎放宽权限**:往 `allow` 加东西、往 `deny` 删东西,影响全队安全,要慎重审。
- **企业级用 managed policy 强约束**:有些规则不能让项目或个人覆盖(强制登录组织、最低版本、禁某些 MCP),用企业 managed settings(第 7 章)。

## 3.6 配置即代码检查清单

- [ ] 工具中立配置进 git(.editorconfig / linter / pre-commit / CI / CONTRIBUTING / Makefile)
- [ ] Claude Code 团队配置进 git(.claude/settings.json / CLAUDE.md / .mcp.json / commands / agents)
- [ ] `.claude/settings.local.json` 在 .gitignore(个人本地不共享)
- [ ] settings.json 里没有硬编码的密钥(用 ${环境变量})
- [ ] 团队权限基线:常用操作 allow、推送 ask、机密 deny
- [ ] 配置改动走 PR 审查,不直接 push
- [ ] 关键安全规则同时在工具中立层强制(不只靠 Claude Code deny)
