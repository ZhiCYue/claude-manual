# 第 8 章 · AI 辅助前端完整工作流

把全册(两大支柱)和前六册串起来,给一个完整的、可照着做的 AI 辅助前端工作流——从需求到上线。

---

## 8.1 完整工作流全景

一个前端功能,从需求到合并,把两大支柱和前几册的能力都用上:

```
需求
  ↓
① 选对工具(第1章)         —— 项目已是 Vite/Next,开发服务器快、反馈循环短
  ↓
② 规格驱动(第6-7章)        —— 较大功能:/opsx:propose 生成 spec
  ↓                            你审 spec(最便宜的纠错点)←★支柱二:不跑偏
  ↓
③ 计划与实现(第一/三/五册)  —— /opsx:apply,Claude Code 照 spec 实现
  ↓                            复杂处用 Plan Mode;现代 React 模式(第3章)
  ↓
④ 地面真相验证(第4章)       —— 跑 Vitest + Playwright 实际操作浏览器  ←★支柱一:有据可验
  ↓                            "我看到它工作了",而非"我觉得对了"
  ↓
⑤ 工具中立质量门(第5章)     —— pre-commit + CI:lint/类型/测试/构建/包体  ←★对所有人强制
  ↓
⑥ 审查(第六册第5章)         —— PR 审查,AI 代码同等审,提交者负责
  ↓
⑦ 归档(第7章)              —— /opsx:archive,spec 与代码同步
  ↓
合并
```

注意这条流里两根支柱怎么交织:**支柱二(②⑦ 的 spec 驱动)框住"建什么",支柱一(①④⑤ 的好环境)保证"建得好、验得了、守得住"。**

## 8.2 贴图驱动:前端独有的迭代方式

前端有 UI,所以有个独特的 AI 辅助方式——**贴图驱动**(第一册第 10.6 节,这里落到前端):

```
> （Cmd+V 粘贴设计稿截图）
  按这张设计稿实现订单卡片组件。React 19 + Tailwind。
  实现后用 Playwright 打开 storybook,截图和设计稿对比,
  逐项修正差异,直到肉眼看不出区别。
```

形成"实现 → 截图 → 对比 → 修正"的闭环。这是前端把"地面真相"(第 4.4 节)用到极致的方式——AI 不只跑测试,还能**看**UI 对不对。

没有设计稿也能反着用:让 AI 截当前页面的图,你在图上指"这块间距太挤",它据此调。

## 8.3 不同规模任务的工作流

不是每个任务都走完整流程(呼应"从简单开始"):

| 任务规模 | 工作流 |
|---|---|
| **改文案/调样式**(小、明确) | 直接让 Claude Code 改 + Playwright 看一眼。不必 SDD。 |
| **加一个组件/小功能**(中) | Plan Mode 出方案 → 实现 → 测试 + 浏览器验证 → PR。可不走 OpenSpec。 |
| **加一个页面/模块**(大、可能模糊) | 完整流程:OpenSpec propose → 审 spec → apply → 验证 → 审查 → archive。 |
| **重构/迁移**(大、难回滚) | 必走 SDD:先写清"迁移后满足什么"的 spec,增量执行,每步验证。 |

判断:**任务越大、越模糊、越多人、越难回滚,越往完整流程靠;越小越明确,越简化。**

## 8.4 前端 AI 工作流的常见反模式

| ❌ 反模式 | 为什么糟 | ✅ 改法 |
|---|---|---|
| 大功能直接让 AI 写,不定 spec | 跑偏,回滚代价大 | 大功能先 OpenSpec propose 定 spec(第6-7章) |
| 只看代码不看 UI | 形似神不似溜过去 | Playwright 实际操作 + 截图验证(第4.4) |
| 团队标准只配 Claude Code hook | 用别的工具的人不受约束 | 放 CI/pre-commit 强制(第5章) |
| 接受 AI 的过时方案(CRA、手动 memo) | 代码一上来就过时 | CLAUDE.md 写明现代约定(第3、4章) |
| 服务端数据塞全局 store | 状态管理一锅粥 | 三分法:TanStack Query 管服务端(第3.6) |
| spec 写了不审就 apply | 失去 SDD 的核心价值 | propose 后认真审 spec(第7.3) |
| 大 PR 走过场审 | AI 代码风险溜过 | 控制 PR 大小,认真审(第六册第5章) |

## 8.5 把七册串起来:前端 AI 开发者的能力地图

到这里,你做 AI 辅助前端开发,可以调用整个系列:

```
做前端功能时,你在用……
│
├── 第一册  Claude Code 的命令、Plan Mode、视觉迭代、子代理
├── 第三册  Playwright MCP(浏览器地面真相)、自己写前端用的 MCP
├── 第四册  给前端任务的好 prompt、防 AI 给过时/过度工程的代码
├── 第五册  探索→计划→执行→验证、上下文管理(大型重构)
├── 第六册  双层设计、团队协作、混合工具、AI 代码审查
└── 第七册  前端选型、React 19 模式、AI 友好的前端项目、规格驱动/OpenSpec
            （本册:把以上落到前端 + 加规格驱动这根支柱）
```

第七册是前端领域的"集大成"——它把通用能力落到前端,并补上规格驱动这个让 AI 在前端不跑偏的关键方法。

## 8.6 落地建议

别想一步到位。分步采纳(呼应第六册第 8 章):

1. **先把工程基础打好**(第1-5章):选对工具、项目 AI 友好、质量门到位。这是支柱一,也是后面一切的地基。
2. **在一个大功能上试一次 SDD**(第6-7章):走一遍 OpenSpec 的 propose→apply→archive,体会"先审 spec"的确定性。
3. **形成你的工作流直觉**:哪些任务值得 SDD、哪些直接做;什么时候贴图驱动;什么时候开 Plan Mode。
4. **团队推广**(第六册):把质量门、spec 工作流、约定推广到团队,在工具中立层强制。

