# 第 0 章 · 心智转变与双层设计原则

这一章建立两个东西:一个心智转变(从"AI 帮我写代码"到"为团队设计 AI 友好的工程环境"),和一条贯穿全册的原则(双层设计)。理解它们,后面每一章都是顺理成章的推论。

---

## 0.1 心智转变:从"用 AI"到"为 AI 设计环境"

个人用 Claude Code,你关心的是"它能帮我做什么"。团队落地,你要关心的是一个更高层的问题:

> **怎么让整个团队的工程环境,既对人友好,也对 AI 友好,且无论用什么 AI 工具都成立?**

这是个**环境设计**问题,不是"工具使用"问题。第一册结尾说过一句话,在这里是主题:

> 衡量进阶的标志:你开始**为 AI 设计环境**(记忆、权限、钩子、约定),而不只是和它聊天。

团队版的这句话是:**你为整个团队设计 AI 友好的工程环境**——清晰的项目结构、成文的约定、版本化的配置、自动化的护栏。这些既帮 AI 干得好,也帮人(尤其新人)上手快。好消息是:**对 AI 友好的环境,几乎总是对人也更友好**——清晰的结构、好的文档、可靠的 CI,本来就是好工程的标志。

## 0.2 个人 vs 团队:冒出来的新问题

一个人用 Claude Code,很多事靠自觉和习惯就行。团队场景,这些"靠自觉"的事必须变成"成文且可执行"的:

| 维度 | 个人 | 团队 |
|---|---|---|
| 项目约定 | 记在脑子里 | 必须成文(否则新人/AI 都不知道) |
| 配置 | 本地随手设 | 该共享的进 git,该统一的统一 |
| 标准执行 | 自己记得跑 lint | 必须自动强制(CI),不靠自觉 |
| AI 代码质量 | 自己把关 | 要有审查纪律和责任归属 |
| 工具 | 你用啥是你的事 | 队友可能用不同工具,规范要兼容 |
| 成本/治理 | 自己的账单 | 团队预算、模型分配、合规 |

这一册就是把右边这一列系统化。

## 0.3 核心原则:双层设计

整本书最重要的一条原则。把团队的规范和配置,分成**两层**:

```
┌─────────────────────────────────────────────────────┐
│  ① 工具中立层(Tool-Neutral Layer)                     │
│     · 项目结构与目录约定                                │
│     · 文档:README / CONTRIBUTING / 架构文档 / 约定文档   │
│     · CI 流水线(测试、lint、构建、安全扫描)             │
│     · git pre-commit hooks(本地强制,所有人触发)         │
│     · linter / formatter 配置(.editorconfig 等)        │
│     · PR 规范、commit 规范、分支保护                     │
│     · AI 生成代码的审查纪律                             │
│  → 对所有人生效,无论用 Claude Code / Cursor / Copilot   │
├─────────────────────────────────────────────────────┤
│  ② Claude Code 配置层                                  │
│     · .claude/settings.json(权限、hooks、插件)         │
│     · CLAUDE.md(项目记忆)                              │
│     · .mcp.json(MCP 服务器)                            │
│     · .claude/commands、.claude/agents                 │
│  → 只对用 Claude Code 的成员生效,给他们额外的自动化       │
└─────────────────────────────────────────────────────┘
```

### 为什么必须分这两层

因为**真实团队几乎都是混合工具的**:有人用 Claude Code、有人用 Cursor、有人用 Copilot、有人裸写。如果你把团队标准只放在 Claude Code 配置层(比如用 Claude Code 的 Hook 强制"提交前跑 lint"),那这条标准**只对用 Claude Code 的人生效**——用 Cursor 的同事根本不触发那个 Hook,照样能提交不合规的代码。

所以这条原则的关键推论(也是本册回答"队友用别的工具怎么办"的核心):

> **想让一条规则对团队所有人生效,就在工具中立层强制它。**
>
> - 要"提交前必过 lint" → 放 **git pre-commit hook**(husky / pre-commit 框架)+ **CI required check**,不是放 Claude Code 的 Hook。
> - 要"PR 必须有测试" → 放 **CI + 分支保护**,不是放 CLAUDE.md 里写一句"请写测试"。
> - 要"不准提交密钥" → 放 **CI 的 secret 扫描 + git hook**,不是只靠 Claude Code 的权限 deny。

**Claude Code 配置层是"锦上添花":** 它让用 Claude Code 的成员获得额外的自动化和便利(比如改完自动 format、危险操作被拦),但它**不是团队标准的执行者**。团队标准的执行者永远是工具中立层。

### 一个对照例子

团队要求:"所有代码提交前必须通过 ESLint。"

| ❌ 只放 Claude Code 层 | ✅ 工具中立层强制 + Claude Code 层增强 |
|---|---|
| 在 `.claude/settings.json` 配一个 PostToolUse Hook 自动 lint | **工具中立层**:配 git pre-commit hook 跑 ESLint + CI 里 ESLint 作为 required check |
| 只有用 Claude Code 的人生效 | 对所有人生效(用 Cursor、裸写都逃不掉 CI) |
| 用 Cursor 的同事照样提交不合规代码 | **Claude Code 层(额外)**:Claude Code 用户再配个 Hook 改完即时 format,体验更顺 |

结果:标准对所有人硬性生效(工具中立层),Claude Code 用户还多一份顺手(配置层)。两层各司其职。

## 0.4 这条原则怎么贯穿全册

带着双层设计,后面各章的位置就清晰了:

```
第 1 章  个人工作流          —— 主要是你自己怎么用(Claude Code 层的个人实践)
第 2 章  让项目 AI 友好       —— 项目结构和文档(工具中立层)+ CLAUDE.md(配置层)
第 3 章  配置即代码           —— 两层各有哪些配置进 git,怎么分
第 4 章  混合工具团队         —— ★这条原则的集中应用:怎么让约定对所有人有效
第 5 章  协作规范             —— PR/审查纪律(工具中立层,对所有人)
第 6 章  质量安全护栏         —— 在 CI 强制(工具中立层)+ Claude Code 权限(配置层)
第 7 章  规模化治理           —— 企业 managed policy(Claude Code 层的强约束)+ 通用治理
第 8 章  落地路线图           —— 怎么分阶段把这两层都建起来
```

## 0.5 三条硬规则(全册基石)

1. **要对所有人生效的规则,放工具中立层强制。** CI / git hooks / 分支保护 / linter 是团队标准的执行者;Claude Code 配置是 Claude Code 用户的便利层。
2. **AI 生成的代码,人负责。** 无论来自哪个模型/工具,提交者负全责,必经人审、有测试。与工具无关。
3. **配置即代码。** 能进 git 的团队约定都进 git——可版本化、可审查、可复制,而非散落在个人本地或脑中。

这三条会在后面反复出现。现在记住核心:**两层分清,标准放对层。**
