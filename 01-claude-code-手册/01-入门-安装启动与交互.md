# 第 1 章 · 入门：安装、启动与交互

## 1.1 安装

```bash
# 方式一：官方安装脚本（macOS / Linux，推荐）
curl -fsSL https://claude.ai/install.sh | bash

# 方式二：npm（需 Node.js 18+）
npm install -g @anthropic-ai/claude-code

# 方式三：Homebrew（macOS）
brew install --cask claude-code

# 验证安装
claude --version        # 例：2.1.174 (Claude Code)
claude doctor           # 自检：安装方式、自动更新、网络连通性
```

Windows 用户：原生支持 PowerShell，也可在 WSL 中安装使用。

## 1.2 登录与账号

首次运行 `claude` 会引导登录，两种方式：

| 方式 | 适合谁 | 计费 |
|---|---|---|
| **Claude.ai 订阅**（Pro/Max） | 个人开发者 | 包月，按额度限速 |
| **API Key**（Anthropic Console） | 团队、CI、按量付费 | 按 token 计费 |

```bash
# 会话内切换账号
/login
/logout

# CI / 脚本环境直接用环境变量
export ANTHROPIC_API_KEY=sk-ant-...
```

企业环境还支持 AWS Bedrock / Google Vertex AI / Azure Foundry 作为模型后端（设置 `CLAUDE_CODE_USE_BEDROCK=1` 等，见附录环境变量表）。

## 1.3 启动方式与常用参数

```bash
cd 你的项目目录       # 重要：在哪个目录启动，Claude 就以哪个目录为工作区
claude               # 交互式会话（最常用）
```

常用启动参数：

```bash
claude "解释这个项目的结构"      # 带初始指令启动
claude -c                      # continue：继续本目录最近一次会话
claude --resume                # 列出历史会话，选择恢复
claude --model sonnet          # 指定模型启动
claude --add-dir ../shared-lib # 额外授权访问工作区之外的目录
claude --effort high           # 设定思考深度（low/medium/high/xhigh/max）
claude -p "任务"               # 无头模式：执行完就退出（第 9 章专讲）
```

> **原理：会话存在哪？** 每次对话的完整记录（含所有工具调用）保存在
> `~/.claude/projects/<项目路径编码>/` 下的 JSONL 文件里。
> `-c` / `--resume` 做的事就是把这份记录重新装回上下文窗口，
> 所以恢复的会话"记得"之前的一切——代价是恢复时窗口立刻就是半满的。

## 1.4 输入技巧：六个核心操作

### ① `@` —— 精确引用文件

```
> 重构 @src/utils/date.py，把所有函数改成纯函数，参考 @docs/style-guide.md 的规范
```

输入 `@` 后有路径自动补全。被 `@` 的文件内容会直接进入上下文，比"你去看一下 date.py"更快更准（省掉它自己搜索定位的轮次）。也可以 `@` 一个目录，它会拿到目录结构。

### ② `!` —— 直接执行 shell 命令

```
> ! git log --oneline -10
```

`!` 前缀的命令立即在本会话执行，**输出同时展示给你和 Claude**。典型用法：把报错现场直接"喂"给它：

```
> ! npm run build
（输出了一堆报错）
> 修复上面的构建错误
```

### ③ `#` —— 快速记忆

```
> # 这个项目的测试必须用 make test 跑，环境变量在 Makefile 里
```

`#` 开头的内容会被存入 CLAUDE.md（会让你选存哪一级），从此每个新会话都自动知道这件事。详见第 3 章。

### ④ `Esc` —— 随时打断

Claude 正在长篇大论或方向跑偏时，按 `Esc` 立即打断，然后补充指令。**打断不丢上下文**，它知道自己刚才做到哪了。这是新手最该克服的心理障碍：不要干等它做完错的事。

### ⑤ `Esc Esc`（连按两下）—— 时光回退

弹出历史消息列表，选一条跳回去重新编辑。配合 `/rewind` 还能把**代码改动**一起回退到那个时间点（第 2 章详解）。

### ⑥ 粘贴图片

直接 `Cmd+V` 粘贴截图（或把图片文件拖进终端）。模型能看图——贴一张 UI 设计稿让它实现、贴一张报错截图让它排查，都是高频用法。

## 1.5 其他实用输入设置

## 1.6 第一次实战

**任务一（只读，体验搜索能力）**：

```
> 这个项目的入口在哪？从入口开始，给我讲一遍一个 HTTP 请求被处理的完整链路
```

观察它如何组合 `Glob`（找文件）→ `Read`（读关键文件）→ `Grep`（追调用链）。

**任务二（动手 + 自我验证）**：

```
> 给 @src/utils.py 的 slugify 函数补充处理中文和 emoji 的逻辑，
  写对应的单元测试，跑通过为止
```

观察 Agent Loop 完整运转：读代码 → 改代码 → 写测试 → 跑测试 → （若失败）修复 → 再跑 → 汇报。

**任务三（体验打断与纠偏）**：

故意下达一个模糊指令，等它开始往不想要的方向走时按 `Esc`，补充约束。感受"协作"而非"许愿"。

## 1.7 提问的艺术（Prompting 实践守则）

这是入门阶段投入产出比最高的技能。对比示例：

| ❌ 低效写法 | ✅ 高效写法 | 原因 |
|---|---|---|
| 「优化一下这个项目」 | 「`npm run build` 目前要 90 秒，目标降到 30 秒以内，先分析瓶颈再动手」 | 目标可度量 |
| 「修一下登录的 bug」 | 「用户报告：手机号带空格时登录失败。复现路径是 @src/auth/login.ts，修复并补回归测试」 | 给了现场和验收 |
| 「写个爬虫」 | 「写一个 Python 脚本抓取 X 页面的标题和价格，存 CSV；只用标准库 + requests + bs4；加重试和限速」 | 约束了技术选型 |
| 「把代码改好看点」 | 「按 @CONTRIBUTING.md 的规范重构 src/api/，只动格式和命名，不改任何逻辑，改完跑 lint 和测试」 | 划定了边界 |

四条通用原则：

1. **给目标和验收标准，不给微观步骤**——把"怎么做"留给它，把"做到什么程度算完成"说死。
2. **给上下文现场**：报错原文、复现步骤、相关文件 `@` 出来。
3. **划定边界**：哪些文件能动、哪些不能动、用什么不用什么。
4. **要求自我验证**：「改完跑 X 确认」让循环的终止条件客观化（见 0.3 节）。
