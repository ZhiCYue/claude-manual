# 第 8 章 · 架构蓝图实战

前面各章是零件:harness、五模式、自主 agent、ACI、上下文工程、多智能体、可靠性。这一章把它们组装起来——给几个常见的架构蓝图,和一套贯穿全册的决策框架。

---

## 8.1 决策框架:从需求到架构

面对一个新需求,这样一路问下来,架构就定了:

```
① 这个问题措辞能解决吗?(第四册 / 本册 1.4)
   是 → 用 prompt engineering,不要 agent
   否 ↓

② 步骤能提前确定吗?(本册 1.1)
   能 → 工作流(选第 2 章的模式)
   不能 ↓

③ 单个 agent 自主能搞定吗?(本册 3.4)
   能 → 单 agent + 好工具 + 好上下文工程 + 护栏
   不能 ↓

④ 任务能切成独立并行子任务,且价值证明 15 倍 token 吗?(本册 6.3)
   能 → 多 agent(编排者-工作者)
   不能 → 退回单 agent,或拆解需求重新评估
```

**这个框架的灵魂是第 1.4 节:从简单开始,复杂度要挣来。** 每往下一层,成本和不可预测性都上升。停在能满足需求的最高一层。

## 8.2 蓝图一:客服自动化(工作流为主)

需求:自动处理客服查询。

```
用户查询
   ↓
[路由] 分类:技术 / 退款 / 一般         ← 模式:路由(2.2)
   ↓
├─ 技术 → [链式] 检索知识库 → 生成回答 → 校验   ← 模式:提示链(2.1)
├─ 退款 → [单 agent] 查订单 → 判断政策 → 需审批则转人工  ← 危险操作护栏(3.5)
└─ 一般 → 单次 LLM 调用                ← 最简层(1.4)
```

要点:大部分是确定性工作流(可预测、一致),只在退款这种需要查多个系统、判断的分支用 agent。危险操作(实际退款)走人工审批护栏。**整体简单,只在必要处复杂。**

## 8.3 蓝图二:编码 agent(单 agent 为主)

需求:根据任务描述自动改代码、修 bug。

```
任务描述
   ↓
[单个自主 agent]                       ← 单 agent(编码不适合多 agent,6.3)
   工具:read / edit / bash / grep      ← ACI 设计(第 4 章)
   环境反馈:跑测试、跑 lint            ← 地面真相(3.3)
   上下文:git 跟踪状态、长任务记笔记    ← 上下文工程(第 5 章)
   护栏:edit 受保护文件被 Hook 拦、迭代上限  ← 护栏(3.5)
   ↓
[自我验证] 跑全量测试确认               ← 自我验证(3.3)
```

要点:编码是深度耦合的,**单 agent 优于多 agent**。成败关键在 ACI(工具设计)和地面真相(让它能跑测试自我验证)。这就是 Claude Code 的架构本质。

## 8.4 蓝图三:深度研究(多 agent)

需求:对一个复杂主题做宽度优先的深度调研,产出报告。

```
研究查询
   ↓
[主 agent] 分析查询、制定策略           ← 编排者(6.1)
   ↓ 清晰委派(目标/格式/边界)         ← 委派原则(6.4①)
   ├─→ [子代理1] 调研方面A(独立上下文)  ┐
   ├─→ [子代理2] 调研方面B(独立上下文)  ┼ 并行          ← 上下文隔离(5.4)
   └─→ [子代理N] 调研方面C(独立上下文)  ┘
   ↓ 每个子代理返回 1000-2000 token 摘要
[主 agent] 综合所有摘要 → 报告
   ↓
[LLM 裁判] 对照 rubric 评估质量          ← 评估(7.3)
```

要点:研究是宽度优先、可并行、子任务独立的,**多 agent 在这里值那 15 倍 token**(质量 +90%、时间 -90%)。关键在清晰委派(防子代理重复劳动)和上下文隔离(脏活在子代理窗口)。这就是 Anthropic 多智能体研究系统的架构。

## 8.5 蓝图四:内容流水线(评估者-优化者)

需求:生成高质量的长文(报告、文案),要打磨到位。

```
主题
   ↓
[生成] 写初稿                          ← 模式:评估者-优化者(2.5)
   ↓ ←──────── 反馈 ────────┐
[评估] 对照标准打分            │
   ↓                        │
合格? ─否→ [带反馈重新生成] ──┘
   ↓是
终稿
```

要点:有清晰的质量标准时,评估者-优化者循环能持续打磨。前提是标准清晰(评估者得能判断好坏)。这是第四册第 7 章"自我修正"的架构化。

## 8.6 组合:真实系统是嵌套的

真实的复杂系统往往是这些蓝图的**嵌套组合**:

```
[路由] 分类请求
   ├─ 简单请求 → 单次调用
   ├─ 标准请求 → [工作流]
   └─ 复杂请求 → [多 agent]
                   └─ 某个子代理内部是 [评估者-优化者循环]
                   └─ 另一个子代理内部是 [提示链]
```

别把这些模式当成互斥的选择,当成**可嵌套组合的积木**。一个 agent 的工具内部可以是个工作流;一个工作流的某步可以是个 agent;编排者派的 worker 可以各有各的内部架构。

**组合的原则还是 1.4**:每一层都用能满足那层需求的最简方案。整体复杂的系统,由许多局部简单的部分组成。

## 8.7 把五册串起来:一个完整 agent 产品的全景

到这里,五册的知识可以串成一个完整 agent 产品的全景图:

```
你的 AI Agent 产品
│
├── 模型调用              ← 第二册:Messages API、流式、结构化输出
├── prompt                ← 第四册:清晰、示例、思考、agentic 调优、模型调优
├── 工具                  ← 第三册:MCP 服务器 / 第二册:工具定义 + 第四册 ACI
├── harness/编排          ← 本册:工作流模式 / 自主 agent / 多智能体
├── 上下文管理            ← 本册第 5 章:压缩、记忆、子代理、即时检索
├── 可靠性                ← 本册第 7 章:评估、追踪、检查点、护栏
└── (交付层)             ← 第一册:Claude Code 作为你自己开发这个产品的工具
```

- **第一册**给你开发工具(Claude Code 是你造产品时用的 harness)。
- **第二册**给你模型能力的接口(API / Agent SDK)。
- **第三册**给你扩展能力的方式(MCP)。
- **第四册**给你和模型沟通的功夫(prompt engineering)。
- **第五册(本册)**给你把这一切组装成系统的架构方法论。

## 8.8 最后的原则

整个 agent 架构的功夫,可以浓缩成贯穿全册的这几句:

1. **从简单开始,复杂度要挣来。** 单次调用 → 工作流 → 单 agent → 多 agent,只在下一层确实更好时才往上爬。
2. **上下文是有限资源。** 无情地最小化 token,主动策展高信号内容。
3. **工具是 agent 的瓶颈。** ACI 值得和 prompt 同等的工程投入。
4. **给 agent 地面真相。** 让它能自我验证,把"我觉得好了"变成"环境证明好了"。
5. **自主在护栏内发生。** 边界写清楚,然后放手。
6. **非确定的系统要可观测、可恢复。** 看得见、查得出、崩了能恢复。

构建正确的系统,而非最复杂的系统。
