# 第 0 章 · Harness 是什么

你在网上看到的"harness"概念,在这一章讲清楚。它是整个 agent 系统的承重结构——理解了它,后面的工作流、自主 agent、多智能体都是在它之上的设计。

---

## 0.1 一句话定义

> **Harness 是运行 agent loop 的编排框架。** 模型只会"说话"(输出文本,包括工具调用请求),harness 是真正把这些话变成行动、并把行动结果喂回模型的那层程序。

第一册第 0 章讲过这个三层结构,这里把它正式命名:

```
┌─────────────────────────────────────────────┐
│  人类 —— 下达目标、审批、验收                    │
├─────────────────────────────────────────────┤
│  HARNESS —— 编排框架(本册的主角)               │
│   · 跑 agent 循环                              │
│   · 真正执行工具调用                            │
│   · 管理上下文(压缩、记忆)                      │
│   · 检查权限、运行护栏                          │
│   · 错误恢复、检查点                            │
├─────────────────────────────────────────────┤
│  模型 —— 只输出下一段文本(含工具调用请求)        │
└─────────────────────────────────────────────┘
```

**Claude Code 是一个 harness**(第一册)。你用第二册第 4 章手写的那个 `while` 循环,也是一个最小的 harness。Agent SDK(第二册第 8 章)给你一个现成的 harness。Managed Agents(第二册第 9 章)是 Anthropic 托管的 harness。**本册教你理解和设计 harness 本身。**

## 0.2 Harness 的职责

一个完整的 harness 要做这些事。理解它们,就理解了 agent 系统的全部工程面:

### ① 跑 agent 循环

最核心的职责。回顾第二册第 4 章手写的循环:

```
模型思考下一步 → 发出工具调用请求 → harness 执行 → 把结果喂回 → 模型继续 → …直到完成
```

harness 负责这个循环的每一轮:发请求、解析响应、判断是否该继续(`stop_reason`)、执行工具、组装下一轮的消息。

### ② 真正执行工具调用

模型说"我想调 `read_file('config.yaml')`",harness 才是那个真去读文件的程序。它决定:

- 工具实际怎么执行(本地函数?HTTP 调用?MCP 服务器?)
- 执行前要不要检查权限、弹审批
- 执行结果怎么格式化喂回模型

### ③ 管理上下文

这是 harness 最被低估、也最决定成败的职责。对话越长,上下文越满,越贵越"笨"(第 5 章的 context rot)。harness 负责:

- 何时压缩历史(compaction)
- 何时清除陈旧的工具结果(context editing)
- 怎么让 agent 跨上下文窗口持久化状态(记忆、文件)

第 5 章整章讲这个。

### ④ 检查权限、运行护栏

harness 是确定性安全的所在地。回顾第一册第 4 章:权限规则、Hooks 都是 harness 层的程序逻辑——**模型被诱导也越不过去**。harness 决定:

- 哪些工具调用直接放行、哪些要确认、哪些禁止
- 危险操作(删除、推送、对外发布)的护栏

### ⑤ 错误恢复与检查点

agent 跑很多步,中间某步失败很常见。harness 负责优雅降级:

- 工具失败了,把错误喂回模型让它换方法(而非整个崩溃)
- 定期检查点,出错能从断点恢复而非从头再来

> 第 7 章会讲到:**agentic 系统里错误会复合**——传统软件里的小故障,在 agent 里可能让它跑偏到完全不同的轨迹。所以错误恢复是 harness 的硬功能,不是可选项。

## 0.3 好 harness 的三个标志

Anthropic 总结的三条核心原则,既是建 agent 的原则,也是评判一个 harness 好坏的标准:

### 简单(Simplicity)

**保持 agent 设计直接。** 最好的 harness 不是功能最多的,是恰好够用的。自主 agent 的实现本质上很简单——"LLM 在循环里基于环境反馈使用工具",就这一句话。别把 harness 做成一个庞然大物。

### 透明(Transparency)

**明确地展示规划步骤。** 好 harness 让你(和用户)看得见 agent 在想什么、做什么、为什么。黑盒 agent 没法调试、没法信任。第一册里 Claude Code 把 TODO、工具调用都显示出来,就是透明性的体现。

### 精心设计的 ACI

**通过充分的文档和测试,精心打造工具与环境的接口(Agent-Computer Interface)。** Anthropic 做 SWE-bench agent 时,花在工具设计上的时间比花在整体 prompt 上还多。第 4 章整章讲 ACI。

## 0.4 你需要自己写 harness 吗?

不一定。选择取决于你要多少控制:

| 选项 | harness 是谁 | 何时选 |
|---|---|---|
| 用 Claude Code | Anthropic(完整的 CLI harness) | 交互式开发,用现成的 |
| 用 Agent SDK | Anthropic 的库,跑在你的进程 | 嵌进产品,要 Claude Code 的能力 |
| 用 Managed Agents | Anthropic 托管(云端容器) | 要托管、文件挂载、跨会话状态 |
| 手写循环 | 你 | 要精细控制:自定义护栏、审批、日志、特殊编排 |

**多数人不该从零写 harness。** 但即使你用现成的,理解 harness 的职责也至关重要——它决定你怎么配置权限、怎么管上下文、怎么设计工具、怎么处理错误。本册讲的是这些**通用的架构原理**,无论你的 harness 是自己写的还是用现成的,都适用。
