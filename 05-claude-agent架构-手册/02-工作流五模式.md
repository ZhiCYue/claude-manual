# 第 2 章 · 工作流五模式

Anthropic 总结了五种经典的工作流模式。它们是构建可预测、可靠 agentic 系统的基础积木。这一章每个模式都给图、给适用场景、给代码骨架。

---

## 2.1 模式一:提示链(Prompt Chaining)

把任务拆成顺序的步骤,每次 LLM 调用处理上一步的输出。步骤间可以加程序化的"门"(gate)校验中间结果。

```
输入 → [LLM 步骤1] → 输出1 → (门:校验?) → [LLM 步骤2] → 输出2 → [LLM 步骤3] → 最终
```

**何时用**:任务能拆成固定的子任务,且用延迟换准确性是值得的。

**例子**:
- 先生成营销文案,再翻译成另一种语言。
- 先列文档大纲,校验大纲,再写完整初稿。

**代码骨架**(呼应第四册第 7 章):

```python
def chain(client, user_input):
    # 步骤 1:生成大纲
    outline = call(client, f"为这个主题列一个文档大纲:{user_input}")

    # 门:校验大纲(程序化检查,不合格就停)
    if len(outline.split("\n")) < 3:
        raise ValueError("大纲太简单,需要重新生成")

    # 步骤 2:基于大纲写初稿
    draft = call(client, f"按这个大纲写完整文档:\n{outline}")
    return draft
```

那个"门"是提示链的精髓——你可以在步骤间插入程序化校验,不合格就重试或中止,保证质量。

## 2.2 模式二:路由(Routing)

先分类输入,再把它导向专门的下游处理。

```
              ┌→ [处理 A]
输入 → [分类] ─┼→ [处理 B]
              └→ [处理 C]
```

**何时用**:复杂任务有**不同的类别**,分开处理更好。

**例子**:
- 客服查询分类(一般问题/退款/技术支持),各走专门的工作流。
- 按难度把问题路由到不同大小的模型(简单的给 Haiku 省钱,难的给 Opus)。

**代码骨架**:

```python
def route(client, query):
    # 分类(可以用结构化输出保证返回合法类别,第二册第 2 章)
    category = call(client, f"""把这个查询分类为:技术/退款/一般。只输出类别词。
查询:{query}""").strip()

    # 路由到专门处理
    if category == "技术":
        return handle_technical(client, query)   # 用技术专用 prompt + 工具
    elif category == "退款":
        return handle_refund(client, query)       # 用退款专用流程(可能要审批)
    else:
        return handle_general(client, query)
```

路由的价值:每个分支可以用**专门优化**的 prompt、工具、甚至模型,比一个万能 prompt 处理所有情况质量更稳。

## 2.3 模式三:并行化(Parallelization)

同时跑多个 LLM 调用,再聚合输出。两个变体:

### 切分(Sectioning):把任务拆成独立的并行子任务

```
        ┌→ [子任务 A] ┐
输入 → ─┼→ [子任务 B] ┼→ [聚合] → 输出
        └→ [子任务 C] ┘
```

**例子**:同时做内容审核和生成回复;并行评估模型的不同性能维度。

### 投票(Voting):同一任务跑多次,取多样的输出

```
        ┌→ [审查 #1] ┐
代码 → ─┼→ [审查 #2] ┼→ [汇总/多数表决] → 结论
        └→ [审查 #3] ┘
```

**例子**:多个审查者并行查代码漏洞(任一发现都重要);多个审核员评估内容是否合规。

**何时用**:任务受益于提速,或需要多个视角来提高置信度。

**代码骨架**(并行用异步或线程):

```python
import asyncio

async def parallelize_voting(client, code):
    # 同一审查任务跑 3 次(投票)
    prompts = [f"审查这段代码的安全漏洞:\n{code}"] * 3
    results = await asyncio.gather(*[acall(client, p) for p in prompts])

    # 聚合:任一发现严重问题就标红
    if any("严重" in r for r in results):
        return "发现严重问题:\n" + "\n---\n".join(results)
    return "未发现严重问题"
```

> 注意:在 agent 场景里,这种"并行子任务再聚合"最新模型可以用**子代理**自动完成(第 6 章),不必你手动编排并行调用。工作流模式是显式编排,multi-agent 是模型自主编排——后面会对比。

## 2.4 模式四:编排者-工作者(Orchestrator-Workers)

一个中央 LLM(编排者)动态地分解任务、委派给工作者 LLM、再综合结果。**和并行化的关键区别:子任务是不可预测的**——编排者根据输入当场决定派几个 worker、各干什么。

```
                  ┌→ [工作者 1] ┐
输入 → [编排者] ──┼→ [工作者 2] ┼→ [编排者综合] → 输出
       (动态分解)  └→ [工作者 N] ┘
                  (数量和任务运行时决定)
```

**何时用**:复杂任务,所需步骤**无法提前确定**。

**例子**:
- 多文件代码改动——改几个文件、怎么改,取决于输入。
- 跨多个来源的信息收集——研究需求是可变的。

**这是工作流和 agent 的交界**:编排者本身有 agent 的特性(动态决策),但整体结构(编排→委派→综合)是固定的。第 6 章的多智能体协作,本质上就是这个模式的规模化。

**代码骨架**(概念示意):

```python
def orchestrator_workers(client, task):
    # 编排者:动态分解任务(返回子任务列表,数量不定)
    subtasks = call_structured(client,
        f"把这个任务分解成独立的子任务:{task}", schema=SubtaskList)

    # 派工作者(可并行)执行每个子任务
    results = [call(client, f"完成这个子任务:{st}") for st in subtasks]

    # 编排者综合
    return call(client, f"综合这些结果给出最终答案:\n{results}")
```

## 2.5 模式五:评估者-优化者(Evaluator-Optimizer)

一个 LLM 生成,另一个 LLM 在循环里提供迭代反馈。

```
            ┌──────── 反馈 ────────┐
            ▼                      │
输入 → [生成者] → 草稿 → [评估者] → 合格? ─否→(带反馈回到生成者)
                                    └是→ 输出
```

**何时用**:存在**清晰的评估标准**,且迭代精炼确实能改善输出。

**例子**:
- 文学翻译——评估者给出细腻的质量反馈,生成者据此改。
- 复杂研究——需要多轮"搜索-分析"迭代。

这就是第四册第 7 章的"自我修正"模式的工作流化版本——把"生成"和"评估"分成两个角色,循环到合格。关键前提:**评估标准要清晰**(评估者得能判断"好没好""哪里不好"),否则循环没有收敛方向。

**代码骨架**:

```python
def evaluator_optimizer(client, task, max_rounds=3):
    draft = call(client, f"完成这个任务:{task}")
    for _ in range(max_rounds):
        evaluation = call(client, f"""对照标准评估这份产出:
        标准:{criteria}
        产出:{draft}
        如果合格,只回复"PASS"。否则指出具体问题。""")
        if evaluation.strip() == "PASS":
            break
        # 带着反馈重新生成
        draft = call(client, f"""根据反馈改进:
        原稿:{draft}
        反馈:{evaluation}""")
    return draft
```

## 2.6 五模式速查与选型

| 模式 | 结构 | 何时用 | 一句话 |
|---|---|---|---|
| 提示链 | 顺序步骤 + 门 | 步骤固定,顺序明确 | 流水线 |
| 路由 | 分类 → 分支 | 有不同类别,分开处理更好 | 分诊台 |
| 并行化 | 多调用 → 聚合 | 提速,或要多视角 | 多人同时干 |
| 编排者-工作者 | 动态分解 → 委派 → 综合 | 步骤无法提前确定 | 项目经理带组 |
| 评估者-优化者 | 生成 ↔ 评估循环 | 有清晰标准,迭代能改善 | 作者+编辑 |

**选型直觉**:

- 步骤已知、固定顺序 → **提示链**
- 输入分几类,各走各的 → **路由**
- 子任务已知、可同时做 → **并行化**
- 子任务运行时才知道 → **编排者-工作者**
- 有明确好坏标准、要打磨 → **评估者-优化者**

这五个不是互斥的——一个真实系统常常嵌套组合(路由的某个分支里是个提示链,提示链的某一步是个评估者-优化者循环)。第 8 章讲组合。
