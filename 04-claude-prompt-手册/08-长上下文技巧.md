# 第 8 章 · 长上下文技巧

处理大文档、海量数据（2 万 token 以上）时，怎么组织 prompt 会显著影响效果。这一章是针对长上下文的几个专门技巧。

---

## 8.1 长数据放顶部

**最重要的一条**：把长文档、大段输入放在 prompt 的**顶部**，在你的问题、指令、示例**之前**。

```
（这里放 20 万字的文档）

———

基于上面的文档，回答：……（你的问题）
```

为什么？把问题放在长文档**之后**（即整个 prompt 的末尾），在测试中能把复杂多文档任务的回答质量提升最多 **30%**。

直觉理解：模型读完一长段材料后，紧接着看到问题，材料还"热"在注意力里；如果问题在材料前面，模型读问题时还没看到材料，读材料时又容易忘了问题。**材料在前、问题在后**，符合"先给背景再提问"的自然顺序。

## 8.2 用 XML 标签组织多文档

处理多份文档时，每份用 `<document>` 标签包裹，内含 `<source>`（来源）和 `<document_content>`（内容）等子标签：

```xml
<documents>
  <document index="1">
    <source>年报2023.pdf</source>
    <document_content>
      {{年报内容}}
    </document_content>
  </document>
  <document index="2">
    <source>竞品分析Q2.xlsx</source>
    <document_content>
      {{竞品分析内容}}
    </document_content>
  </document>
</documents>

分析年报和竞品分析。找出我们的战略优势，并建议 Q3 的重点方向。
```

这种结构让模型清楚：有几份文档、每份的来源是什么、边界在哪。回答时它能精确地说"根据年报（文档1）……而竞品分析（文档2）显示……"，引用得清楚。

注意这里同时用了 8.1（文档在前、问题在后）和第 3 章（XML 结构）。两个技巧叠加。

## 8.3 引用接地（Grounding in Quotes）

处理长文档任务时，一个强力技巧：**让模型先引用文档里的相关原话，再执行任务。** 这帮模型从文档的噪音里聚焦到相关部分。

```xml
你是一个医生助手，帮助医生诊断。

<documents>
  <document index="1">
    <source>症状描述.txt</source>
    <document_content>{{症状}}</document_content>
  </document>
  <document index="2">
    <source>既往病历.txt</source>
    <document_content>{{病历}}</document_content>
  </document>
</documents>

先从病历和病史里找出与当前症状相关的原话，放进 <quotes> 标签。
然后基于这些引用，列出有助于诊断的信息，放进 <info> 标签。
```

为什么有效？

- **聚焦**：先引用，强迫模型在海量文本里定位真正相关的部分，过滤噪音。
- **可验证**：你（和医生）能看到结论是基于哪些原话，而不是模型凭空生成——这对高风险场景（医疗、法律、财务）尤其重要。
- **减少幻觉**：要求"引用原文"会抑制模型编造文档里没有的内容。

这个"先引用再分析"的模式，是长文档问答、检索增强（RAG）场景的标准做法。

## 8.4 长上下文与 prompt caching

长上下文意味着大量 token，每次重发都贵。回顾第二册第 6 章：**固定不变的长内容应该缓存**。

典型场景：你对同一份大文档问很多问题。把文档放顶部并加缓存断点，文档部分只在第一次全价处理，后续请求从缓存读（约 0.1x 成本）：

```python
response = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=4096,
    system=[{
        "type": "text",
        "text": f"<documents>{大文档}</documents>",
        "cache_control": {"type": "ephemeral"},   # 缓存这段大文档
    }],
    messages=[{"role": "user", "content": "基于上面的文档，回答：……"}],
)
```

注意缓存的铁律（第二册 6.6）：缓存是前缀匹配，文档要放在稳定的前缀位置、问题放在断点之后。这正好和 8.1"数据在前、问题在后"一致——**长上下文的最佳实践和缓存的最佳实践天然吻合**。

## 8.5 超长任务：状态外置

当任务长到单次上下文窗口装不下（大型迁移、长程 agent），prompt 技巧之外还要做**状态管理**——把状态从对话里挪到文件里。这是第 9 章（agentic）和第一册第 10 章（文档接力）的主题，这里先点出衔接：

> 上下文窗口是易失的小内存，文件系统是持久的大硬盘。长任务的状态（计划、进度、测试结果）放文件，每个上下文窗口只装当前需要的部分。

## 8.6 长上下文技巧速查

| 技巧 | 怎么做 | 为什么 |
|---|---|---|
| 数据放顶部 | 长文档在前，问题在后 | 复杂任务质量 +30% |
| XML 组织文档 | `<document>` + `<source>` + `<document_content>` | 边界清晰、引用精确 |
| 引用接地 | 先让模型引用原文，再分析 | 聚焦、可验证、防幻觉 |
| 缓存固定内容 | 大文档加 `cache_control` | 省成本（约 0.1x） |
| 状态外置 | 超长任务把状态存文件 | 突破窗口上限 |

这几个技巧叠加使用——长文档放顶部、用 XML 结构、加缓存、要求引用接地——能让大文档任务又准、又省、又可信。
