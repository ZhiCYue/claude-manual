# 第 1 章 · 构建工具与框架选型（2026）

选错构建工具,后面所有开发体验和 AI 迭代速度都受连累。这一章给你 2026 年 6 月的当前现状和一棵选型决策树——包括正面回答"webpack + React 还该不该用"。

---

## 1.1 当前格局(2026 年 6 月)

先给结论,再展开:

| 工具 | 现状 | 一句话 |
|---|---|---|
| **Vite** | 新建 SPA 的事实标准,开发者满意度约 95% | 速度优先,SPA/仪表盘/要最大控制权 |
| **Next.js** | 全栈框架领头羊,App Router 稳定、Server Components 默认、Turbopack 逐步取代内部 webpack | 要 SSR/SEO/路由/全栈的应用 |
| **webpack** | 偏 legacy。CRA(基于 webpack)已停止维护,新项目用它**需要明确理由** | 已有项目、微前端、深度定制 |

React 官方文档现在的推荐是:**框架(Next.js / Remix)用于完整应用,Vite 用于 SPA。** 注意——**官方推荐里已经没有 Create React App 了**(CRA 已停止维护)。如果 AI 给你建议用 CRA,那是过时的知识(第 0.6 节的提醒)。

## 1.2 为什么 Vite 赢了新项目

Vite 解决的核心痛点是**开发服务器慢**:

- 开发时用**原生 ES 模块**直接喂给浏览器,**不打包**——所以冷启动是秒级,而非 webpack 那种随项目增大越来越慢的几十秒。
- 配置极简,开箱即用,不用写一大坨 webpack config。
- 生产构建底层用成熟的打包器,且 Vite 正在向 **Rust 实现的打包器(Rolldown)** 迁移,追求更快的构建。

对 AI 辅助开发的额外好处:**快的开发服务器 = AI 迭代快**。AI 改完、你(或它通过浏览器)立刻看到结果,反馈循环短。

## 1.3 选型决策树

```
你的项目需要什么?
│
├── 要 SSR / SEO / 全栈(服务端逻辑、路由、数据获取一体)
│   └── Next.js(App Router)
│       · 内容站、电商、需要 SEO 和首屏性能的应用
│       · Server Components 默认,服务端取数,客户端包更小
│
├── 纯客户端 SPA(仪表盘、内部工具、嵌入式应用,SEO 不重要)
│   └── Vite + React
│       · 开发体验最好、配置最简、控制权最大
│       · 配 React Router 做路由,TanStack Query 管服务端状态(第 3 章)
│
├── 已有的 webpack 项目
│   └── 留在 webpack,别为赶时髦迁移(第 2 章讲怎么用好)
│       · 除非 webpack 的慢已经严重拖累团队,才评估迁移到 Vite
│
└── 微前端 / 模块联邦
    └── webpack 的 Module Federation 仍是最成熟方案(第 2 章)
        · 或评估 Vite 的模块联邦插件(生态在追赶)
```

**一句话决策**:全新项目,要全栈/SEO 选 **Next.js**,纯 SPA 选 **Vite**;已有 webpack 项目就留着;微前端看 **Module Federation**。

## 1.4 关于 webpack + React:正面回答

你专门问了 webpack + React。诚实的回答分两面:

**新项目:不建议从 webpack 起步。** CRA 停止维护、配置繁琐、开发服务器慢,Vite 在几乎所有维度更好。从零搭 webpack + React 在 2026 年需要一个明确的理由。

**但 webpack 远没有死,这些场景它仍是对的:**

1. **已有的大型/企业项目**就在 webpack 上——迁移有成本和风险,如果没痛到不行,留着。
2. **微前端**:Module Federation 是 webpack 的杀手锏,这一领域它仍最成熟。
3. **深度定制构建管线**:有大量自定义 loader/plugin、特殊构建需求的场景。

如果你属于这些情况,第 2 章讲怎么把现代 webpack 5 用好(以及何时该考虑迁移)。如果你是新项目又没有这些硬需求——认真考虑 Vite,别给自己找罪受。

## 1.5 Turbopack:Next.js 的内部演进

如果你用 Next.js,它内部的打包器正在从 webpack 换成 **Turbopack**(Rust 实现,Vercel 出品):

- Turbopack 把 Next.js 的构建速度拉近 Vite 的水平。
- 它在开发模式已稳定,正逐步成为 Next.js 的默认。
- 你用 Next.js 时通常不直接接触它——框架封装了——但知道"Next 内部不再是慢 webpack"有助于理解为什么新版 Next 开发体验好了很多。

## 1.6 选型对 AI 辅助开发的影响

构建工具的选择直接影响 AI 辅助的体验:

- **快的开发服务器(Vite / Turbopack)= 短的反馈循环**:AI 改完立刻能验证,无论是你看还是它通过 Playwright 看(第 8 章)。
- **约定优于配置(Next.js / Vite)= AI 少猜**:框架有清晰的约定(文件路由、约定的目录),AI 据此能推断该把东西放哪。一坨高度自定义的 webpack config,AI(和新人)都更难理解。
- **主流选择 = AI 知识更准**:Vite / Next 是当前主流,AI 对它们的知识比小众工具更准、更新。

这也是为什么"选对工具"是支柱一(好环境)的第一块——它不只是性能问题,也是 AI 辅助效率问题。

## 1.7 选型检查清单

- [ ] 全新项目:要全栈/SEO → Next.js;纯 SPA → Vite。不要用 CRA(已停维护)。
- [ ] 已有 webpack 项目:留着,除非慢到严重拖累(第 2 章)。
- [ ] 微前端:评估 Module Federation(第 2 章)。
- [ ] 确认 AI 给的工具建议是当前的(警惕它推荐 CRA 等过时方案)。
- [ ] 优先选主流(Vite/Next)——开发体验、生态、AI 知识准确度都受益。
