# 附录 · 模式速查、决策树与反模式

---

## A. 复杂度阶梯(最高原则)

```
从下往上,只在下一层确实更好时才爬:

4. 多 agent(编排者-工作者)    ← 最贵(15× token),最后考虑
3. 单 agent(模型自主)         ← 步骤无法预定时
2. 工作流(固定路径编排)        ← 步骤可预定时
1. 单次 LLM 调用(+检索+示例)   ← 大多数需求停在这
0. prompt engineering          ← 措辞能解决就不要 agent
```

**从简单开始,复杂度要挣来。** 构建正确的系统,而非最复杂的系统。

## B. 工作流五模式速查

| 模式 | 结构 | 何时用 | 一句话 |
|---|---|---|---|
| 提示链 | 顺序步骤 + 门 | 步骤固定,顺序明确 | 流水线 |
| 路由 | 分类 → 分支 | 有不同类别,分开处理更好 | 分诊台 |
| 并行化 | 多调用 → 聚合 | 提速,或要多视角 | 多人同时干 |
| 编排者-工作者 | 动态分解 → 委派 → 综合 | 步骤无法提前确定 | 项目经理带组 |
| 评估者-优化者 | 生成 ↔ 评估循环 | 有清晰标准,迭代能改善 | 作者+编辑 |

## C. 选型决策树

```
① 问题措辞能解决吗?
   是 → prompt engineering(第四册)
   否 ↓
② 步骤能提前确定吗?
   能 → 工作流
        ├ 步骤固定顺序 → 提示链
        ├ 分几类各走各 → 路由
        ├ 子任务已知可并行 → 并行化
        ├ 子任务运行时才知 → 编排者-工作者
        └ 有好坏标准要打磨 → 评估者-优化者
   不能 ↓
③ 单 agent 自主能搞定吗?
   能 → 单 agent + ACI + 上下文工程 + 护栏
   不能 ↓
④ 能切独立并行子任务,且价值证明 15× token?
   能 → 多 agent
   不能 → 退回单 agent / 重新拆解需求
```

## D. Harness 的五大职责

```
① 跑 agent 循环
② 真正执行工具调用
③ 管理上下文(压缩/记忆/子代理)
④ 检查权限、运行护栏
⑤ 错误恢复、检查点
```

## E. 上下文工程三技术

| 技术 | 做什么 | 适合 |
|---|---|---|
| 压缩(compaction) | 总结历史、清掉已处理的工具结果 | 长对话流 |
| 结构化记笔记 | 把状态写到上下文外的文件 | 迭代式开发 |
| 子代理架构 | 隔离上下文,只返回 1-2k token 摘要 | 并行研究 |
| (即时检索) | 维护轻量标识符,运行时按需取 | 大语料 |

## F. 多 agent 成本账

```
聊天      1× token
单 agent  约 4× token
多 agent  约 15× token

回报(研究类任务):质量 +90.2%,时间 -90%
划算条件:任务价值能证明 15× token,且任务可切独立并行子任务
不划算:编码(深度耦合)、需共享上下文、需实时协调
```

## G. 反模式清单

| ❌ 反模式 | 为什么糟 | ✅ 改法 |
|---|---|---|
| 措辞能解决也上 agent | 徒增成本和不可预测性 | 先试 prompt(第 1.4 节) |
| 步骤固定却用 agent | 失去可预测性和一致性 | 用工作流(第 2 章) |
| 编码任务上多 agent | 深度耦合、需共享上下文 | 单 agent(第 6.3 节) |
| 不算 token 账就上多 agent | 烧 15× token 不自知 | 先算价值是否值(第 6.2 节) |
| 委派指令含糊 | 子代理重复劳动 | 给目标/格式/边界(第 6.4①) |
| 简单查询派一堆子代理 | 协调成本吃掉收益 | 努力匹配复杂度(第 6.4②) |
| 工具描述随便写 | 把 agent 引上错路 | 像 docstring 一样写(第 4 章) |
| 工具集臃肿重叠 | 模型困惑、上下文膨胀 | 聚焦 + tool search(第 4.5 节) |
| 把所有数据预加载进上下文 | context rot,注意力稀释 | 即时检索(第 5.5 节) |
| 不给 agent 自我验证手段 | "我觉得好了"不可靠 | 给地面真相工具(第 3.3 节) |
| 关掉所有检查求自主 | 危险 | 护栏内放手(第 3.5 节) |
| 用断言精确匹配测 agent | agent 非确定,测不了 | LLM 裁判 + 人工(第 7.3-7.4) |
| 没有检查点的长程 agent | 第 99 步崩了从头来 | 检查点 + 可恢复(第 7.6 节) |
| 一上来建大评估集 | 浪费,改动效果戏剧性 | 从 20 个真实查询起步(第 7.2 节) |

## H. 评估与可靠性清单

```
评估:  20 个真实查询起步 → LLM 裁判规模化 → 人工抓边界
可观测:全链路追踪、监控决策模式(可不看内容保护隐私)
可靠:  检查点、重试喂回模型、从出错处恢复、彩虹部署
护栏:  权限/审批、沙箱、停止条件、关键节点人在回路
```

## K. 全册一页纸

```
核心问题:任务复杂到要 agent 自主/多 agent 协作时,系统怎么架构?

最高原则:从简单开始,复杂度要挣来

复杂度阶梯:prompt → 单次调用 → 工作流 → 单 agent → 多 agent

Harness 五职责:循环 / 执行工具 / 管上下文 / 护栏 / 错误恢复

工作流五模式:链式 / 路由 / 并行 / 编排者-工作者 / 评估者-优化者

自主 agent:LLM 在循环里基于环境反馈使用工具(给它地面真相)

ACI:工具是瓶颈,值得和 prompt 同等的工程投入

上下文工程:上下文是有限资源,压缩/记笔记/子代理/即时检索

多 agent:质量 +90% 但烧 15× token,只在值得时用,委派要清晰

可靠性:非确定系统要可观测(追踪)、可恢复(检查点)、有护栏
```
