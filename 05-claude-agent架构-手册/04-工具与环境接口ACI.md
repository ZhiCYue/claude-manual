# 第 4 章 · 工具与环境接口 ACI

Anthropic 做 SWE-bench agent 时,**花在工具设计上的时间比花在整体 prompt 上还多**。工具是 agent 和世界交互的接口——接口设计得好不好,直接决定 agent 成不成。这一章讲 ACI(Agent-Computer Interface)。

---

## 4.1 ACI 是什么、为什么重要

ACI = Agent-Computer Interface,agent 与计算机/环境交互的接口,主要就是**工具**。

类比:HCI(人机交互)决定人用软件用得顺不顺。ACI 决定 agent 用工具用得顺不顺。一个 Anthropic 反复验证的论断:

> **"agent-工具接口和人机接口一样关键。"** 糟糕的工具描述会把 agent 引上完全错误的路;改进工具描述能让任务完成时间减少 40%。

换句话说:**工具需要和整体 prompt 同等的工程严谨度。** 你不会随便写个按钮就指望用户会用,同样别随便定义个工具就指望 agent 会用对。

## 4.2 工具设计的核心原则

### ① 给模型足够的"思考空间"

设计工具的输入输出格式时,给模型足够的 token 在提交前思考。别强迫它在还没想清楚时就吐出一个紧凑的结果。

### ② 用互联网文本里自然出现的格式

模型在训练中见过海量的某些格式(markdown、JSON、常见的命令行语法)。**用这些自然格式,别发明奇怪的自定义格式。** 比如让 agent 写代码改动,用标准的 diff 格式比你自创的格式好——它见过无数 diff。

### ③ 消除格式负担

别让工具要求模型做容易出错的格式工作:

- 不要求精确的行数计数。
- 不要求大量转义。
- 不要求模型维护它容易算错的东西。

把这些负担从模型身上拿掉,让它专注于决策。

### ④ 工具文档要像给初级开发者的 docstring

回顾第三册第 2 章——这是 MCP 工具设计的核心,这里再强调:

- 包含**示例用法**、边界情况、输入要求。
- 描述清晰得像写给初级开发者看的 docstring。
- 参数名和描述**无歧义**。

```python
# ❌ 模糊
@tool("search", "搜索")
def search(q): ...

# ✅ 清晰得像 docstring
@tool("search_orders", """按条件搜索客户订单。
当用户询问订单状态、历史订单、或需要按客户/日期/金额查订单时使用。
返回最多 50 条匹配的订单,JSON 格式。""")
def search_orders(
    customer_id: str,    # 客户 ID,如 "cust_12345"
    status: str = "all", # 订单状态:pending/shipped/delivered/all,默认 all
    since: str = None,   # 起始日期 ISO 格式,如 "2026-01-01",可选
): ...
```

## 4.3 防错设计(Poka-yoke)

Poka-yoke 是日语"防呆"——通过设计让错误**不可能发生**,而不是靠提醒避免。把它用在工具设计上:**改变参数的设计,让 agent 不可能用错。**

Anthropic 的经典例子:他们的 SWE-bench agent 一开始用**相对路径**,agent 切出根目录后就经常迷路、找错文件。改成**要求绝对路径**后,这类导航错误彻底消失了——不是靠提醒 agent"注意路径",而是靠改变接口让错误不可能发生。

防错设计的思路:

- 看 agent 常犯什么错 → 改接口让那个错犯不出来。
- 用绝对路径而非相对路径(消除"当前目录在哪"的歧义)。
- 用枚举而非自由字符串(消除拼错的可能,第二册第 2 章的 `Literal`/enum)。
- 让工具拒绝危险或无意义的输入(在工具内部验证,第三册第 7 章)。

## 4.4 工具表面设计:bash 还是专门工具?

回顾第二册第 4.8 节,这是 agent 架构的核心判断,在 ACI 语境下再深入:

| | bash 工具 | 专门工具 |
|---|---|---|
| 给 agent | 最大的程序化能力(几乎能做任何事) | 针对性的能力 |
| 给 harness | 一个不透明的命令字符串 | 类型化参数,可拦截/渲染/审计/并行 |

**经验法则**:从 bash 起步求广度;当你需要**门控、渲染、审计、并行化**某个动作时,把它提升为专门工具。

- **安全边界**:难撤销的操作(删除、外部 API、发消息)提升为专门工具,才能门控。
- **新鲜度检查**:专门的 `edit` 工具能在文件被改动时拒绝写入,bash 做不到。
- **并行调度**:只读工具(`glob`、`grep`)能标记并行安全,bash 里 harness 分不清并行安全的 grep 和不安全的 push。

## 4.5 工具集要克制(token 效率)

回顾第三册第 6.7 节和第四册——每个工具的描述都常驻上下文。工具集的设计要考虑 token 效率:

- **最小化功能重叠**和"该用哪个工具"的歧义。臃肿的工具集会把 agent 推进不清晰的决策点。
- **返回精简、token 高效的格式**。工具返回一大坨原始数据会吃光上下文(第 5 章)。让工具返回提炼过的、agent 真正需要的。
- **参数描述性强、无歧义**,贴合模型的固有强项。

工具太多的代价:模型困惑(该用哪个?)+ 上下文膨胀(所有 schema 常驻)。工具库很大时,用 **tool search**(动态发现相关工具,而非全部塞进去)。

## 4.6 增强环境反馈的工具

第 3.3 节讲过,agent 靠环境反馈工作。有些工具的价值就在于**提供地面真相**:

- **测试运行工具**:让 agent 跑测试,知道代码对没对。
- **截图/浏览器工具**(如 Playwright MCP):让 agent 看到 UI 真实样子。
- **裁剪/缩放工具**:让 agent 放大看图片细节——Anthropic 测试显示给 Claude 裁剪工具能持续提升视觉评估表现。
- **代码执行**:让 agent 跑代码看真实输出,而非凭空推断。

设计 agent 时,问自己:**它怎么知道自己做对了?** 然后给它提供那个地面真相的工具。

## 4.7 测试和迭代你的工具

工具不是写完就完——要像调 prompt 一样调工具:

- 在 workbench / Inspector(第三册第 7 章)里大量测试,找出 agent 用错工具的地方。
- 看 agent 怎么用你的工具——它误解了哪个参数?走错了哪条路?
- 根据观察改进描述、改参数设计(防错)、调返回格式。

> Anthropic 做 SWE-bench 时,在工具上花的优化时间超过整体 prompt——因为**工具是 agent 能力的瓶颈**。一个描述含糊的工具,能让最强的模型也跑偏。把工具当一等公民来设计。

## 4.8 ACI 设计清单

设计 agent 的工具集时过一遍:

- [ ] 每个工具的描述清晰得像给初级开发者的 docstring(含示例、边界、输入要求)
- [ ] 参数名和描述无歧义
- [ ] 用了自然格式(diff、JSON、markdown),没发明奇怪格式
- [ ] 没有易错的格式负担(行数、转义)
- [ ] 应用了防错设计(绝对路径、枚举、输入验证)
- [ ] 工具集聚焦,无重叠歧义
- [ ] 返回精简、token 高效的格式
- [ ] 危险操作是专门工具且可门控
- [ ] 提供了 agent 自我验证所需的地面真相工具
- [ ] 在 workbench/Inspector 里测过,改进了发现的用错点
