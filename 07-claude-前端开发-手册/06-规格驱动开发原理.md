# 第 6 章 · 规格驱动开发原理

这是本册支柱二的核心,也是回答"怎么让 AI 实现功能时不跑偏"的关键方法。规格驱动开发(Spec-Driven Development, SDD)——写码前先就"要建什么"达成一致——把 AI 的产出从"我猜你想要"变成"我们审过的确定的东西"。

---

## 6.1 问题:AI 写码的不可预测性

AI 辅助开发最大的痛点:**不可预测**。你给一个需求("加个登录页"),AI 可能:

- 加得和你想的不一样(交互、字段、校验规则各凭想象)。
- 在实现中途越走越偏,做了一堆你没要的。
- 看起来对、实则关键细节(边界、可访问性、错误处理)缺失("形似神不似",第 0.2 节)。

需求越模糊、任务越大,跑偏越严重。前端尤其明显——UI 有大量"没说但你心里有"的预期。

## 6.2 解法:先定 spec,再写码

规格驱动开发的核心思想:

> **在写任何代码之前,先把"要建什么"写成 spec(规格),审查到满意,AI 再照着实现。**

这把不确定性前移到一个**廉价、可审查**的环节:

```
传统(易跑偏)              规格驱动(可控)
─────────────            ─────────────────────
模糊需求 → AI 直接写码      模糊需求 → 写 spec → 审 spec(廉价纠错)→ AI 照 spec 写
   ↓                          ↓
跑偏了才发现,回滚一堆代码    在 spec 阶段就对齐,代码阶段确定性高
```

为什么有效——和第一册 Plan Mode、第五册"探索→计划→执行"是同一个道理:

> **改 spec 是改几句话,改写歪的代码是回滚一堆 diff。** 把对齐放在最便宜的环节(spec)做,纠错成本最低。

## 6.3 SDD 和你已知的东西的关系

如果你读过前几册,SDD 不是全新概念,而是几个东西的"结构化、持久化"版本:

| 你已知的 | SDD 把它升级成 |
|---|---|
| 第一册 Plan Mode(只读出方案,审完再执行) | spec 不再是会话里转瞬即逝的一段话,而是**持久化、进 git、可审查**的文件 |
| 第五册"探索→计划→执行→验证" | 把"计划"这一步**固化成产物**(spec 文件),不依赖记忆 |
| 第四册"给目标和验收标准" | spec 就是详尽的目标 + 验收标准,写成文档 |
| 第六册"约定放工具中立的唯一来源" | spec 是**工具中立**的——所有 AI 助手都能读的"要建什么"的契约 |

所以 SDD 是把这些好习惯**做成一个明确的、可落地的工作流**,而不是靠每个人自觉去 plan、去写验收标准。

## 6.4 spec 里应该有什么

一个好的功能 spec 通常包含:

- **目的/动机**(为什么做这个):背景、要解决的问题。
- **需求**(要建什么):功能点、行为、规则,尽量具体。
- **验收标准/测试场景**(怎么算做对了):可验证的条件。前端的话,包括交互、边界、错误态、可访问性。
- **技术方案**(怎么实现):用什么组件/状态/接口,涉及哪些文件。
- **任务清单**(分几步做):可勾选的实现步骤。

注意这和第四册"给目标和验收标准"一脉相承——spec 就是把"目标 + 验收 + 方案 + 步骤"写成结构化文档。

## 6.5 spec 是可审查的契约

SDD 的一大价值:**spec 是文本,能审、能进 git、能 diff。**

- **审查前置**:你在 AI 写一行代码前就审查"要建什么",这是纠错最便宜的点(6.2 节)。
- **可追溯**:spec 进 git,变更有历史,谁改了什么需求看得见。
- **代码与意图同步**:spec 记录了"为什么这么做",不会像普通代码那样意图丢失。

对团队尤其有价值:**spec 让"要建什么"成为团队可审查的对象**,而不是某个人和 AI 私下聊出来的东西。这呼应第六册的协作规范——把关键决策变成可审查的产物。

## 6.6 SDD 是工具中立的(重要)

回到第六册的双层设计。SDD 有个关键特性:**spec 是工具中立的**。

一份 spec 是普通文本/markdown,**任何 AI 助手都能读**:用 Claude Code 的人让它照 spec 实现,用 Cursor 的人也让它照同一份 spec 实现。所以:

> **spec 是 spec-driven 版的"工具中立唯一来源"**(第六册第 2.4 节)——它是全队共享的"要建什么"的契约,不绑定任何特定 AI 工具。

在混合工具团队里,这特别有用:大家用不同的 AI 工具,但照着**同一份审过的 spec** 干活,产出的方向是一致的。OpenSpec(第 7 章)正是把这套工具中立的 spec 工作流自动化的工具(支持 25+ AI 助手)。

## 6.7 SDD 什么时候用,什么时候不用

和前几册的"从简单开始"一致——SDD 有成本(写 spec、审 spec),不是每个任务都值得:

| 适合 SDD | 不必 SDD |
|---|---|
| 较大的功能(新页面、新模块) | 改个文案、调个样式 |
| 需求模糊、容易理解偏差 | 需求明确的小修小补 |
| 多人协作、要对齐 | 你一个人的探索性尝试 |
| 影响面大、改错代价高 | 低风险、易回滚的改动 |
| 迁移、重构这类大任务 | 一行的 bug 修复 |

经验法则:**任务越大、越模糊、越多人、越难回滚,SDD 越值。** 小的、明确的、低风险的,直接用第一册的常规工作流(甚至直接让 AI 做)更快。

## 6.8 规格驱动原理小结

- **问题**:AI 写码不可预测,需求越模糊越跑偏。
- **解法**:写码前先定 spec,审到满意再让 AI 实现——把对齐放在最便宜的环节。
- **本质**:把 Plan Mode / "计划先行" / "给验收标准" 做成持久化、可审查、工具中立的工作流。
- **工具中立**:spec 是所有 AI 助手都能读的共享契约——混合工具团队的"要建什么"唯一来源。
- **何时用**:大的、模糊的、多人的、难回滚的任务。
