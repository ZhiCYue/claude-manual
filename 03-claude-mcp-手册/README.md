# 自己动手写 MCP 服务器（第三册）

> 编写时间：2026-06，基于 Model Context Protocol 官方文档（modelcontextprotocol.io）与官方 Python/TypeScript SDK。
> 前置阅读：第一册第 7 章（*用* MCP 服务器）、第二册第 5 & 9 章（API 里的 MCP）。这一册教你*造* MCP 服务器。

## 这册讲什么

前两册里 MCP 一直是"别人写好的、你拿来用的"东西。这一册反过来：你成为 MCP 服务器的**作者**，把任何系统——你公司的内部 API、一个数据库、一台设备、一套业务流程——封装成 Claude（以及任何支持 MCP 的 AI 应用）能调用的能力。

写完一个 MCP 服务器，它就能同时被 Claude Code、Claude 桌面端、Cursor、VS Code、你自己用 Agent SDK 写的程序……所有 MCP 宿主即插即用。**写一次，处处可用**——这是 MCP 作为开放协议的核心价值。

## 全书主线：一个协议，三种原语

MCP 的全部内容可以浓缩成一句话：

> **MCP 用 JSON-RPC 定义了 AI 应用（宿主）和外部系统（服务器）之间交换上下文的标准方式。服务器通过三种原语暴露能力：Tools、Resources、Prompts。**

这三种原语的区别，是理解整个协议的钥匙——它们的差别不在技术，而在**谁来控制**：

| 原语 | 是什么 | 谁控制 | 类比 |
|---|---|---|---|
| **Tools（工具）** | 模型能主动调用的函数 | **模型** | 函数调用 |
| **Resources（资源）** | 只读的数据源 | **应用** | GET 请求 / 文件 |
| **Prompts（提示词）** | 预制的指令模板 | **用户** | 斜杠命令 |

整本书围绕这三者展开。理解了"控制权归属"，你就知道一个能力该做成工具、资源、还是提示词。

## 三条贯穿全册的硬规则

1. **能力归类靠"控制权"**：模型该主动调用的 → Tool；只读上下文 → Resource；用户主动触发的模板 → Prompt。（第 0 章详解）
2. **docstring 和类型注解就是协议**：FastMCP 从你的函数签名生成 JSON Schema、从 docstring 生成工具说明。模型完全靠这些文本决定怎么用——写清楚它们等于写好了 API。
3. **第三方服务器 = 给 AI 开新权限**：你写的服务器跑在用户机器上、能访问用户的数据。安全不是可选项（第 7 章）。
