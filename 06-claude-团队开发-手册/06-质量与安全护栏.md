# 第 6 章 · 质量与安全护栏

护栏是让团队"放心用 AI"的基础设施。这一章讲怎么在团队层面建质量和安全护栏——关键是:**对所有人生效的护栏放工具中立层(CI/git hook),Claude Code 权限是额外的便利防线。**

---

## 6.1 护栏的双层落地

回顾第 0、4 章,护栏也遵循双层原则。同一个安全目标,两层各做一份:

```
安全目标:"密钥绝不能进仓库"
│
├─ 工具中立层(对所有人,是保障)
│   · git pre-commit hook:gitleaks / detect-secrets 扫描暂存区
│   · CI:secret 扫描作为 required check
│   · .gitignore:.env、secrets/ 忽略
│
└─ Claude Code 层(对 Claude Code 用户,是便利+第一道防线)
    · .claude/settings.json:deny Read(.env*) / Read(secrets/**)
    · → 防止 Claude Code 把密钥读进上下文
```

**关键认识(再强调)**:Claude Code 的 `deny` 只拦得住用 Claude Code 的人。要"密钥绝不进仓库"对所有人成立,**必须**靠工具中立层(git hook + CI 扫描)。Claude Code 的 deny 是减少密钥被 AI 读取的风险,是好的第一道防线,但不是唯一防线。

## 6.2 质量护栏:CI 是核心

团队质量护栏的核心是 CI。把所有"必须通过"的检查设为 required check + 分支保护,对所有 PR、所有人生效:

```yaml
# .github/workflows/ci.yml（工具中立,对所有人）
name: CI
on: [pull_request]
jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: 测试
        run: make test
      - name: Lint
        run: make lint
      - name: 类型检查
        run: make typecheck
      - name: 安全扫描
        run: |
          pip install detect-secrets
          detect-secrets scan --baseline .secrets.baseline
```

配合分支保护(仓库 Settings → Branches):勾选"require status checks to pass"、"require review"。这样**任何人的 PR**,不过这些检查都合不进去——无论他用 Claude Code、Cursor 还是裸写(第 4 章)。

## 6.3 本地护栏:git pre-commit hook

CI 是最后关卡,但在提交时就拦住问题更省事。git pre-commit hook 在任何人 `git commit` 时触发(工具中立):

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.x.x
    hooks:
      - id: ruff           # lint
      - id: ruff-format    # format
  - repo: https://github.com/gitleaks/gitleaks
    rev: v8.x.x
    hooks:
      - id: gitleaks       # 密钥扫描
```

团队成员装一次(`pre-commit install`),之后每次提交自动跑。**这对所有人生效**,是工具中立层的本地护栏。

> 可以把"装 pre-commit"写进 README 的 setup 步骤,或在 CI 里检查 pre-commit 是否运行过,确保没人漏装。

## 6.4 敏感数据的多重防护

密钥、PII、内部数据是最该护住的。多重防护(全部工具中立,加 Claude Code 便利):

| 防护 | 层 | 防什么 |
|---|---|---|
| `.gitignore` 忽略 .env/secrets | 工具中立 | 不小心 add 进去 |
| git hook 密钥扫描(gitleaks) | 工具中立 | 提交时拦住漏网的密钥 |
| CI 密钥扫描 | 工具中立 | PR 时最后拦截 |
| 用环境变量/密钥管理,不硬编码 | 工具中立 | 从根上不出现明文密钥 |
| Claude Code deny Read(.env*) | Claude Code | 防 Claude Code 读密钥进上下文 |
| 给 AI 用的数据库账号只读 | 工具中立 | 限制 AI(任何工具)的破坏半径 |

> **原则**:敏感数据的防护必须是工具中立 + 多重的。任何单一防线(尤其单一工具的防线)都不够。Claude Code 的 deny 是有价值的一层,但绝不是唯一一层。

## 6.5 AI 代码的安全审查

第 5.4 节讲了 AI 代码的质量风险,安全维度也要专门看(审查时,与工具/模型无关):

- **注入风险**:AI 生成的 SQL 拼接、命令拼接、路径拼接,是否有注入漏洞?
- **越权**:权限检查是否完整,有没有遗漏?
- **敏感信息泄露**:日志、错误信息、返回值里有没有泄露密钥/PII?
- **依赖风险**:AI 引入的新依赖,是否可信、是否有已知漏洞?
- **过度信任外部输入**:AI 可能漏掉边界的输入验证。

可以让 CI 集成安全扫描(SAST 工具、依赖漏洞扫描如 Dependabot),作为对所有人的自动护栏;同时人工审查关注上述点。第一册的 `/security-review` 命令可辅助 Claude Code 用户自查,但不替代团队的 CI 安全门。

## 6.6 无头模式/自动化的护栏

团队如果用 `claude -p`(第二册第 9 章)或 Agent SDK 做自动化(CI 里自动审 PR、批量处理),护栏更关键——没人盯着:

- **窄白名单**:`--allowedTools` 只给刚好够用的工具,别放行 push、删除。
- **`--max-turns`**:限制循环轮数,防失控烧钱。
- **隔离环境**:在沙箱/容器跑,无敏感数据。
- **只读优先**:自动化任务能只读就只读。

这呼应第五册第 3.5 节的护栏和第 7 章的可靠性——自动化的自主性必须在护栏内。

## 6.7 质量与安全护栏检查清单

- [ ] CI 把测试/lint/类型/安全扫描设为 required check(工具中立,对所有 PR)
- [ ] 分支保护:必过 CI、必有审查、不直推 main
- [ ] git pre-commit hook:lint + format + 密钥扫描(工具中立,所有人)
- [ ] 敏感数据多重防护:gitignore + git hook 扫描 + CI 扫描 + 不硬编码
- [ ] 给 AI 用的数据库/外部账号最小权限(只读优先)
- [ ] Claude Code 层:deny 机密文件(额外的第一道防线,不是唯一)
- [ ] AI 代码审查覆盖安全维度(注入、越权、泄露、依赖)
- [ ] 自动化(claude -p / SDK)用窄白名单 + max-turns + 隔离环境

> **本章一句话**:护栏要双层落地,但**"必须守住"的底线一定在工具中立层(CI/git hook)强制**。Claude Code 的权限和 hooks 是宝贵的便利层和第一道防线,永远不是团队安全的唯一防线。
