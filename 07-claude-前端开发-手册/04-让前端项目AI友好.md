# 第 4 章 · 让前端项目 AI 友好

这是支柱一(好环境)的核心。承接第六册第 2 章的通用原则,落到前端:项目结构、约定文档、测试作地面真相、组件约定怎么写给 AI。

---

## 4.1 前端的"AI 友好"=好的前端工程

回顾第六册:AI 友好基本等于好工程。前端版的具体抓手:

- **清晰的、按功能组织的结构** → AI 知道新东西放哪
- **TypeScript** → 减少歧义,AI 少猜
- **成文的组件/状态约定** → AI 照约定写,不自由发挥
- **测试(尤其能跑浏览器的)** → AI 能验证 UI 改对没改对(地面真相)
- **现代约定写下来** → AI 不给过时代码(第 3 章)

## 4.2 项目结构:按功能,不按类型

前端项目最常见的结构错误:按**类型**组织(所有组件堆 `components/`、所有 hook 堆 `hooks/`)。小项目还行,一大就失控。

**按功能/领域组织**(feature-based):

```
src/
├── features/                # 按业务功能组织
│   ├── auth/
│   │   ├── components/      #   该功能的组件
│   │   ├── hooks/          #   该功能的 hook
│   │   ├── api.ts          #   该功能的数据访问
│   │   └── types.ts
│   ├── orders/
│   └── dashboard/
├── shared/                  # 跨功能共享的
│   ├── components/         #   通用 UI 组件(Button、Modal)
│   ├── hooks/
│   └── lib/
├── app/                     # 应用级(路由、布局、provider)
└── ...
```

为什么对 AI 友好:**一个功能的所有东西在一起**,AI 改"订单"功能时,相关代码都在 `features/orders/`,它能快速建立局部认知、知道新东西该放哪。按类型组织则要 AI 在 `components/`、`hooks/`、`api/` 之间来回跳,认知成本高。

## 4.3 文档作为契约(前端版)

承接第六册第 2.3、2.4 节。前端项目的约定文档要写清这些**代码里看不出的事**:

```markdown
<!-- AGENTS.md 或 CONTRIBUTING.md 里的前端约定(工具中立,各 AI 工具都读) -->

## 技术栈
- React 19 + Vite + TypeScript（strict）
- 状态:本地 useState、服务端 TanStack Query、全局 Zustand
- 样式:Tailwind
- 用 React Compiler,不要手动 useMemo/useCallback

## 结构约定
- 按功能组织:features/<功能>/,共享的放 shared/
- 一个功能的组件/hook/api/types 都在该功能目录下

## 组件约定
- 函数组件 + hooks,不用 class
- 可复用逻辑抽自定义 hook
- 通用 UI 组件放 shared/components,要有 Storybook 故事

## 不要做的
- 不要把服务端数据塞进 Zustand(用 TanStack Query)
- 不要手动 memo（React Compiler 处理）
- 不要用 CRA、不要装 file-loader 这类过时工具
```

这份约定是**工具中立的唯一来源**(第六册原则):CLAUDE.md `@import` 它,用 Cursor 的同事的配置也引用它。改一处全同步。

## 4.4 测试作为 AI 的地面真相(前端关键)

这是前端 AI 辅助的一个独特优势,务必用好。回顾第五册第 3 章:**AI 靠环境反馈(地面真相)工作**。前端的地面真相有两层:

### 单元/组件测试:逻辑的地面真相

用 **Vitest + React Testing Library** 测组件行为:

```jsx
test("点击增加按钮,计数加一", async () => {
  render(<Counter />);
  await userEvent.click(screen.getByText("增加"));
  expect(screen.getByText("计数: 1")).toBeInTheDocument();
});
```

让 AI 改完组件"跑 `npm test` 确认"——它就有了"改对没改对"的客观依据。

### 浏览器测试:UI 的地面真相

前端的特殊之处:UI 能**被看到**。用 **Playwright**(可通过 MCP 接给 Claude Code,第三册/第一册)让 AI 实际操作浏览器、截图、验证:

```
> 实现这个登录页的"记住我"功能,写完后:
  打开 localhost:3000/login,实际勾选"记住我"登录,
  截图确认刷新后仍保持登录态
```

AI 驱动真实浏览器验证——把"我觉得 UI 对了"变成"我看到它工作了"。这对前端的"形似神不似"问题(第 0.2 节)是关键防线。

> **设计原则**(第五册第 4.6 节):问自己"AI 怎么知道前端改对了?"然后给它那个地面真相——能跑的测试、能截图的浏览器。前端在这点上得天独厚,别浪费。

## 4.5 类型即文档

TypeScript 的类型是 AI(和人)能读的契约。好的类型减少 AI 的猜测:

```typescript
// 清晰的类型告诉 AI 数据长什么样、函数怎么用
interface Order {
  id: string;
  status: "pending" | "shipped" | "delivered";  // 枚举式,AI 不会瞎填
  total: number;  // 单位:分(在注释或类型名里说明)
  createdAt: string;  // ISO 8601
}
```

`strict: true` 起步。类型清晰的项目,AI 写出来的代码更不容易出类型错误,你的 CI(第 5 章)也能用 `tsc` 兜住。

## 4.6 CLAUDE.md 的前端补充

承接第六册第 2.5 节,前端项目的 CLAUDE.md(配置层,@import 工具中立约定 + Claude 特有补充):

```markdown
# 项目记忆（前端）

完整约定见 @AGENTS.md（技术栈、结构、组件约定都在那）。

## Claude Code 特有补充
- 开发服务器:`npm run dev`（Vite,端口 3000）
- 跑测试:`npm test`（Vitest）
- e2e:`npm run e2e`（Playwright）
- 改完 UI,用 Playwright MCP 实际打开页面验证,别只看代码
- 改组件后跑 `npm test` 和 `npm run typecheck`

## 已知的坑
- shared/components 的组件改动影响面大,改前先看谁在用
```
