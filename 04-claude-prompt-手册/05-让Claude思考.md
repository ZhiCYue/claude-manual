# 第 5 章 · 让 Claude 思考

复杂推理、数学、多步任务上，让模型"先想后答"能显著提升质量。这一章讲 adaptive thinking、effort 参数、手动链式思维、以及自我检查——以及怎么用 prompt 引导思考。

---

## 5.1 为什么思考有效

回到"新同事"隐喻：交给新人一道难题，逼他立刻脱口而出答案，和给他时间在草稿纸上推演，结果天差地别。思考给模型一个"草稿空间"，让它在给出最终答案前先推理、规划、自我纠正。

对这些任务尤其有用：

- 多步推理（逻辑题、规划）
- 数学计算
- 工具调用后的反思（拿到结果后想想下一步）
- 复杂编码任务
- 长程 agent 循环

## 5.2 Adaptive Thinking（推荐）

回顾第二册第 2 章：最新模型（Fable 5 / Opus 4.6+/4.7/4.8 / Sonnet 4.6）用 **adaptive thinking**——模型自己动态决定何时思考、思考多少。

```python
response = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=64000,
    thinking={"type": "adaptive"},
    output_config={"effort": "high"},   # low | medium | high | xhigh | max
    messages=[{"role": "user", "content": "一步步解决这个问题……"}],
)
```

模型根据两个因素校准思考量：**`effort` 参数**和**问题复杂度**。effort 越高、问题越复杂，思考越多；简单问题它直接回答。在内部评估中，adaptive thinking 可靠地优于固定预算的扩展思考。

> **prompt 视角的关键点**：thinking 默认是关的（不传 `thinking` 参数就不思考）。要思考就显式传 `{"type": "adaptive"}`。effort 控制深度（第二册第 2 章有完整的五档说明）。

## 5.3 用 prompt 引导思考行为

思考行为是**可以用 prompt 引导**的。几个实用模式：

### 引导反思

```
收到工具结果后，仔细反思它们的质量，确定最优的下一步再继续。
用你的思考来规划和迭代，然后采取最佳行动。
```

### 抑制过度思考

如果模型思考得比你想要的多（大或复杂的系统提示容易导致这种情况），加引导：

```
扩展思考会增加延迟，只应在它能切实提升答案质量时使用——
通常是需要多步推理的问题。拿不准时，直接回答。
```

### 让它锁定方向、别反复横跳

最新模型（尤其高 effort）有时会做大量前期探索、反复权衡。如果不想要：

```
决定怎么解决一个问题时，选定一个方法并坚持下去。
除非遇到直接与你的推理矛盾的新信息，不要反复推翻决定。
如果在两个方法间犹豫，选一个走到底——之后失败了还可以再纠正。
```

## 5.4 偏好"一般指令"而非"逐步规定"

一个反直觉但重要的原则：**"想得透彻些"往往比你手写的分步计划产生更好的推理。** 模型的推理常常超出人类能规定的细致程度。

```
# 不必这样事无巨细地规定
请按以下步骤思考：1. 先列出所有变量 2. 然后……3. 接着……

# 这样往往更好
请仔细、透彻地推理这个问题，再给出答案。
```

例外：当你需要模型遵循一个**特定的、不可偏离的流程**时，才详细规定步骤。一般的推理任务，给方向、留空间。

## 5.5 让它自我检查

一个高价值的小技巧：让模型在结束前验证自己的答案。这能可靠地抓出错误，尤其在编码和数学上：

```
在你完成之前，对照以下标准验证你的答案：
- 边界情况（空输入、最大值、负数）都处理了吗？
- 逻辑和题目要求一致吗？
- 有没有计算错误？
如发现问题，修正后再给最终答案。
```

呼应第一册第 0 章的原则：**模型检查既有内容的能力，强于一次性写对的能力。** 自我检查把这个优势用起来。

## 5.6 思考关闭时的手动链式思维（CoT）

不开 thinking 时（比如为了省成本或降延迟），你仍能用 prompt 引导分步推理——这就是经典的"链式思维"（Chain of Thought）。用结构化标签把推理和答案分开：

```
解决这个问题。先在 <thinking> 标签里一步步推理，
再在 <answer> 标签里给出最终答案。
```

模型会在 `<thinking>` 里展开推演，在 `<answer>` 里给结论。你能看到推理（调试用），又能干净地提取答案（只给用户看 `<answer>`）。

> **注意**：thinking 关闭时，Opus 4.5 对"think"这个词特别敏感，容易触发过多推理。这种情况下用"consider"（考虑）、"evaluate"（评估）、"reason through"（推敲）等替代词。中文里"想一想"也可换成"分析""推敲""评估"。

## 5.7 示例配合思考

第 2 章提过：在 few-shot 示例里用 `<thinking>` 标签展示推理模式，模型会把这种风格泛化到自己的思考中：

```
<example>
问题：……
<thinking>
（展示你想要的推理步骤）
</thinking>
答案：……
</example>
```

这是教模型"按某种特定方式推理"的有效手段——比文字描述推理方式更直观。

## 5.8 思考的实用决策

| 场景 | 怎么做 |
|---|---|
| 复杂推理/数学/规划 | 开 adaptive thinking，effort 设 high+ |
| agent 多步任务 | 开 adaptive thinking（它自动在工具调用间穿插思考） |
| 简单问答/分类 | 不开思考（默认），或 effort low |
| 要看推理过程但模型简洁了 | adaptive + `display: "summarized"`（第二册第 2 章） |
| 思考太多拖慢响应 | 降 effort，或加"拿不准时直接回答"的引导 |
| 不开思考但要分步 | 手动 CoT，用 `<thinking>`/`<answer>` 标签 |
| 关键任务怕出错 | 加自我检查指令 |

> effort 的完整说明（五档、各档适用场景）在第二册第 2 章。第 10 章会讲 Fable 5 / Opus 4.8 上 effort 的调优——它在新模型上比以往更重要。
