# 第 3 章 · 上下文与记忆：CLAUDE.md

## 3.1 解决什么问题

模型每次新会话都是"失忆重启"的——它不记得你昨天说过"这个项目用 pnpm"。CLAUDE.md 是写在磁盘上的**长期记忆**：每次会话启动时自动注入上下文，相当于给每个"新生的 Claude"塞一份入职手册。

> **原理：为什么 CLAUDE.md 里的话比对话里说的更"管用"？**
> CLAUDE.md 被拼接在系统提示词附近，处于模型注意力的高权重位置，且**每轮都在**（对话中途说的话会随历史变长被稀释、被 compact 压掉）。所以判断标准很简单：**说过两次以上的事，就该进 CLAUDE.md。**

## 3.2 记忆的层级

按加载顺序（都会加载，后者补充前者）：

| 层级 | 路径 | 作用范围 | 是否进 git |
|---|---|---|---|
| 企业级 | 系统目录（IT 统一下发） | 全公司 | — |
| **用户级** | `~/.claude/CLAUDE.md` | 你机器上所有项目 | 否 |
| **项目级** | `<项目根>/CLAUDE.md` | 本项目所有协作者 | ✅ 是 |
| 项目本地 | `<项目根>/CLAUDE.local.md` | 仅你本人 | 否（gitignore） |
| 子目录级 | `<子目录>/CLAUDE.md` | 在该子目录工作时按需加载 | 是 |

各层级分工示例：

```markdown
<!-- ~/.claude/CLAUDE.md（用户级：个人偏好，跟项目无关） -->
- 始终用中文回复
- git commit message 用英文，遵循 Conventional Commits
- 解释概念时先给结论再给原因
```

```markdown
<!-- 项目根/CLAUDE.md（项目级：团队共识） -->
# 项目：订单服务

## 常用命令
- 启动开发环境：`docker compose up -d && make dev`
- 跑测试：`make test`（不要直接 pytest，环境变量在 Makefile 里）
- 数据库迁移：`make migrate`
- 代码检查：`make lint`（提交前必须通过）

## 架构铁律
- 分层：`src/api/`（路由）→ `src/services/`（业务）→ `src/repos/`（数据访问），
  禁止跨层调用，禁止在 api 层写业务逻辑
- 时间一律 UTC，存 ISO 8601 字符串
- 对外接口改动必须同步更新 `docs/api.md`

## 已知的坑
- `tests/integration/` 依赖本地 Redis，没起 Redis 会大面积超时而不是报连接错误
- Mac 上 `make dev` 第一次跑需要先 `brew install libpq`
```

```markdown
<!-- src/billing/CLAUDE.md（子目录级：模块特例） -->
本模块处理资金，规则从严：
- 金额一律用 Decimal，出现 float 直接算 bug
- 任何改动必须先写测试（TDD）
- 不允许吞异常，所有异常必须带上下文重新抛出
```

## 3.3 三种维护方式

```bash
# 方式一：/init —— 项目第一次用 Claude Code 时执行
#   Claude 扫描代码库（构建脚本、README、目录结构）自动生成初稿
/init

# 方式二：# 速记 —— 日常随手积累（推荐主力方式）
> # 部署脚本在 scripts/deploy.sh，但必须先跑 scripts/preflight.sh

# 方式三：/memory —— 打开文件直接编辑、整理
/memory
```

最佳实践节奏：`/init` 起步 → 日常用 `#` 积累 → 每隔几周用 `/memory` 整理去重。

## 3.4 导入语法：`@` 引用拆分大文件

CLAUDE.md 支持导入其他文件，适合内容多了以后拆分：

```markdown
# 项目记忆

基本约定见下，详细规范按需阅读：
- 完整代码规范：@docs/coding-style.md
- API 设计准则：@docs/api-guidelines.md
- 个人偏好：@~/.claude/my-prefs.md
```

被 `@` 的文件会在加载 CLAUDE.md 时一并读入（最多递归 5 层）。

## 3.5 写好 CLAUDE.md 的原则与反模式

**该写什么——"无法从代码推断的事"**：

- ✅ 命令（构建/测试/部署的正确姿势）
- ✅ 约定（团队的"不成文规定"成文化）
- ✅ 坑（踩过的、反直觉的）
- ✅ 边界（哪些目录是生成代码不要动、哪些文件是历史遗留别学它）

**反模式**：

| ❌ 反模式 | 为什么 | 修正 |
|---|---|---|
| 把目录结构整个贴进去 | 代码自己会说话，Claude 看得比你贴得准，还会过时 | 删掉，只写"分层规则" |
| 写 800 行面面俱到 | 每次会话都占几千 token，且重点被稀释 | 主文件 ≤100 行，细节用 `@` 导入 |
| 「写出高质量的代码」 | 无信息量的指令等于没写 | 写具体可执行的："函数不超过 50 行" |
| 从不更新 | 过时的记忆比没有记忆更糟（它会照做） | 发现 Claude 按旧规则办事时立即修 |

**一个检验方法**：把 CLAUDE.md 给一个新入职同事看——如果对人有用，对 Claude 就有用；如果人看了觉得是废话，删。

## 3.6 自动记忆

Claude Code 还有一套**自动记忆**（auto-memory）：Claude 在工作中可以自行把"项目的非显见事实、你的偏好与反馈"写入 `~/.claude/projects/<项目>/memory/` 目录，下次会话自动召回。

- 它和 CLAUDE.md 互补：CLAUDE.md 是**你主动写**的契约，自动记忆是**它观察积累**的笔记。
- 你可以直接让它记：`> 记住：以后这个项目的 PR 描述都用中文写`
- 也可以审计它记了什么：`> 看看你的 memory 里都存了什么，过时的删掉`
