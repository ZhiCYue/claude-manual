# 第 5 章 · 前端的工具中立质量门

第六册的双层设计落到前端:把"必须做到"的前端质量标准放工具中立层(CI / git pre-commit)强制,对所有人(用任何 AI 工具)生效。这一章是前端版的质量门配置。

---

## 5.1 回顾:为什么放工具中立层

第六册的核心铁律,在前端同样成立:

> **想让一条前端质量标准对团队所有人生效,就在工具中立层强制它(CI / git pre-commit / 分支保护)。** 别指望 Claude Code 的 hook——它只约束用 Claude Code 的人,拦不住用 Cursor 或裸写的同事。

前端尤其需要这个,因为前端团队的工具更杂(各种编辑器、各种 AI 助手),而前端的质量问题(类型错误、lint、坏掉的测试、过大的包)又特别容易溜进去。

## 5.2 前端质量门的组成

一套完整的前端工具中立质量门:

| 检查 | 工具 | 在哪强制 |
|---|---|---|
| 代码风格/格式 | ESLint + Prettier | pre-commit + CI |
| 类型 | `tsc --noEmit` | pre-commit(快的话)+ CI |
| 单元/组件测试 | Vitest | CI(pre-commit 可跑快子集) |
| e2e 测试 | Playwright | CI |
| 包体预算 | bundle 分析 / size-limit | CI |
| 密钥扫描 | gitleaks | pre-commit + CI |

全部工具中立——对所有人、所有 AI 工具生效。

## 5.3 ESLint + Prettier

前端代码风格的基础。配置进 git,所有编辑器和工具读同一份:

```jsonc
// eslint 配置(扁平配置,ESLint 9+)
// 关键:用 typescript-eslint + react-hooks 插件
// react-hooks 插件能抓 hook 用法错误,对 AI 生成的代码尤其有用
```

Prettier 管格式(缩进、引号、换行),ESLint 管代码质量(未用变量、hook 依赖、潜在 bug)。两者分工,别让它们打架(用 eslint-config-prettier 关掉冲突规则)。

> 对 AI 生成代码的价值:ESLint 的 `react-hooks/exhaustive-deps` 等规则能抓出 AI 常犯的 hook 依赖错误;`@typescript-eslint` 能抓类型相关的坏味道。这些是 AI 代码的常见出错点(第六册第 5.4 节)。

## 5.4 git pre-commit:本地第一道关

承接第六册第 6.3 节。前端的 pre-commit:

```yaml
# .pre-commit-config.yaml(或用 husky + lint-staged)
# 提交时只对暂存的文件跑,快
repos:
  - repo: local
    hooks:
      - id: eslint
        name: ESLint
        entry: npx eslint --fix
        language: system
        files: \.[jt]sx?$
      - id: prettier
        name: Prettier
        entry: npx prettier --write
        language: system
        files: \.(jsx?|tsx?|css|json)$
```

用 **lint-staged** 只对暂存文件跑(快)是前端的常见做法:

```jsonc
// package.json
"lint-staged": {
  "*.{js,jsx,ts,tsx}": ["eslint --fix", "prettier --write"],
  "*.{css,json,md}": ["prettier --write"]
}
```

任何人 `git commit` 都触发——工具中立,对所有人。

## 5.5 CI:最后的强制关卡

承接第六册第 6.2 节。前端 CI 把所有"必须过"的检查设为 required check + 分支保护:

```yaml
# .github/workflows/ci.yml(前端版)
name: CI
on:
  pull_request:
    branches: [main]
jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: npm }
      - run: npm ci
      - run: npm run lint          # ESLint
      - run: npm run typecheck      # tsc --noEmit
      - run: npm test               # Vitest
      - run: npm run build          # 确保能构建
  e2e:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: npm }
      - run: npm ci
      - run: npx playwright install --with-deps
      - run: npm run e2e            # Playwright e2e
```

配合分支保护(不直推 main、必过 CI、必有审查)。这样**任何人的 PR**,lint/类型/测试/构建/e2e 不过都合不进去——无论他用什么 AI 工具(第六册第 4 章)。

## 5.6 包体预算:前端特有的质量门

前端有个独特的质量维度:**包体大小**。AI 容易不知不觉引入大依赖、或写出导致包体膨胀的代码。在 CI 加包体预算:

```jsonc
// 用 size-limit 或类似工具,设预算,超了 CI 失败
"size-limit": [
  { "path": "dist/assets/index-*.js", "limit": "200 KB" }
]
```

这把"包不能太大"变成自动强制的规则,而不是靠人记得跑 bundle analyzer。对 AI 辅助尤其有用——AI 可能为了实现功能引入大库,包体预算会当场拦住。

## 5.7 两层在前端的配合

回顾第六册第 4.4 节"一个标准两层落地"。前端的"代码必须过 lint",三层防护:

```
① 写时(Claude Code 层,仅 Claude Code 用户)
   .claude/settings.json 配 PostToolUse Hook:改完 .tsx 即时 eslint --fix
   → 体验顺,但只是便利

② 提交时(工具中立层,所有人)
   lint-staged + pre-commit:任何人 commit 都跑 ESLint

③ PR 时(工具中立层,所有人)
   CI 把 lint 设 required check + 分支保护
   → 任何人不过 lint 都合不进
```

**便利可以工具特定(①),保障必须工具中立(②③)。** 这是第六册的铁律在前端的落地。

## 5.8 前端质量门检查清单

- [ ] ESLint + Prettier 配置进 git(含 react-hooks、typescript-eslint 插件)
- [ ] pre-commit + lint-staged:提交时 lint/format 暂存文件(所有人)
- [ ] CI required checks:lint / typecheck / test / build / e2e(所有 PR)
- [ ] 分支保护:不直推 main、必过 CI、必有审查
- [ ] 包体预算(size-limit)防包膨胀
- [ ] 密钥扫描(gitleaks)
- [ ] (Claude Code 用户额外)PostToolUse Hook 写时即时 lint/format
- [ ] 记住:必须做到的放工具中立层(②③),Claude Code 层(①)只是便利
