# 第 9 章 · Agentic Prompt 进阶

这一章是"进阶"真正的价值所在。当模型不只是一问一答、而是作为 **agent** 自主调工具、跑长任务时，一套新的 prompt 技巧浮现出来——工具触发、并行调用、自主与安全的平衡、子代理、状态跟踪、防过度工程、防幻觉。这些是构建可靠 AI agent 的核心 prompt 功夫。

---

## 9.1 工具触发：让模型动手，而非只建议

最新模型为精确遵循指令而训练。一个微妙但常见的坑：**你说"能不能建议一些改进"，模型会给建议而不是真去改**，哪怕你本意是让它动手。

```
# 模型只会建议
能不能给这个函数提些性能改进建议？

# 模型会真的改
改进这个函数的性能。

# 或
对认证流程做这些修改。
```

要模型采取行动，**用祈使句明确要求动作**，别用商量的口吻。

### 用系统提示设默认行为

想让模型默认更主动（倾向于动手）：

```
<默认行动>
默认实施修改，而不只是建议。如果用户意图不明确，推断最有用的可能行动并执行，
用工具去发现缺失的细节，而不是猜测。尽量推断用户是否想要工具调用
（如改文件或读文件），据此行动。
</默认行动>
```

反过来，想让模型默认更谨慎（除非明确要求才动手）：

```
<不要先于指令行动>
除非明确被要求修改，不要贸然进入实现或改文件。当用户意图模糊时，
默认提供信息、做调研、给建议，而不是采取行动。
只在用户明确要求时才进行编辑、修改或实现。
</不要先于指令行动>
```

选哪个取决于你的产品——编码 agent 通常要前者（主动），咨询型助手通常要后者（谨慎）。

### 别用过激的语言（重要）

最新模型（Opus 4.5/4.6+）对系统提示的遵循度比老模型高得多。**为老模型写的"反偷懒"过激措辞，在新模型上会过度触发。**

```
# ❌ 老模型时代的写法，新模型上会过度触发
CRITICAL: 你必须在每次回答前使用搜索工具！

# ✅ 新模型上这样就够
需要查证当前信息时使用搜索工具。
```

第 10 章会专门讲这个——它是从老模型迁移过来最该改的一处。

## 9.2 优化并行工具调用

最新模型擅长并行执行工具——研究时跑多个推测性搜索、一次读多个文件、并行执行 bash 命令。这行为开箱即用成功率就很高，但可以用 prompt 拉到接近 100% 或调节激进度：

```
<使用并行工具调用>
如果你打算调用多个工具且它们之间没有依赖，就并行发起所有独立的工具调用。
只要动作能并行就优先同时调用，而非顺序调用。例如读 3 个文件时，
并行发 3 个调用同时把 3 个文件读进上下文。尽可能最大化并行调用以提速。
但如果某些工具调用依赖前面调用的结果（如用前一步的输出作参数），
就不要并行，而要顺序调用。绝不要用占位符或猜测缺失的参数。
</使用并行工具调用>
```

要减少并行（比如并行 bash 拖垮了系统）：

```
顺序执行各操作，每步之间短暂停顿以确保稳定性。
```

## 9.3 自主与安全的平衡

没有引导时，最新模型可能采取难以撤销或影响共享系统的行动——删文件、强推、发到外部服务。要让它在危险操作前先确认：

```
考虑你行动的可逆性和潜在影响。鼓励你采取本地的、可逆的行动
（如编辑文件、跑测试），但对难以撤销、影响共享系统、或可能有破坏性的行动，
先询问用户再进行。

需要确认的行动示例：
- 破坏性操作：删除文件或分支、删数据库表、rm -rf
- 难以撤销：git push --force、git reset --hard、修改已发布的提交
- 对他人可见：推送代码、在 PR/issue 评论、发消息、改共享基础设施

遇到障碍时，不要把破坏性操作当捷径。例如不要绕过安全检查（如 --no-verify），
不要丢弃可能是进行中工作的陌生文件。
```

这把"可逆/不可逆"作为判断标准——本地可逆的放手做，难撤销的先问。呼应第一册第 4 章的权限思想，但这里是用 prompt 而非硬性权限来引导。

## 9.4 子代理编排

最新模型能识别哪些任务适合委派给专门的子代理，并主动这么做，不需要明确指令。要利用这个能力：

1. **提供定义良好的子代理工具**，在工具定义里描述清楚。
2. **让模型自然编排**——它会恰当地委派。
3. **警惕过度使用**——Opus 4.6 对子代理有强烈偏好，可能在一个直接的 grep 就够的场景也派子代理。

如果看到子代理用得过多，加引导说明何时该用、何时不该：

```
当任务可以并行、需要隔离的上下文、或涉及不需共享状态的独立工作流时，使用子代理。
对简单任务、顺序操作、单文件编辑、或需要跨步骤保持上下文的任务，直接做，不要委派。
```

## 9.5 状态跟踪与长程任务

最新模型擅长长程推理和状态跟踪——能做一部分复杂任务、保存状态、用新的上下文窗口继续。prompt 上的配合：

### 用结构化格式存结构化状态

```json
// tests.json —— 结构化的状态文件
{
  "tests": [
    {"id": 1, "name": "认证流程", "status": "passing"},
    {"id": 2, "name": "用户管理", "status": "failing"}
  ],
  "total": 200, "passing": 150, "failing": 25, "not_started": 25
}
```

### 用自由文本存进度笔记

```
// progress.txt
第 3 轮进度：
- 修复了认证 token 校验
- 更新了用户模型处理边界情况
- 下一步：排查 user_management 测试失败（测试 #2）
- 注意：不要删测试，可能导致功能缺失
```

### 用 git 跟踪状态

git 提供了"做过什么"的日志和可恢复的检查点。最新模型特别擅长用 git 跨会话跟踪状态。

### 管理上下文限制

如果你的 agent 框架会压缩上下文或允许把状态存到文件（如 Claude Code），告诉模型这一点，免得它接近上下文上限时过早收尾：

```
你的上下文窗口接近上限时会被自动压缩，让你能从中断处无限继续。
因此，不要因为 token 预算而提前停止任务。接近 token 预算上限时，
把当前进度和状态存到记忆里再让上下文刷新。始终尽可能持久、自主，
完整完成任务，即使预算快用完。绝不要因为剩余上下文而人为提前停止任务。
```

## 9.6 防过度工程

最新模型（Opus 4.5/4.6+）有过度工程的倾向——创建多余文件、加不必要的抽象、构建没要求的灵活性。看到这种行为，加引导保持最小：

```
避免过度工程。只做明确要求或确实必要的修改。保持方案简单、聚焦：

- 范围：不要添加功能、重构代码、或做超出要求的"改进"。修个 bug 不需要顺手
  清理周围代码。一个简单功能不需要额外的可配置性。
- 文档：不要给你没改的代码加 docstring、注释或类型注解。只在逻辑不自明处加注释。
- 防御性编码：不要为不可能发生的场景加错误处理、回退或校验。信任内部代码和框架保证。
  只在系统边界（用户输入、外部 API）做校验。
- 抽象：不要为一次性操作创建辅助函数或抽象。不要为假想的未来需求设计。
  恰当的复杂度是当前任务所需的最小复杂度。
```

## 9.7 防止"只为通过测试"和硬编码

模型有时会过度聚焦于让测试通过，而非写通用解——或用辅助脚本绕过标准工具。要防止：

```
请用可用的标准工具写一个高质量、通用的解决方案。不要创建辅助脚本或变通办法
来更"高效"地完成任务。实现一个对所有合法输入都正确的方案，而不只是测试用例。
不要硬编码值或写只对特定测试输入有效的方案。要实现真正解决问题的逻辑。

聚焦于理解问题需求、实现正确的算法。测试是用来验证正确性的，不是用来定义解的。
给出遵循最佳实践和软件设计原则的、有原则的实现。

如果任务不合理或不可行，或某些测试本身有错，请告诉我，而不是绕过它们。
解决方案应当健壮、可维护、可扩展。
```

## 9.8 防幻觉（agentic 编码）

最新模型更不容易幻觉，但要进一步抑制、确保答案基于真实代码：

```
<调查后再回答>
绝不臆测你没打开过的代码。如果用户引用了某个具体文件，回答前你必须先读那个文件。
回答关于代码库的问题前，务必先调查、阅读相关文件。
调查之前绝不对代码下任何断言，除非你确定答案正确——给出有据、无幻觉的回答。
</调查后再回答>
```

这把"先看再说"变成硬性要求，抑制"凭印象答"导致的幻觉。

## 9.9 减少临时文件

最新模型有时会创建新文件用于测试和迭代（把 Python 脚本当临时草稿纸）——这通常能改善结果。但如果你想减少凭空多出的文件：

```
如果你为迭代创建了临时文件、脚本或辅助文件，任务结束时清理掉它们。
```

## 9.10 Agentic prompt 速查

| 想要 | prompt 方向 |
|---|---|
| 模型动手而非只建议 | 用祈使句；`<默认行动>` 片段 |
| 模型更谨慎 | `<不要先于指令行动>` 片段 |
| 最大化并行工具 | `<使用并行工具调用>` 片段 |
| 危险操作先确认 | 列出需确认的行动（删除、强推、对外可见） |
| 控制子代理用量 | 说明何时该用/不该用 |
| 长任务不提前收尾 | "上下文会自动压缩，不要因预算停止" |
| 防过度工程 | "只做要求的，不加多余抽象/校验/文档" |
| 通用解而非应付测试 | "测试是验证不是定义解，不要硬编码" |
| 防幻觉 | "回答代码问题前必须先读文件" |

> 这些片段是可直接复制的起点（附录有汇总）。但记住第 0 章的方法论：**针对你的实际场景，用评估驱动地调**，别照搬。
