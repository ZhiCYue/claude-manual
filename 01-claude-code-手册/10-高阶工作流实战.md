# 第 10 章 · 高阶工作流实战

前九章是零件，这一章是整机。八个经过实践验证的工作流，每个都标注了用到的章节。

## 10.1 黄金四步法：探索 → 计划 → 执行 → 验证

（用到：Plan Mode④、子代理⑧）适用于一切复杂任务。

```
① 探索（plan 模式，Shift+Tab）
   > 阅读支付模块相关代码，先不要写任何东西。
     理解现有退款流程的实现，特别是和对账模块的交互
② 计划
   > 我要支持部分退款。给出实施计划，列出改动文件、风险点、测试策略
   （审阅计划，来回讨论到满意，批准）
③ 执行
   > 按计划实施。每完成一步跑相关测试再进行下一步
④ 验证
   > 全部完成后：跑全量测试；自查一遍完整 diff，
     站在审查者角度找问题；最后写一份变更总结
```

为什么有效：①② 强迫"先想后做"；你在 ③ 之前介入，纠错成本最低；④ 利用"验证比生成可靠"——模型检查既有代码的能力强于一次写对的能力。

## 10.2 TDD：让验收标准变成铁律

（用到：提问原则①、Hooks⑥可选）

```
> 需求：购物车满 300 减 50，可与会员 95 折叠加，叠加时先满减后打折。
  我们用 TDD：
  第一步：只写测试，覆盖普通/叠加/边界（恰好300、空车）场景。
  此时测试应该失败。写完停下来给我看测试用例。

（审阅测试 —— 此刻你审的其实是"需求理解对不对"，比审实现代码容易得多）

> 用例 4 的预期值错了，叠加时应该是 (358-50)*0.95。改完后开始写实现，
  让所有测试通过。不许修改测试来迁就实现。
```

原理：测试是**机器可验证的需求**。Agent Loop 的终止条件从模糊的"我觉得写好了"变成客观的"测试全绿"，跑偏空间被压缩到极小。配合第 6 章的 Hook 还能彻底锁死"不许改测试"：PreToolUse 拦截对 `tests/` 的 Edit。

## 10.3 多实例并行：git worktree

（用到：上下文隔离⓪）

```bash
# 创建互不干扰的工作副本
git worktree add ../myapp-feat-export -b feat/export
git worktree add ../myapp-fix-login   -b fix/login

# 三个终端，三个独立的 Claude
cd myapp            && claude     # 终端 1：主线，和你讨论架构
cd ../myapp-feat-export && claude # 终端 2：开发导出功能
cd ../myapp-fix-login   && claude # 终端 3：修登录 bug

# 任务完成后清理
git worktree remove ../myapp-feat-export
```

每个实例有独立目录、独立上下文、独立分支。你的角色从"结对程序员"升级为"带几个工程师的 lead"：巡回各终端，分派任务、回答提问、验收结果。配合第 6 章的 Stop 通知 Hook，谁干完了系统自动提醒你去验收。

## 10.4 超大任务：文档接力法

（用到：上下文经济学⓪、CLAUDE.md③）

单次会话装不下的任务（大型迁移、全库改造），用**文件做外部记忆**，把状态从对话里挪到磁盘上：

```
会话 1：
> 目标：整个项目从 JavaScript 迁移到 TypeScript。
  不要动手，先制定分阶段计划写入 MIGRATION.md：
  每阶段一个独立可合并的单元，标注涉及文件、完成标准、验证命令，
  并设一个"进度"区域记录状态

会话 2…N（每个阶段一个干净的会话）：
> 读 MIGRATION.md，执行下一个未完成阶段。
  完成标准达成后更新进度区，commit，然后停下
```

原理：上下文窗口是易失的小内存，文件系统是持久的大硬盘。**计划和进度放硬盘，每个会话只装当前阶段**——这就绕开了窗口上限，且任何一个会话搞砸了都可以丢弃重来（状态在文件里，没丢）。

## 10.5 让 Claude 审 Claude

（用到：子代理⑧、/code-review）

```
写完一个功能后：
方式 A：/code-review                    # 内置审查命令，审当前 diff
方式 B：> 让 code-reviewer 子代理审查这次改动   # 8.4 节的自定义审查员
方式 C：开一个全新会话专门审查            # 最彻底的隔离
```

原理同人类校对：写作时的思路会让作者"看不见"自己的错误。子代理/新会话的上下文里**没有写代码时的思路**，审查视角是干净的。重要改动建议形成习惯：写（会话 1）→ 审（隔离审查）→ 按审查意见修（回会话 1）。

## 10.6 视觉迭代：贴图驱动前端开发

（用到：图片输入①、Playwright MCP⑦）

```
> （Cmd+V 粘贴设计稿截图）
  按这张设计稿实现订单卡片组件。实现后用浏览器打开 storybook，
  截图和设计稿对比，逐项修正差异，直到肉眼看不出区别
```

形成"实现 → 截图 → 对比 → 修正"的闭环。没有设计稿也可以反着用：让它截图当前页面，你在图上指着说"这块间距太挤了"。

## 10.7 代码考古：快速读懂陌生代码库

（用到：Explore 子代理⑧、提问艺术①）

接手新项目的第一天：

```
> /init                                    # 顺手生成 CLAUDE.md
> 这个项目的技术栈和分层结构是怎样的？数据流从哪进从哪出？
> 用 Explore 找出核心领域模型的定义位置，画一张它们之间的关系图（mermaid）
> 如果我要加一个"导出报表"功能，按现有架构应该动哪些层？先别写代码
> git log 里最近三个月改动最频繁的是哪些文件？这些热点说明了什么？
```

半天达到过去一两周的熟悉度。要点：从"整体形状"问到"具体路径"，让它画图，多问"为什么这样设计"。

## 10.8 安全审计流水线

（用到：子代理并行⑧、无头模式⑨）

交互式深度审计：

```
> 并行派三个子代理做安全审计：
  ① src/api/ 的输入校验与越权风险  ② 全库 SQL 拼接与注入风险
  ③ 依赖盘点（过期版本、已知 CVE）
  汇总成分级报告，每项给出 文件:行号 和修复建议
```

固化为 CI 守门员（结合 9.5 节）：每个 PR 自动安检，发现高危直接评论阻塞。

## 10.9 成本与效率调优清单

| 手段 | 章节 | 效果 |
|---|---|---|
| 任务完成就 `/clear` | 2 | 省 token + 提质量 |
| 设计用 Fable/Opus，批量执行切 Sonnet/Haiku | 2 | 省 50%+ 额度 |
| 调研交给子代理 | 8 | 主窗口不膨胀 |
| CLAUDE.md 控制在百行内 | 3 | 每会话固定开销最小化 |
| 卸载不用的 MCP | 7 | 砍掉常驻工具声明 |
| 无头脚本用 `--max-turns` + 窄白名单 | 9 | 防失控烧钱 |
| `--effort` 按需调档 | 1 | 简单任务别开 max |

定期 `/cost` 和 `/context` 复盘，你会很快建立"什么贵、什么占地方"的直觉。
